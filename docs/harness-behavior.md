# Harness behavior & Switchboard handling — canonical reference

The single lookup surface for **how each harness behaves in the scenarios Switchboard cares about, how our adapter/frontend currently handles it, and where the gaps are.** Built 2026-05-27 from a code+research audit of all four adapters and the frontend.

**How this relates to the other docs:**
- **system-design §9** is the *design-level* capability matrix (native / derived / unavailable per capability) and the process model. It stays the canonical design statement; this doc is the *operational* companion (observed shapes → our handling → gaps).
- **`docs/research/archive/<harness>-cli-observed.md`** are the *captured-in-time provenance* — the raw probes (actual strings, exit codes, fixtures) behind the cells here. They are frozen (no longer updated) and live under `research/archive/`; treat **this** doc as the lookup surface and the archived `-observed` files as the evidence it cites: [claude](research/archive/claude-code-cli-observed.md), [codex](research/archive/codex-cli-observed.md), [gemini](research/archive/gemini-cli-observed.md), [antigravity](research/archive/antigravity-cli-observed.md).

**Legend:** ✅ handled correctly · ⚠️ partial / stale · ❌ gap. "Displayed where" distinguishes the **transcript** (per-turn) from the **sidebar** (per-agent) from a **project banner** (startup auth probe).

## 1. Failures

Each turn ends in exactly one terminal outcome: `Completed` · `Failed { kind, message }` · `Cancelled { source }`. `kind ∈ { HarnessError, AdapterFailure, AuthFailure }` (`crates/harness/src/events.rs:347`, `#[non_exhaustive]`). **There is no `UsageLimit`/quota kind.**

### 1.1 Turn-completion detection

| Harness | Signal | Detection |
|---|---|---|
| Claude | `result` event, `is_error:false` | `claude_code/parser.rs:parse_result` ✅ |
| Codex | `turn.completed` | `codex/parser.rs:parse_turn_completed` ✅ |
| Gemini | `result.status:"success"` **OR** (`result.status:"error"` + answer streamed this turn + benign trailing-error signature) | `gemini/parser.rs:parse_result` / `classify_result_failure` ✅ |
| Antigravity | process exit **+** terminal `PLANNER_RESPONSE` answer in `transcript.jsonl` (exit code is useless — 0 on everything) | `antigravity/parser.rs:is_terminal_answer` + `classify_outcome` ✅ |

**Gemini streamed-then-error rescue (G17).** Gemini frequently appends an empty/malformed trailing step *after* it has already streamed a complete answer, tainting `result.status` to `"error"`. When answer content streamed this turn AND the failure carries the benign signature `"empty response or malformed tool call"` (matched on the distinctive tail phrase, case-insensitive — not the generic "invalid stream" prefix), the turn is reclassified `Completed`. The match is narrow and fails safe: signature drift sends such turns back to FAILED, never the reverse. See §1.2 and G17.

### 1.2 Generic harness error

| Harness | Observed shape | Classified | Message | Display | Gap |
|---|---|---|---|---|---|
| Claude | `result.is_error:true` and/or `api_error_status` (`subtype` stays `"success"` — don't trust it) | `HarnessError` | verbatim `result` text | transcript red text + failed chip | exit code now **gates** success: a non-zero/signal exit after successful `result`s fails the turn with folded telemetry (G22) — the old log-only exit-code reconciliation is gone |
| Codex | `turn.failed.error.message`, exit 1 | `HarnessError` | verbatim (JSON-unwrapped) | transcript | — |
| Gemini | `result.status:"error"`; message from `error.message`, else captured from a standalone `type:"error"` event one line earlier | `HarnessError` | real message (no longer empty) | transcript | benign trailing-error reclassifies to `Completed` (§1.1, G17) |
| Antigravity | `Error:` line on stdout (e.g. timeout), exit 0 | `HarnessError` | verbatim `Error:` line | transcript | errors with **no** `Error:` line escape (see 1.4) |

**Gemini message capture (G17).** The terminal `result.status:"error"` line frequently has **no** `error` field at all. The real reason rides a separate top-level `type:"error"` event one line earlier; the parser now captures it (`last_error_message` on `GeminiParserState`) and falls back to it, so a failed Gemini turn surfaces a real message (was empty before). The same captured signature drives the streamed-then-error reclassification (§1.1). **RESIDUAL RISK:** a turn that genuinely failed *after* streaming only partial content, emitting this same benign signature, would be shown as success — accepted trade vs. a misleading FAILED chip on every other complete turn. See `gemini/parser.rs`.

**Uniform authored auth messages (cross-cutting).** Every adapter authors its `AuthFailure` message — naming the harness and the recovery command — instead of surfacing the raw harness text (`"401 Unauthorized"`, `"Please set an Auth method"`, `"Not logged in · Please run /login"`). The user sees one consistent actionable line regardless of which harness's auth surface fired. **None mention "reload Switchboard"** — reactive-auth posture: discover on send, sign in, send again. Constants: `CLAUDE_AUTH_MESSAGE` / `CODEX_AUTH_MESSAGE` / `GEMINI_AUTH_MESSAGE` (`parser.rs` / `codex/parser.rs` / `gemini/parser.rs`); `ANTIGRAVITY_AUTH_MESSAGE` (`antigravity/mod.rs`).

**Reactive-only auth (working UI).** Auth has exactly one user-facing surface in the working UI: a failed turn in the transcript carrying the authored message. There is no proactive startup probe, no project-level "X not authenticated" banner, no per-agent status icon, and no agent-creation auth gate. A logged-out harness is discovered when the user sends to one of its agents; sign in, send again. This is the same path for "never tried" and "tried and failed" — one mental model, nothing to go stale. The backend `check_*_auth` Tauri commands feed the getting-started surface (no-project state), not the working UI.

**Proactive status (`HarnessStatusList`, two opt-in placements).** Install/auth status is shown proactively in exactly two places — the **no-project orientation surface** and a **Settings → "Supported CLIs"** section — both rendering the shared `HarnessStatusList`. Per harness it shows install status (present-on-PATH + version, via `get_harness_install_status` over the trait `HarnessAdapter::version()`) and a best-effort auth ✓/✗ (the four `check_*_auth` probes, including the Claude keychain probe). All four harnesses are uniform ✓/✗ — no "unsupported"/"?" case. The auth marks are **presence heuristics, not validity checks** (the authoritative test is a successful send); a ✗ never blocks anything, and an API-key user may show ✗ yet send fine. Probes refresh on mount and on window refocus (`visibilitychange`), so both placements stay current and neither reintroduces the removed interruptive/stale mid-work banner. The **version** is the parsed semver token (`subprocess::parse_cli_version`), not the raw `--version` line (CLIs pad differently: `2.1.156 (Claude Code)`, `codex-cli 0.134.0`, bare `0.44.0`); **no "update available" detection** — the CLIs self-update/notify, and a remote latest-version comparison is external-data-currency burden (§37) we don't take on. The "fix" hints name the **verified** recovery command per harness (`claude auth login`; `codex login`; run `gemini` / `agy`, which have no login subcommand). Caveat parity with the reactive probes: Claude's keychain presence heuristic is macOS-only and can false-positive on a stale entry (as Codex's `auth.json` check already does).

### 1.5 Cancellation

Uniform and correct across all four: token-driven (`select!`), adapter kills its process group and ends the stream with **no** terminal event, dispatcher synthesizes `Cancelled { source }`. Codex/Gemini/Antigravity all exit 0 on SIGTERM, which the token path correctly sidesteps. ✅ (each has a `live_<harness>_cancel_*` test).

### 1.6 Adapter / parse failure

All four synthesize `AdapterFailure` for malformed JSON, stdout read errors, and EOF-without-terminal (with the stderr tail appended); the frontend `heartbeat_timeout` path also synthesizes one. ✅ Gemini-logged-out and Antigravity-quota used to mis-land here; both now route to their correct `AuthFailure` / `HarnessError` classifications with authored messages (see G1/G2 in §4 — closed).

## 2. How failures surface in the UI (cross-cutting — the biggest finding)

**`error_kind` is carried to the frontend but read by zero components.** The Rust classification (`AuthFailure` vs `HarnessError` vs `AdapterFailure`) flows through `reducers.ts` into `Turn.error_kind` — and `UnifiedTranscript.svelte` renders *every* failed turn identically: the verbatim `turn.error` message in red (`text-status-failed`) + a "failed" `StatusChip`. This is by design now: a per-kind render would split the auth surface across two places (transcript + something else), reopening the disjoint-surfaces problem we just removed. The classification is still wire-carried (kept for future retry-policy decisions); the lever is the *message*, not the render.

**"Transcript disappears" when unauthenticated:** under reactive-only auth, an unauthenticated harness writes no session file, so hydration shows zero turns (correctly empty); the user discovers the auth state by sending, at which point the transcript renders the failed turn with the authored message. No banner stacks above to compete with the transcript content. The empty body is just the no-history state, not a "something is wrong" signal.

### 3.1 Event ⟂ on-disk parity (what survives restart)

For metadata-flavored fields, where a datum lives determines whether Switchboard can re-show it after the app restarts. The TUI never has this question — it doesn't restart mid-session — so any field Switchboard loses on restart is a **TUI-parity gap**. Four classes:

| Class | Meaning | Restart behavior |
|---|---|---|
| **A. Full parity** | Stream + harness session file agree | ✅ Survives — re-read from session file |
| **B. Disk-canonical (we enrich)** | Stream lacks it; session file has it; we re-read at turn-end | ✅ Survives — same source on rehydrate |
| **C. Stream-only, in-memory** | Lives in events; no on-disk equivalent; held in `Runtime` per agent | ❌ **Lost** until next event |
| **D. Absent everywhere** | Not in stream, not on disk, not in any harness file | — Impossible — TUI parity ceiling (workaround at best) |

| Field | Claude | Codex | Gemini | Antigravity |
|---|---|---|---|---|
| Assistant text / tool calls / per-turn final usage | A | A | A (B for read-tool output) | (file is the only source — A by construction) |
| `rate_limit` snapshot | **C** | B (session file) | — (none emitted) | D (CLI log only) |
| Overage flag (`isUsingOverage`) | **C** | n/a | n/a | n/a |
| Context window | **C** (`result.modelUsage`) — persisted to the metadata sidecar (M2), overlaid on reopen | B (session file) | D | D |
| Per-turn cost (`total_cost_usd`) + overage attribution | **C** (`result` record) — persisted to the per-agent **turn-metadata sidecar** (M4), re-joined onto the hydrated turn by `message.id` on reopen | n/a (no cost) | n/a | n/a |
| Model | A | B | A | B-ish (fragile `USER_SETTINGS_CHANGE` scrape — announced on change, carried forward on unchanged resumes; reconstructable from the transcript) |
| MCP servers — live connection status | **C** (`system/init`) | D (config-only) | D (config-only) | D (config-only) |
| Tools registry | A (`system/init`) | D | A | D |

**Persisted tool failure signals (Codex / Antigravity).** Live streams use structured fields where available: Codex reads the streamed command `exit_code`, and Antigravity uses `status:"FAILED"` when the transcript reports it. Durable records can lack those fields: Codex persists command wrapper text in `function_call_output`, and Antigravity `RUN_COMMAND` records can report `status:"DONE"` even when the wrapper says the command failed. The parsers therefore treat the harness wrapper header as a version-sensitive failure contract: Codex reads `Process exited with code <n>` only before the `Output:` body, and Antigravity reads `The command failed with exit code:` only before `Output:` / `Stdout:` / `Stderr:`. Matches inside command output are ignored to avoid false positives. Re-probe on CLI output-format changes.

**The C cells are the restart gaps.** Four were identified for Claude: **`rate_limit_event`** (used %, `isUsingOverage`, reset times), **`result.modelUsage.contextWindow`** (the context bar's denominator), **per-turn `result.total_cost_usd` + overage attribution** (the inline cost / "using credits" marker), and **`system/init.mcp_servers[].status`**. The first three are now **closed**: the rate-limit payload (G13) and `context_window` (M2) by the last-write-wins per-agent **metadata sidecar** (`.meta.json`), and per-turn cost + overage (M4) by a separate per-agent **turn-metadata sidecar** (`.turnmeta.jsonl`, an append-log — one record per real-spend turn — re-joined onto the hydrated turn by `message.id`; see §3.1-cost below). The rate-limit payload is re-read with an "as of …" qualifier; `context_window` carries **no** staleness qualifier (a model's window is fixed, so the value doesn't go stale). The fourth (MCP live status, G14) remains open — also class-C, but deferred because a stale "up 3 days ago" indicator may be worse than none. (Mid-stream token deltas across all four are also class-C but have no UI surface today, so the loss is theoretical.) The metadata sidecar persists the class-C `rate_limit` and `context_window` snapshots; class-B sources (Codex's session-file rate-limit and context window) are not duplicated into it — the gate for the rate limit is the event's `RateLimitSource`, and for the window the `TurnEnd.context_window_source` (`StreamOnly` persisted, `SessionFileBacked` not). The turn-metadata sidecar gates on a per-turn join key being present **and** the turn being real-spend, not on harness identity — Claude is the only harness reporting cost in v1, so the gate is self-selecting and the dispatcher stays harness-agnostic.

**§3.1-cost — the M4 per-turn join (no backfill).** Cost (`total_cost_usd`) and the overage snapshot arrive on Claude's stream-only `result` record, which names no message and may never be written to the session file. The join key is the turn's **final non-subagent assistant `message.id`** — verified to round-trip identically between the live stream and the on-disk session file (a tool-use message can span two records sharing one `message.id` but with two different record-level `uuid`s, so `message.id` is the per-logical-message anchor; record `uuid` is per-record and was rejected). The live path stamps that id on `AdapterEvent::TurnEnd.stable_message_id`; the session-file parser stamps the same id on the hydrated `Turn::Agent.stable_message_id`. At turn-end the dispatcher appends `{message_id, total_cost_usd, spend, captured_at}` to the turn-metadata sidecar; on reopen the app re-joins by `message.id` and fills the hydrated turn's `spend` + `usage.total_cost_usd` (fill-if-empty). Best-effort throughout: a missing/corrupt log (or a corrupt line) degrades to "render nothing for that turn," never a failed hydration. **No backfill** — turns that completed before M4 shipped have no record and render no cost/overage on reopen; expected and documented, not a bug.

**§3.1-hydration — the stable hydration key (idempotent re-reads; refresh eligibility).** Distinct from the cost-join `stable_message_id` (Claude-only, private, `skip_serializing`'d) is a frontend-facing **`hydration_key`** on every hydrated `Turn::Agent` and on the live `NormalizedEvent::TurnEnd`. Its job is dedup: the frontend hydrate merge keys on it (falling back to `turn_id` when absent), so re-reading a session file never duplicates **agent** turns — `turn_id` is minted fresh per parse (`Uuid::now_v7()`) and can't recognize an already-loaded turn, the stable key can. **Scope:** only agent turns from key-bearing harnesses carry a key; **user turns are keyless** (the harness file doesn't know Switchboard's `send_id`) and fall back to the per-parse-fresh `turn_id`, so a re-read over an already-populated slice would *not* dedup them. The split-source design keeps user content owned by the journal overlay (replaced wholesale, dup-safe); a refresh must re-merge only agent turns through the keyed path, never harness user turns into a per-agent slice. Two properties matter and gate different things:

- **Q1 — re-parse-stable** (byte-identical across repeated parses of the *same* file). Enables reopen/refresh dedup of disk-only agent turns. Per harness: **Claude** ✅ the turn's **first** non-subagent assistant `message.id` (parse-invariant across a mid-flight vs completed read — distinct from the cost-join `stable_message_id`, which is the *final* assistant id and moves until the turn ends); **Codex** ✅ the turn's **`turn_context.turn_id`** (the observed shape annotates this the per-turn UUID; we deliberately do *not* use `task_started.turn_id`, whose per-turn-uniqueness is unconfirmed — a non-unique dedup key silently *drops* a new turn as "already seen", strictly worse than the duplication a too-volatile key causes); **Gemini** ✅ the turn's first `gemini` record `id` (a UUID, not the illustrative `g1`/`g2`); **Antigravity** ❌ no native per-turn id — keyless, the merge falls back to `turn_id`.
- **Q2 — live-matched** (the *live* stream carries the same id the file will store, so a turn that streamed live *and* is on disk dedups as one). Strictly stronger; **required for whole-file refresh**. Per harness: **Claude** ✅ confirmed — the **first** non-subagent assistant `message.id` round-trips live↔disk and is what the live `TurnEnd.hydration_key` carries (the cost-join `stable_message_id`, the *final* id, round-trips separately per §3.1-cost); pinned by the ignored live drift-guard `live_claude_hydration_key_matches_live_turn_end`. The Claude live stream **also announces this identity early** — a `TurnIdentity` event at the first assistant message — so the frontend stamps `hydration_key` onto the turn *while it is still streaming*, letting a concurrent switch-back refresh dedup against the in-flight turn instead of duplicating it (the reducer's `turn_identity` arm stamps + reconciles, and protects the active live turn from being superseded by a disk re-read). **Codex** ✅ **probe-verified for the durable send↔turn link, but not for refresh.** The live `TurnEnd.hydration_key` now carries the turn's `turn_context.turn_id`, sourced from the **post-terminal enrichment re-read** of the session file (`emit_terminal_with_enrichment`), so it equals the parser's `hydration_key` **by construction** — confirmed by the two-turn drift-guard `live_codex_hydration_key_matches_live_turn_end` (keys match per turn *and* vary across turns, i.e. no stale carryover). This is what lets the M2 `TurnLink` correlate Codex turns to sends. But it is **not** the strict live-matched property refresh needs: the key becomes available only at the *terminal* (read from disk by enrichment), not *mid-stream*, so a concurrent switch-back refresh of an in-flight Codex turn has no key to dedup on. Codex therefore stays **reopen-stable + link-eligible, but refresh-ineligible** — `supports_refresh` stays off. (Note we deliberately do **not** source from the live stream's `task_started.turn_id`, a different id the parser rejects for unconfirmed per-turn uniqueness; enrichment reads the exact `turn_context.turn_id` field the parser uses.) **Gemini** ❓ **unprobed** — disk key captured (Q1 ✅) but the live `TurnEnd` leaves `hydration_key: None`; a probe must establish **which** record's id the live stream carries (the disk key is the turn's *first* `gemini` record id). **Antigravity** ❌ never eligible (no key).

So **refresh eligibility is Claude-only today** (3-of-4 reopen-stable for disk-only agent-turn dedup; 1-of-4 live-matched). Flipping Gemini on is a one-line capability change once its probe passes (`HarnessKind::supports_refresh` is the single gate; the backend's `project_session_fingerprints` derives `refresh_capable` from it). Codex is a distinct case: it has a probe-verified **durable send↔turn key** (for the `TurnLink` correlation) but *not* a mid-stream live key, so it stays refresh-ineligible until/unless the live stream is shown to carry `turn_context.turn_id` mid-turn. `hydration_key` is best-effort and additive: a keyless turn simply dedups by `turn_id`, exactly as before.

**Durable send↔turn correlation (the M2/M3 key-join), per harness:** **Claude** ✅ and **Codex** ✅ join each content-bearing turn to its send by `hydration_key` via a journal `TurnLink` (Claude's first assistant `message.id`; Codex's `turn_context.turn_id` — both probe-verified live↔disk). **Antigravity** ❌ has no per-turn id, so it stays on **positional correlation** (order/count) — the fragile path a stray harness record can shift; recorded as a known limitation (**G24**). **Gemini** is out of scope (decommissioned; not extended). Per-harness user-side **bookkeeping audit** (the M1 principle applied out): Claude's bare `/compact` is the one confirmed non-prompt record that leaked into `Turn::User` (fixed, M1). Codex (`user_message`) and Antigravity (`USER_INPUT` carrying `<USER_REQUEST>`) show **no** non-prompt user-side record in the fixtures/probes — nothing to reclassify; Codex's compaction shape is **unprobed** (not triggerable on demand), left as-is rather than invented.

**§3.1-refresh — the staleness refresh (TUI-continued turns).** On switching **back** to an already-loaded project, Switchboard re-reads a refresh-capable agent's session file if it changed since last hydration, so a turn the user added by continuing the session in the harness's own TUI appears. The change check is a cheap backend `stat` (`project_session_fingerprints`, no parse) gated on **`(source_path, modified_at, byte_len)`** together (`byte_len` is the more reliable signal for an append-only JSONL; `source_path` catches a moved file, e.g. Gemini candidate selection); the expensive `load_project_conversation` parse runs **only** when that check shows movement, so an unchanged file is never re-parsed. The re-read uses the **project** path (not per-agent) so user content stays journal/overlay-owned, and applies agent turns through the M2 keyed merge for refresh-capable agents **only** — a non-refresh-capable agent that ran a turn in Switchboard this session already advanced its own file with a live key of `None`, so re-applying its disk copy would duplicate the live turn (the live-vs-disk hazard the gate prevents). The overlay drops any journaled row whose `send_id` is already live in a slice, so a re-read of the journal can't double a this-session send. **Benign failure modes:** a `stat` false-positive → a safe, deduped no-op re-read; a file read mid-write → the per-line parsers ignore the half-written trailing line and pick it up next refresh. **Known limitations** (documented, not bugs): (1) refresh fires on project **switch-back**, not while you stay inside a project — there is no live file-watch; the switch covers the "go to the TUI, come back" path. (2) In the rare *mixed* case — a **non**-refresh-capable agent continued in the TUI at the same time as a refresh-capable one triggers the re-read — the wholesale-replaced overlay surfaces that agent's new imported TUI **prompt** while its **reply** is (correctly) skipped, leaving a prompt with no answer until the next full app restart (which re-hydrates everything safely, since no live turns exist at open). It is not cleanly avoidable: imported prompts are keyless (fresh per-parse id), so filtering "new" non-refresh imported prompts would also erase legitimate existing non-refresh imported history — the rare, transient, restart-healed orphan is the better trade for a Claude-only-at-launch feature.

**§3.1-status — the EOF tail turn's terminality (in-flight vs finished).** Claude's session file has no per-turn completion marker, so the disk parser infers the tail turn's status from the **last assistant `stop_reason`** (`claude_code/session_file.rs::eof_tail_status`). The documented final reasons — `end_turn` / `max_tokens` / `stop_sequence` / `refusal` / `model_context_window_exceeded` — mark `Complete`; the continuation reasons (`tool_use`, `pause_turn`) and any unknown/absent reason mark `Streaming` — a **positive allowlist**, so a new/unrecognized stop_reason errs toward "still running," never "falsely finished" (a negative "not `tool_use` ⇒ done" check would mis-classify `pause_turn`). The allowlist must carry *every* documented terminal reason, though — dropping one (e.g. `refusal`) renders a genuinely-finished turn as a permanent spinner; the canonical list is the [Anthropic stop-reason reference](https://platform.claude.com/docs/en/build-with-claude/handling-stop-reasons). An open tool (a `tool_use` whose result never bound — keyed on `output: None`, not the optional `completed_at` timestamp) also forces `Streaming`. Only the EOF **tail** turn is affected — every earlier turn is closed `Complete` by the following user prompt, which is definitive proof it ended. This is what lets a mid-flight re-read (switch-back/reopen while a turn streams, or a crash) render the tail as in-progress and dedup it against the live turn, instead of stacking a complete-looking turn with a perpetually-spinning tool. The disk parser can't distinguish "running right now" from "crashed/cancelled on a tool round" — both present as a trailing `tool_use` and both become `Streaming`; that residual is a status-display nuance resolved by the journal/dispatcher, not a dedup problem (the identity is stable either way). Asymmetric with Codex, whose per-turn `task_complete` marker lets *its* parser default a markerless tail to `Failed`.

**§3.1-provenance — user-prompt provenance (`promptSource`) and the housekeeping skip.** Claude's session file echoes the user's prompt back as a `type:"user"` record, so the merge must decide which of those the journal already owns (drop the harness copy) vs. which to render imported. Earlier this was inferred by **counting** harness turns against journal sends in order — which silently desyncs on long sessions, because the `user` channel also carries records that are **not** Switchboard sends, scattered through the history. Claude tags each `user` record with **`promptSource`**, a 4-value enum (corpus counts across all local sessions, 2026-06-21): **`sdk`** (1810 — dispatched by an SDK/headless client, i.e. Switchboard), **`typed`** (99 — typed into the bare TUI), **`queued`** (2 — typed into the TUI while a turn ran), **`system`** (1 — system-injected), and **absent** (3832 — older CLI with no marker; note this is the *majority*, so the absent→fallback path is the common one for historical transcripts, not an edge case). The parser (`claude_code/session_file.rs`) classifies each prompt into `UserPromptSource::{Sdk, External, Unknown}` (`sdk`→`Sdk`; `typed`/`queued`→`External`; absent/other→`Unknown`) on `Turn::User.source` (`#[serde(default, skip_serializing)]` — backend-private, never on the IPC wire, like `stable_message_id`). The merge (`commands.rs::classify_turns_by_provenance`) pairs slot-consuming `Sdk` prompts to journal sends (suppressing that copy) and renders `External` / `Unknown` / unpaired-`Sdk` / pre-journaling-`Sdk` imported; an agent with no positive provenance (all `Unknown`) falls back to the count path unchanged. **Housekeeping skip (load-bearing ordering):** before reading `promptSource`, the parser drops `user` *string* records that are not prompts — `isCompactSummary` / `isMeta` markers, and a maintained denylist of leading auto-generated tags (`<command-message>` ×713, `<command-name>` ×132, `<local-command-stdout>` ×100, `<task-notification>` ×101+, `<local-command-caveat>`, `<local-command-stderr>`). **`<system-reminder>` is absent from that denylist and does not need to be there** — Claude's harness reminders only ever appear as *array*-content blocks on `user`/`assistant` records, and the denylist is consulted only for `user` **string** records. Verified across 2.1.160→2.1.205: `message.role == "system"` has never been written to disk in any version, and 2.1.201's move of Sonnet 5 off the mid-conversation system role was an **API-wire** change with no session-file footprint. The invariant is load-bearing: a future CLI that emitted a reminder as *string* content would classify it as a real user prompt and render it as a typed send. Re-check on bump. The skip runs **first** because some housekeeping records carry `promptSource:"sdk"` (e.g. `<task-notification>` ×2) — mapping first would let them masquerade as a dispatched send and consume a journal slot, re-introducing the drift. A skip counter is logged at `finalize` so a denylist that stops matching is noticeable. **Bare slash-command reclassification (render/correlate split):** the denylist above catches the `<command-name>/compact</command-name>` *wrapper*, but Claude also writes the **bare** `/compact` back into the user channel — a lone slash-command string with **no `promptSource` and no `isMeta`**, so it slips past both `isMeta` and the prefix denylist and used to land as a real `Turn::User { Unknown }`. One such in-window `Unknown` prompt trips the straddling guard (below), abandoning provenance for the fragile count path, *and* the post-compaction continuation turn desyncs the count — the reported "latest prompt renders twice" bug. Fix (`is_bare_slash_command_record`): a `user` string record that is **provenance-less (`Unknown`) AND an exact match against a confirmed-command allowlist** (`KNOWN_BOOKKEEPING_SLASH_COMMANDS`, today just `/compact`) is routed to a non-correlating `SystemMarker::SlashCommand` — it **renders** (the user sees the command ran) but never consumes a send slot or advances a correlation counter (the render/correlate split, shared with the compaction summary marker). Both halves are load-bearing: keying on *absent* provenance (not `∉{typed,queued}`) keeps an `sdk`-dispatched `/compact` from the compose bar correlating, and an **exact allowlist** (not a `/token` shape rule) keeps both a genuine pre-`promptSource` prose dispatch (also `Unknown`) and a real lone-token prompt (`/review`) from being reclassified on a guess. The allowlist is confirmed ground truth (traced from the real auto-compaction session): the bare `/compact` record precedes the `compact_boundary` + `isCompactSummary`, and its `<command-name>` wrapper actually lands *seven records later* (after the summary), so a look-behind sibling-pairing would never fire — the allowlist is the honest ceiling of text-based detection. **Known limitations:** (1) a *manual* `/compact <focus…>` (compact with arguments) is **not** matched — indistinguishable from prose by any text rule; rarer than auto-compact, surfaced by the `commandish_not_reclassified` finalize breadcrumb, and backstopped by the M2 key-join. The real structural signal is the record's `parentUuid` chain (the auto-compact record is a child of the pre-compact `## Context Usage` hook), out of scope here. (2) This is Claude-specific; Codex/Antigravity bookkeeping shapes are audited separately. **Harness coupling** (gap-register class): this leans on Claude-internal fields that can change between CLI versions; the `Unknown` fallback bounds the blast radius to "no worse than the count path," and a `#[ignore]` live test (`live_claude_*`) asserts a dispatched prompt still writes `promptSource:"sdk"` so `make test-live` catches a rename. **Distinct shapes, same name:** the *live* `system/task_notification` **event** (subagent progress, G20 §4) and the on-disk `<task-notification>` **user-record** are different things — the former is a stream event mapped to `Liveness`, the latter is housekeeping the parser drops. Note the drop's boundary semantics have narrowed twice: **two** housekeeping shapes are **mid-turn continuations** (they land mid-dispatch and the process keeps responding), so they do not close the pending agent turn — (1) the `<task-notification>` record (G22; `is_task_notification_housekeeping` — a *typed* prompt merely starting with the tag remains a real prompt and boundary), and (2) any `isMeta` metadata row that is not a compaction summary (`is_meta_continuation`). The `isMeta` rule is deliberately **flag-based, not text-shape-based**: a genuine turn boundary is always marked by a real prompt record, so continuation is safe for the whole `isMeta` class, whereas matching on display text (`[Image: …]`) would silently resurrect the split when the CLI rewords the string. It rests on one invariant — **a dispatched prompt (`sdk`/`typed`/`queued`) never carries `isMeta`** (a violation would drop the prompt and merge the following turn backward) — guarded live by `live_claude_dispatched_prompt_never_carries_is_meta`, and the breadth is pinned by `non_image_meta_row_is_also_a_continuation`. The observed `isMeta` mid-turn shape is the `[Image: original WxH…]` description row Claude writes immediately after a `tool_result` whose content is an image — i.e. every time an agent reads a screenshot. Treating it as a boundary split one dispatched turn into several disk turns, each minting its own `hydration_key`; the live turn carries only the *first* fragment's key (`TurnIdentity` / `TurnEnd.first_message_id`), so on a switch-back refresh every later fragment failed the hydrate merge's key-dedup and rendered as a **duplicate turn** (and as fragmentation on plain reopen). Traced from a real duplication incident (2026-07-12): a 6-tool turn whose two screenshot reads split it into 1-, 1-, and 4-tool disk fragments with three distinct keys. `isCompactSummary` is explicitly excluded from the continuation: compaction is a genuine boundary whose continuation deliberately takes a distinct key — which means the compaction split still shares this live↔disk key-mismatch hazard by design (a live turn spanning an auto-compact re-reads as two keyed disk turns, and a refresh would surface the continuation as a new turn); tracked as **G28**. Every other housekeeping record remains a boundary. **Straddling sessions:** a file spanning the CLI version that introduced `promptSource` has both `Sdk` and journaled-region `Unknown` prompts; the merge routes those whole agents to the count fallback (an `Unknown` in the journaled region is a pre-marker dispatch the journal owns — importing it would duplicate it), so they're no worse than today but don't get the full fix. Detected as "any `Unknown` user turn at/after the agent's first journaled send"; pre-journaling `Unknown` (attached history) does not trigger it.

**Known limitation — a reload-stranded `Streaming` partial heals on next hydration, not live (deliberate, not a regression).** If the frontend reloads *mid-turn* (Vite HMR / `Cmd-R`), the backend keeps running the turn but the reloaded frontend has lost the dispatcher `turn_id`, so the turn's continuing live events are dropped (the reducer's unknown-`turn_id` guards) and a `Streaming` partial is left rendering a spinner. It heals on the **next hydration** — typically the next switch-back refresh after the turn completes, which supersedes the partial with the finished turn — **not live**. So a stuck spinner immediately after a reload is expected and self-corrects on the next project switch; it is not a regression in the in-flight-turn dedup work. Closing it *live* needs a dispatcher↔frontend resync that re-attaches the live stream to the hydrated turn (deferred "fix-c", out of scope for this work). This is distinct from the switch-back case, which the early `hydration_key` (§3.1-hydration) already collapses.

**Class B is *not* a gap** — we already re-read the session file at turn-end (Codex rate-limits, Codex model/context, Gemini tool output) and on project open, so the data is durable. The latency cost (one extra file read) is the trade.

**Class D is what no implementation can fix without external data** — e.g. Gemini does not emit a context window anywhere; Antigravity emits no token usage to disk. Document these as TUI parity ceilings; workarounds (hardcoded model→context maps) are case-by-case decisions.

Verified causal: flipping that field flips the model `agy -p` runs (self-reported + `USER_SETTINGS_CHANGE` transcript line). **But that file is harness-owned, and Switchboard never mutates harness-owned config** — so it's off the table. It is also **global, not per-agent** (one value for all Antigravity usage; concurrent agents would race), and per-child config-dir/`HOME` isolation was declined (HOME also anchors `agy`'s auth + brain dir, and authoring even an isolated settings.json is still producing harness config). **Net: Antigravity agents run on whatever model the user selected in Antigravity itself; Switchboard surfaces model selection as unavailable for this harness — a user-facing capability ceiling, not unbuilt work.**

*Display, however, is recoverable per turn.* Antigravity announces the model in a `USER_SETTINGS_CHANGE` transcript sentence both on turn 1 (`from None to X`) and on any later turn where the model **changes** — verified 2026-05-30: a resume after editing `settings.json` wrote `from Gemini 3.1 Pro (High) to Claude Sonnet 4.6 (Thinking)`. Unchanged resumes emit no sentence, so the per-turn model is reconstructed by **carrying the last announced model forward**. This works **live** (the sentence rides the turn's own record, so it lands at turn-end; hold the running value in per-agent state) and **on reopen** (walk the transcript chronologically). So Antigravity's per-turn model *history* is accurate even though its model *selection* is not ours to drive.

### 3.5 Session branching (`/branch` / `--fork-session`)

> **Status: Claude-only headless support. Codex interactive-only. Gemini/Antigravity unavailable.**

"Branch" means diverging the conversation tree at the current tip: a new session ID is created that carries the full prior context, but subsequent turns go into the new session and don't affect the original. The TUI equivalent is Claude Code's `/branch` slash command. **Note on naming confusion:** the CLI flag for this is `--fork-session`, not `--branch`. The TUI's `/fork` command is a different operation — it spawns a background agent that *inherits* the conversation rather than branching it (see G20 for the background-agent subagent channel). `--fork-session` is the headless equivalent of `/branch`, not of `/fork`.

| Harness | Mechanism | Headless (-p) | Wire signal | Session-file shape |
|---|---|---|---|---|
| **Claude** | `--fork-session` flag, used with `--resume <session-id>` or `--continue` | ✅ **confirmed** — `claude -p --verbose --resume <id> --fork-session "<prompt>"` works; turn completes and the new session file is written to disk | **No `parent_session_id` in `system/init`** — the forked session's init event carries only its own new `session_id`; the branch relationship is not announced on the stream | New `.jsonl` in the same project dir; records carry `parentUuid` links that cross back into the original session file, forming a contiguous tree. The fork point record in the new file has a `parentUuid` matching a `uuid` in the original session. The forked file also receives the full prior context as `attachment` records (~132 records for a 145-record original — verified 2026-06-17 @ 2.1.172). |
| **Codex** | `codex fork` subcommand | ❌ **interactive-only** — `codex fork` opens a session picker; `codex exec` has no fork subcommand or flag. No headless path. | — | — |
| **Gemini** | None observed | ❌ | — | — |
| **Antigravity** | Server-side automatic fork only — when a resumed conversation UUID no longer exists server-side, `agy` silently forks a fresh conversation and the adapter detects + re-captures the new UUID (the "fork-and-heal" in G15). No user-triggered fork. | ❌ | — | — |

**Switchboard implication.** Because `--fork-session` works headlessly, Switchboard can expose "Fork session" as an agent row action for Claude Code agents without requiring the user to leave the app. The command shape is `claude -p --verbose --resume <session_locator> --fork-session -- "<first prompt on fork>"`, producing a new agent whose `session_locator` is the new session ID from the forked session's `system/init` event. The Codex asymmetry (no headless fork) matches system-design §9's documented fork capability matrix. Gemini and Antigravity show an explanatory tooltip.

**`parent_session_id` is absent from the stream.** Switchboard has no wire signal that a session is a fork — the new session's `system/init` looks identical to any fresh session. The branch relationship lives only in the on-disk `parentUuid` chain. Switchboard does not need to read or track this chain; it is the harness's internal tree, not Switchboard's.

### 3.6 Tool vocabularies & facet mapping

> **Status: probed live 2026-07-10** (claude 2.1.206, codex 0.143.0, agy 1.0.16; gemini 0.44.0 **unprobeable** — see G26). Captures live under `crates/harness/tests/fixtures/<harness>/` (`tool-vocabulary*`, `file-change*`, `apply-patch*`). Each adapter maps its raw tool vocabulary to a normalized `ToolFacet` (`crates/harness/src/facets.rs`) at both call sites (stream parser + session-file parser); the raw `name`/`input` ride alongside untouched.

**Path absoluteness (probe checklist result).** Every observed file path is **absolute** in every harness: Claude `file_path`/`path`, Codex `apply_patch` section paths and `workdir`, Antigravity `TargetFile`/`AbsolutePath`/`SearchPath`. Adapters still normalize defensively (lexical join against the turn's cwd where one is known), but no harness currently requires it.

**Claude (2.1.206).** Live stream `tool_use` blocks and on-disk session records carry identical `{name, input}` — one classifier serves both. Observed shapes: `Bash {command, description}` → Shell; `Read {file_path}` → Read; `Edit {file_path, old_string, new_string, replace_all}` → Edit (one file, one pair); `Write {file_path, content}` → Write; `Grep {pattern, path}` / `Glob {pattern, path}` → Search; `TaskCreate {subject, description}` / `TaskUpdate {taskId, status}` → Todo; `mcp__<server>__<tool>` → Mcp. **Vocabulary drift found by this probe:** `MultiEdit` and `TodoWrite` no longer exist in 2.1.206 (requested via `--tools` and silently dropped) — `Edit` absorbed multi-edit, the `Task*` family replaced todos. Both legacy names are still classified (historical session files may contain them) from their documented shapes — the only unverified mappings in the set (G27). `Task` (subagent dispatch) is deliberately unmapped → Other.

**Codex (0.143.0).** Edits do **not** ride in shell heredocs (the pre-probe assumption). Two representations, asymmetric in content:

| Path | Record | Carries |
| --- | --- | --- |
| live stream | `file_change` item: `{id, changes:[{path, kind:"update"\|"add"\|…}], status}` | **paths + change-kind only — no content, no diff** |
| session file | `custom_tool_call` name `apply_patch`, `input` = patch text (`*** Begin Patch` / `*** Update File: <path>` / `@@` hunks / `*** Add File:` / `*** End Patch`); paired `custom_tool_call_output` by `call_id`; plus `event_msg` `patch_apply_end` with structured per-file `{type, unified_diff\|content, move_path}` | **full before/after content** |

So a live Codex Edit facet carries the file list with empty edit pairs; the content-bearing facet is parsed from the session file's patch text. The adapter closes the gap at turn end: the **existing** post-terminal enrichment re-read (already performed every Codex turn for per-turn usage) also extracts the current turn's patch facets and emits a facet upgrade keyed to the live tool ids (correlated ordinally with a path-set guard), so diffs appear seconds after the turn completes without a reload. Shell divergence (pre-existing, unchanged): live `command_execution` `command` is the wrapped string (`/bin/zsh -lc '…'`), disk `function_call` name `exec_command` `arguments.cmd` is the raw command with `workdir` — same Shell facet, different `command` spelling and `name`. **Before facets landed, Codex edits were invisible entirely**: the live parser skipped unknown item types (`file_change` → drop) and the session-file parser handled only `function_call`/MCP records (`custom_tool_call` → drop). Unobserved and left to degrade gracefully: a `delete` change-kind, `*** Move to:` sections (§5).

**Gemini (0.44.0).** Unprobeable — the CLI fails auth for individual accounts (G26). Facet classification is deliberately not built: both Gemini parsers stamp `Other`, which renders through the generic path exactly as before. The pre-sunset capture (`fixtures/gemini/tool-use.stream.jsonl`) shows `read_file {file_path}` / `update_topic {…}` under `tool_name`/`parameters` if mapping is ever revisited.

**Antigravity (agy 1.0.16).** Richly structured — six builtin tools, full content, absolute paths (supersedes the shell-only reading of the older `tool-use.transcript.jsonl` capture): `run_command {CommandLine, Cwd, WaitMsBeforeAsync}` → Shell; `view_file {AbsolutePath}` → Read; `replace_file_content {TargetFile, TargetContent, ReplacementContent, StartLine, EndLine, AllowMultiple}` → Edit; `write_to_file {TargetFile, CodeContent, Overwrite}` → Write; `grep_search {Query, SearchPath, IsRegex, CaseInsensitive}` → Search; `list_dir {DirectoryPath}` → Other. **Arg encoding trap:** in `transcript.jsonl` (the file the adapter reads) every arg value is a *string containing JSON* — `"StartLine": "1"`, `"TargetContent": "\"foo\""` — so the classifier JSON-decodes each value, falling back to the raw string. (The sibling `transcript_full.jsonl` carries raw-typed args but is not the adapter's source.) `tool_calls` ride on `PLANNER_RESPONSE` records identically live (tail) and on reopen — one classifier, both paths.

The prompt-correlation watch (`correlate_conversation_dir`) is retained as a **fallback** only — invoked if the Google-internal log line ever moves — so a CLI bump degrades to the older (racy but functional) path instead of hard-failing; `live_antigravity_cli_log_names_the_conversation` guards the log-line contract so that drift is caught before users see it. (The interim 30s timeout bump is gone — the timeout is removed entirely.) **Persistence target:** the captured conversation id is no longer written to a per-agent `.antigravity.jsonl` sidecar — the adapter emits a normalized `SessionLocatorCaptured` event and the dispatcher persists it to the agent's registry record (`AgentRecord.session_locator`) on first capture and on a fork-and-heal; a persist failure fails the turn. A logged-out / never-captured agent simply carries no locator, and new Antigravity agents write no session-link sidecar.
- ✅ **G16 — Antigravity ran the model's tools against `$HOME`, not the project dir.** **Closed by passing `--add-dir <cwd>`.** *Root cause:* setting the child's `current_dir(cwd)` is **not** enough to give `agy` a working directory for tools. With no "active workspace" established, `agy` reports "no active workspace set" and runs `run_command`/file tools with `Cwd` defaulting to `$HOME` — verified live (agy 1.0.3): a dispatch in a directory asked to read a file *in that directory* ran `cat <file>` from `$HOME` and couldn't find it. Affects every Antigravity agent's first turn (the workspace was never established at all). *Fix:* the adapter now passes `--add-dir <cwd>` on every dispatch (`agy` is one-shot, so the workspace is re-established each turn) — see `antigravity/mod.rs::build_args`. The interactive "Resume in terminal" command mirrors it (`resume.rs::interactive_resume_command` takes the cwd and adds `--add-dir` for Antigravity). Probed and confirmed: with `--add-dir <project dir>` the model reads project files; without it, it falls back to `$HOME`. Covered by `build_args_establishes_the_workspace_via_add_dir`, the resume unit test, and the live `live_antigravity_emits_tool_started_and_tool_completed_for_file_read`.

  **Known limitation — hidden working directories (deliberate won't-fix; see §7.3).** `agy` refuses to add a directory as a workspace when **any component of its path is hidden** (dot-prefixed): it logs `Failed to add workspace folder "<dir>": … is hidden: ignore uri` and proceeds workspace-less, so `--add-dir` is silently dropped and the model's tools fall back to `$HOME`. The turn still completes (exit 0, a reply) but the model is blind to the project — confusing-but-not-loud. Verified (agy 1.0.3, 2026-05-30) for both a hidden **leaf** (`~/.dotfiles`-style) and a visible leaf under a hidden **ancestor** (`~/.config/<proj>`, `~/.local/<proj>`). Normal project paths (`~/repos/foo`, `~/code/foo`) are unaffected — the overwhelming common case. **Decided: not guarded in code.** A pre-spawn path check would reimplement agy's hidden rule (drift risk), and a log-scan would cost a wasted dispatch; for a case this rare the guard's risk outweighs the reward. Documented here as the surfacing. (It is also why the live test uses a non-hidden tempdir prefix — `tempfile`'s default `.tmp` prefix is itself hidden.)

**Failure accuracy (Gemini):**
- ✅ **G17 — Gemini taints a complete turn `result.status:"error"`.** **Closed.** Gemini frequently (~3 of 5 tool-use runs, CLI 0.44.0, model `auto`) appends an empty/malformed trailing step *after* it has already streamed a complete answer, flipping `result.status` to `"error"` — the user saw a red FAILED chip on a perfect answer. Two parts to the fix in `gemini/parser.rs`: (1) the parser now captures the standalone `type:"error"` event's `message` (the terminal `result` line often has no `error` field at all), so a genuinely-failed Gemini turn surfaces a real reason instead of an empty chip; (2) when answer content streamed this turn AND the failure carries the benign `"empty response or malformed tool call"` signature (matched on the distinctive tail phrase, not the generic "invalid stream" prefix), the turn is reclassified `Completed`. The match is narrow and fails safe (signature drift → back to FAILED, never falsely-success). **RESIDUAL RISK:** a turn that genuinely failed *after* streaming only partial content, emitting this same benign signature, would be shown as success — accepted trade vs. a misleading FAILED chip on every other complete turn. Covered by parser unit tests, the fixture-driven `benign_trailing_error_fixture_completes_despite_result_error` (replays the real captured failing stream), and the live `live_gemini_streamed_answer_completes_despite_trailing_error`.

**Dispatch / argument construction:**
- ✅ **G18 — a prompt beginning with `-` broke dispatch on 3 of 4 harnesses.** **Closed.** A message whose first character is `-` (e.g. a markdown bullet) was parsed as a CLI flag and failed the turn *before any model call* — production-hit on a Codex `exec resume` (`unexpected argument '- '`; the harness exits with no terminal stream event → `AdapterFailure`). Probed across all four (2026-06-04): **Codex** (clap; prompt is a bare positional) and **Claude** (commander; `-p`/`--print` is *boolean* and the prompt is a positional) both abort (`unknown option '- …'` for Claude); **Gemini** (yargs; prompt is the `-p` *value*) aborts with `Not enough arguments following: p`; **Antigravity** (`agy`) tolerates a leading-dash value. The fix is **per-parser, not uniform**: Codex & Claude pass positionals after a `--` end-of-options separator (Claude's prompt moved to the **last** arg, since nothing flag-like may follow `--`); Gemini switches to the attached `--prompt=<value>` form (`--` can't guard an option *value*, only a positional); Antigravity unchanged. Shell metacharacters / leading newlines are **not** affected — the prompt is a single `Command::args` argv element with no shell in the path. See `codex/mod.rs::build_args`, `claude_code/mod.rs::build_args`, `gemini/mod.rs::build_args` (fixtures `fake_claude`/`fake_gemini` updated to match the new arg shapes; `fake_codex`/`fake_agy` unchanged). Guarded by `live_{claude,gemini,antigravity}_dash_leading_prompt_completes` plus the dash-leading turn-2 in `live_codex_resume_reuses_session` — drift tripwires, since this lives entirely in each CLI's parser and a bump can re-break it.
- ✅ **G21 — Antigravity long turns were cut at 5 minutes by `agy`'s built-in `--print-timeout`.** **Closed by passing an explicit `--print-timeout 24h`.** *Root cause:* `agy -p`'s `--print-timeout` **defaults to `5m0s`** and the adapter didn't override it, so every Antigravity turn silently inherited a 5-minute whole-turn wall-clock ceiling — **not** an initial-response wait (probe-confirmed 2026-06-20, agy 1.0.10: a turn that produced its first `PLANNER_RESPONSE` in ~3s and was still making progress was killed at the deadline, emitting `Error: timed out waiting for response` — exit 0, no terminal answer → `HarnessError`, §1.4). It bit long agentic turns — e.g. a PR review whose test suite pushes past 5 minutes — which surfaced to the user as a turn that "didn't hear back." **Antigravity-only:** `agy` is the sole one of the four CLIs with a built-in headless cap (Claude/Codex/Gemini expose no timeout flag and self-terminate on their own completion signal), and the cap contradicted the dispatcher's deliberate no-turn-deadline posture. *Fix:* the adapter now passes `--print-timeout 24h` on every dispatch (`antigravity/mod.rs::build_args`, const `PRINT_TIMEOUT`). `--print-timeout 0` is **not** a disable sentinel — it means *zero wait* and times out immediately (probed) — so a large finite value is required; `24h` never cuts a real turn yet still eventually reaps a truly-wedged `agy` rather than leaking it. Covered by `build_args_passes_generous_print_timeout` and exercised end-to-end by the `live_antigravity_*` turns (a name-drift guard on the flag). (An automatic backstop for a zero-progress wedge, if ever wanted, belongs in the dispatcher for *all* adapters, not an Antigravity-only timer — see the `antigravity/mod.rs` producer note.)

**Hydration robustness:**
- ✅ **G19 — hydration failures were invisible and unrecoverable.** **Closed (M1).** When `load_transcript` (per-agent) or `load_project_conversation` (per-project) failed, the error was `console.warn`'d and dropped — an agent showed an empty transcript with no explanation, and a failure (which means the on-screen history can't be trusted) was silent with no way to retry short of an app restart. Now every hydration failure is **first-class, surfaced, and recoverable**: the error text is retained on agent/project state (`AgentRuntime.hydration_error`; `ProjectConversationState.error`), rendered where the user is looking, and copyable verbatim for a bug report. Three surfaces share one shape (a reusable `ErrorDetailsDialog`, *not* a global error bus): **(a)** a whole-project conversation-load failure renders an in-transcript failed block with **Retry** + **Details**; **(b)** a project open/lock failure keeps the center-pane state and gains **Details**; **(c)** a per-agent history failure (one corrupt session file, or an attach-time load) renders pinned transcript-region chrome ("Couldn't load `<agent>`'s history", Retry + Details) — *not* an interleaved turn, since a failed agent contributes no turns to anchor against — plus the existing Sidebar line, now carrying Retry + Details too. **Retry** clears the relevant sticky guard (`hydrationAttempted` / `hydrationStarted`) and re-runs; a failed load applies *nothing* (all-or-nothing at the IPC boundary), so retry-after-failure cannot duplicate turns even before the idempotent-merge work. The backend error text was made path-bearing — `LoadTranscriptError::Io` now names the session-file path, not just the OS error — so the copied message identifies the file. **Known limitation:** retry re-reads from scratch (whole-file); incremental re-read is deferred.

- **G28 — a live turn spanning an auto-compaction duplicates its continuation on switch-back refresh (open).** The disk parser deliberately splits a turn at a compaction boundary — `SystemMarker::Compaction` between a pre-compact turn and a continuation with a **distinct** `hydration_key` (pinned by `compaction_split_gives_continuation_a_distinct_hydration_key`) — which is the right reopen rendering. But the *live* turn streams as ONE turn keyed by its first assistant message, so on a same-session switch-back refresh (`hydrateProject` re-merge) the pre-compact fragment collapses into the resident while the continuation matches no resident key and is prepended as a **new turn** — the same user-visible duplication class as the `isMeta` image-row split (fixed in §3.1), with a rarer trigger: auto-compact must fire mid-turn while the project stays loaded. Self-heals on app restart (reopen has no live resident to collide with). *Fix direction:* merge-side reconciliation — a compaction-aware collapse of a keyed disk continuation into the resident turn that spans it. Needs its own design pass: a naive time-span-overlap heuristic risks eating legitimate TUI-continued turns, which are exactly what the refresh exists to pick up. *Repro/pin when fixed:* a hydrate-reducer test feeding a resident live turn keyed `msg_A` + a disk re-read of `[turn(msg_A), Compaction marker, turn(msg_B)]`, asserting the continuation reconciles instead of prepending.

**Subagent / delegation surfacing:**
- **G20 — Claude's live subagent-progress channel is emitted by the CLI but not surfaced (low priority; needs a UX decision).** When a Claude turn delegates with the `Agent` tool, the parent's stream carries a structured per-subagent **progress channel** — originally three `system` event subtypes (the 2026-07-02 update below adds a fourth), **none tagged with `parent_tool_use_id`** (so they are not caught by the subagent-suppression rule below): **`task_started`** (`task_id`, `tool_use_id`, `description`, `subagent_type`, `task_type`, full `prompt`), **`task_progress`** (emitted repeatedly as the subagent works — `description` = its *current activity* e.g. `"Reading crates/core/Cargo.toml"`, `last_tool_name`, and `usage.{tool_uses, total_tokens, duration_ms}`, where **`tool_uses` is the running count**), and **`task_notification`** (terminal — `status:"completed"`, `summary`, final `usage`). Separately, each subagent-internal `assistant`/`user` record now also carries top-level `subagent_type` + `task_description` fields. This channel is exactly what the Claude **TUI** renders as the `Running N agents… | <desc> N tool uses` panel. **Switchboard surfaces none of it today** — a delegation shows as a single opaque `Agent` tool chip (with its `description`). Two independent reasons: (1) the subagent's own `tool_use`/`tool_result` records are **deliberately suppressed** by the conservative skip-on-`parent_tool_use_id` rule (avoids mis-attributing subagent work to the parent and keeps the live view matching the on-disk collapse — §6 "2.1.153 subagent" note); (2) the `task_*` system events carry no `parent_tool_use_id`, so they reach the system-event handler, which extracts only `system/init` — **every other subtype now maps to `Liveness`** (denylist posture, G22), so the channel keeps the frontend heartbeat armed through long background waits, but none of its *content* (descriptions, tool-use counts) is surfaced. **(Update 2026-07-02 @ 2.1.198:** the subtype vocabulary drifted — **`task_updated`** joined the family (observed alongside `task_progress`, which a trivial fast task never emits at all), a `task_notification` can carry **`status:"stopped"`** (a pending Bash-background task torn down at process exit), and an unrelated **`thinking_tokens`** subtype appeared; the fixed three-name inventory above is per-2.1.170. This drift is why the parser classifies by denylist, not by name — see G22.) **Why this is a decision, not just plumbing:** the channel is **live-only** — Claude does not persist subagent internals to the main session file (they live in a `<session-id>/subagents/agent-<id>.jsonl` sidecar we don't read), so any rich live panel would **collapse back to the single `Agent` chip on reopen**. That live-only-then-goes-stale asymmetry is the same reason we drop Gemini reasoning (§3.2). Surfacing it would need a new normalized progress event, per-subagent reducer state (a turn can run several at once), a nested transcript component, and an accepted answer for the reopen asymmetry. **Model-independent** — `claude-fable-5` and `claude-opus-4-8` emit byte-identical shapes (verified 2026-06-09 @ 2.1.170 on the same parallel-subagent review task; only the contents/counts differ). Recorded here so the harness data shape is known if/when delegation-progress display is prioritized.
- ✅ **G22 — Claude *background* agents fragmented the turn on reload and truncated it live.** **Closed (disk + live + forward).** *Upstream change:* when an `Agent`-tool task runs with `run_in_background: true`, its completion notification makes the same `claude -p` process keep responding — **N background completions → N+1 internal `init`→`result` cycles in one process**, with a **vocabulary split** between the two representations: the **stream** shows multiple `system/init` + multiple `result` events (plus the G20 task-lifecycle events), while the **disk** session file shows multiple `stop_reason:"end_turn"` records with a `<task-notification>…` **user-string record** (`promptSource:"sdk"`) between cycles. Probed 2026-07-02 @ 2.1.198 (sanitized captures: `docs/research/archive/claude-background-agent-*.jsonl`); probe-verified vs. inferred is marked in §6's 2.1.198 entry — the load-bearing facts (cycle structure, irregular result timing, cost cumulativity via the multi-model run) are all probe-verified; a *failed*-status notification remains unobserved (handling is status-agnostic, nothing hinges on it). *What broke:* **(disk)** every user-string record closed the pending agent turn, so one dispatch reconstructed as N+1 fragments — reload rendered one response as several messages, and the idle-source forward carried only the last fragment; **(live)** `parse_result` treated every `result` as the terminal and the producer stopped reading at the first one — the turn completed early, later cycles never streamed (unread pipe, potential pipe-full stall), the first cycle's telemetry was reported as the dispatch total, and a live forward truncated. **Result timing is irregular** (exit-batched *and* mid-stream interleaved both captured), and both in-stream boundary heuristics are refuted by the captures (a new `init` can follow a result-count/init-count match; an intermediate result can arrive with zero pending tasks) — **the only reliable terminal boundary is the stream ending.** *Fix:* **(disk)** `<task-notification>` records are mid-turn *continuations*: `session_file.rs::is_task_notification_housekeeping` (housekeeping-AND-prefix — typed/queued prompts starting with the literal tag stay real prompts *and* boundaries) early-returns before `close_current_agent` with no deferred-tool_result flush; **(live)** the turn ends at **process exit, gated on exit status**: successful `result`s fold into `ParserState::pending_completed_terminal` (kept-last = whole-dispatch) and surface as `Liveness`; `run_producer` reaps at stdout EOF and emits the folded terminal — `Completed` on clean exit, `Failed` (with the folded telemetry) on dirty exit, since an intermediate success is not proof the dispatch finished; error paths (malformed line, read error) also preserve folded telemetry; **(forward)** the disk join now inserts live's `\n\n` block separator (`forward.rs`), restoring byte-identical manual/workflow forwards; **(dispatcher)** post-terminal turn-scoped events are dropped at the boundary (`AdapterEvent::is_turn_scoped`, exhaustive match). *Telemetry contract:* **`TurnUsage` = whole-dispatch totals as billed** — token fields summed across the final result's `modelUsage` entries (subagent work included; camelCase, selection consistent with `select_context_window`), `total_cost_usd` from the harness's own cumulative top-level field (= Σ `modelUsage[*].costUSD`, probe-verified), occupancy unchanged (final parent call's context — the context bar is *not* an aggregate-token display). *Tests:* `session_file.rs` reunification/boundary suite, `forward.rs` separator suite + the cross-module `forward_of_reunified_dispatch_carries_all_cycles_text`, `claude_adapter.rs` `background_agent_*` fixtures (exit-batched from the real capture; interleaved synthesized with divergent per-result telemetry) + `exit1_after_*` gate tests + `malformed_json_after_success_result_keeps_folded_telemetry`, dispatcher `duplicate_terminal_from_adapter_is_dropped`, and the `live_claude_background_agent_completes_as_one_turn` drift guard. *Residuals (open, accepted):* (1) within-turn block granularity — live coalesces text into one `\n\n`-separated block, disk renders one `<Markdown>` block per item; visually near-identical, cosmetic; (2) a hard kill mid-background-work reads as `Complete` with partial content **on reload** (`eof_tail_status` sees the last cycle's `end_turn`; the *live* path now fails correctly via the exit gate, and while the app runs the hydrate merge protects the streaming turn — disk-side, post-kill only); (3) surfacing `task_progress`/`task_updated` descriptions as a background-activity hint (G20's open UX decision; only the heartbeat is wired); (4) **historical cost re-attachment** — a pre-fix background-agent turn persisted at most one overage-gated spend-sidecar entry keyed to the `stable_message_id` seen at its *first* result, which in interleaved orderings is a mid-dispatch id; after reunification that key no longer matches the turn's final-message key, so the cost badge silently stops re-attaching for that narrow historical slice (self-healing for all post-fix dispatches; documented, not migrated). No README entry — the fix is transparent to users, and a spinner that stays on through quiet background work is expected behavior, not a limitation.

### 7.1 Claude extended-thinking redaction & resume-wedge

All one root cause: since **v2.1.69** Claude Code requests the `redact-thinking-2026-02-12` API beta, which **strips the text from `thinking` blocks** (empty `thinking:""`, `signature` retained). Decompiled gate (#31326): `showThinkingSummaries !== true && getFeatureFlag("tengu_quiet_hollow", …)` — the flag is **server-controlled**, so the redact header ships unless the user sets `showThinkingSummaries: true`. Two consequences: (a) reasoning text is invisible to any programmatic consumer (us), and (b) the signed-but-empty blocks, re-sent on `--resume`, get a permanent `400`.

**Update 2026-06-01 (@ 2.1.159) — the redaction is now per-model.** The server flag has demonstrably moved: **`claude-sonnet-4-6` returns full, non-empty reasoning text** (live `thinking_delta` + on disk), while **`claude-opus-4-8` (and `claude-fable-5`, @ 2.1.170) still redact to empty**. So consequence (a) is **lifted for Sonnet 4.6, still active for Opus 4.8 / Fable 5** — Claude reasoning is no longer universally unavailable; Switchboard surfaces Sonnet reasoning both live and on reopen via the `ThinkingWidget` (§3.2). Consequence (b) (the resume-wedge) was already verified not-exposed @ 2.1.157 (see table) and is unaffected by which model carries the text.

**Update 2026-07-01 (@ 2.1.197) — re-redacted: the flag moved back to universal.** Sonnet 5 shipped as the new default and the `sonnet` alias now resolves to `claude-sonnet-5`, which **redacts to empty like Opus** (0 `thinking_delta` chars, single `signature_delta`, 3/3 runs — `live_claude_sonnet_thinking_is_redacted`). Third flip in three probes (universal @ 2.1.157 → Sonnet-only @ 2.1.159 → universal @ 2.1.197), underscoring: **re-probe per-model on every bump — the gate moves.** No code change: the adapter's redacted path already surfaces `Liveness` live and nothing on reopen.

**Correction 2026-07-08 (@ 2.1.205) — "universal" was never true: Haiku 4.5 un-redacts.** The 2.1.197 note above concluded "no default-selectable Claude model returns reasoning prose." That was **wrong** — it generalized from a probe set of Opus / Sonnet / Fable and never probed **Haiku 4.5, which is a selectable model in Switchboard's picker** (`MODEL_OPTIONS`, `src/lib/agentSelection.ts`). Probed @ 2.1.205 with Switchboard's exact spawn flags, main-agent (not sidechain), 3/3 runs: **Haiku streams 1756–7823 chars of `thinking_delta` and persists non-empty `thinking` blocks to the session file**, while Opus 4.8 and Sonnet 5 both stream 0 chars. Corroborated independently in local transcripts (a `claude-haiku-4-5` sidechain wrote a 467-char thinking block @ 2.1.198). So consequence (a) is **model-scoped, not universal**: a Haiku agent in Switchboard renders reasoning in the `ThinkingWidget` today. **No code change** — the adapter has always handled both paths; only the documentation was wrong. **Coverage:** `live_claude_haiku_thinking_is_not_redacted` now pins the un-redacted path (previously untested live — when the `sonnet` alias moved to Sonnet 5, the Sonnet guard was inverted to assert redaction, silently leaving the non-empty `Thinking` mapping and its disk reconstruction with no live coverage). **Opus/Fable still have no guard.** **Fable 5 could not be re-probed** @ 2.1.205 (gated behind additional usage credits); its redacted status is carried forward from 2.1.197 and is therefore unverified at this version.
