# Git view — implementation plan

A parallel-track feature: a top-level **Git view** that sits alongside the existing Projects view, giving the user a single-glance picture of their git repos → branches, with each branch's local status, which branches are checked out in a worktree (and where), the ability to create worktrees (optionally spinning up a Switchboard project in the new worktree), and a read-only diff inspector. Branches are the primary unit; a worktree is shown as an attribute of the branch that's checked out in it.

## Design principle

Every contract and UI decision in this feature answers to one rule:

> **Clarity at a glance for someone who uses Git but isn't a Git expert and runs AI agents across branches and worktrees. Surface what they need to _know_ and _act on_; resist GitKraken-style busyness.**

Concretely, the questions a user is trying to answer — in priority order — are:

1. **Are there uncommitted changes here?** (work in progress, not yet saved to a commit)
2. **Is this done — merged into the default branch?** (the cleanup signal: safe to delete)
3. **Is my work backed up / pushed?** (local-only / N unpushed / in sync / behind)
4. **Is this branch stale — its remote was deleted?** (the "dangling" cleanup signal)

When a contract or UI choice is ambiguous, pick the option that answers one of these questions more clearly, and prefer fewer, plainer states over exhaustive git fidelity. This is why the read model below uses an explicit, named `SyncState` rather than raw ahead/behind `Option`s: each state maps 1:1 to a badge a non-expert understands.

## How to read this doc

This is a standalone implementation plan for a feature that is **not** part of the v1 milestone roadmap (`2026-05-12-v1.md`). It does not block, and is not blocked by, any v1 milestone. It is sequenced as its own arc of five milestones, each independently demoable and reviewable.

The plan is written for an implementing agent that will **read the referenced code before acting but was not part of the design conversation.** Everything in the conversation that can't be recovered from the codebase is captured here; everything recoverable from the code (naming, local structure, which helper to reuse) is intentionally left to the agent's judgment against the actual code.

## Scope

**In scope (v1 of this feature):**

- A top-level **view toggle** between "Projects" (existing) and "Git" (new).
- A read-only **tree**: tracked repos → branches, with each branch's status badges (uncommitted changes, sync-vs-remote, merged / not merged, dangling) and, where a branch is checked out, its worktree path + the two warning states (orphaned worktree, prunable record). **Both local and remote branches** are surfaced, visibly distinguished, with a toggle to show local-only / remote-only / both.
- **Project linking**: each checked-out branch's worktree shows which Switchboard projects (if any) live in that directory.
- **Repo registry**: a user-global tracked-repo list with an explicit "Add Repo" action, auto-population when a Switchboard working directory that lives in a git repo is added, and a "Remove from view" action that touches only the list.
- **Non-destructive actions**: create worktree (the one write operation), open in terminal / editor / Finder, copy path/branch name, reveal repo in Finder, manual refresh (global and per-repo), background fetch for sync state (global and per-repo).
- **Create-worktree flow** with an optional "also create a Switchboard project here" checkbox wired to the existing project-creation path.
- **Read-only detail panel**: changed-files list + inline syntax-highlighted diff.

**Explicitly out of scope (deferred):**

- **All destructive git operations**: `git worktree remove`, `git worktree prune`, branch deletion. The tree *surfaces* the prunable/orphaned warning states (so the user isn't confused), but offers no one-click destructive remedy in v1. Recorded as a known limitation per milestone.
- Commit / push / pull / stash / merge / rebase — anything that mutates history or the index.
- GitHub/GitLab integration (PRs, issues, CI status), branch-to-branch comparison, clone/init.
- Filesystem watching / interval polling for live refresh. v1 is **staleness-gated refresh on view entry + manual refresh** — no background poll, no fs-watch (see M3 decision D3).
- Agent-tagging on worktrees (the standalone plan's future item — Switchboard's project-linking supersedes the need in v1).
- **Submodule, stash, and Git-LFS status** are not computed or shown (see M1 known limitations).

### Relationship to system-design non-goals

## Architecture decisions (made in the design conversation)

These are load-bearing and chosen over alternatives; they must survive into the code (as the new crate's module docs and as comments at the decision sites), not just live here.

1. **New pure-Rust crate `crates/git/` (`switchboard-git`), no Tauri dependency.** Mirrors `core`/`harness`/`dispatcher`. All git logic is unit/integration-testable without Tauri. The `app` crate depends on it and exposes it via `#[tauri::command]` shims, exactly as it does for the other crates.

2. **`git2` for local reads; shell out to the `git` CLI for fetch and worktree creation.** Reads (branches, worktrees, status, ahead/behind, merged-check, diffs) are pure-local and fast through `git2` — no credentials, no network. **Fetch** and **`git worktree add`** are shelled out via `tokio::process::Command` because (a) fetch needs the user's configured credential helpers / SSH agent, which `git2`'s credential callbacks reproduce poorly and fragilely, and (b) `git`'s own error messages for a failed worktree add are better than library-level errors and must be surfaced verbatim to the user. State this split in the crate's module doc; it's the single most important design choice here.

3. **Sync state is computed locally; fetch only refreshes the inputs.** A branch's sync-vs-remote (ahead/behind its upstream) is a local `git2` computation (cheap, offline) — it's simply *stale* until a fetch updates `refs/remotes/...`. So: the status read never blocks on the network; a separate, slower background fetch (shelled out, graceful-degradation on failure) keeps the remote-tracking refs current so the next local read is fresh. The result is reported as a single explicit `SyncState` (see M1) — `InSync` / `Ahead(n)` / `Behind(n)` / `Diverged` / **`LocalOnly`** (no upstream configured — a clear, common, fine state, **not** an error) / `Unknown` (genuinely uncomputable, rare). `LocalOnly` is deliberately distinct from `Unknown`: a clean never-pushed feature branch is "local only," not "couldn't determine." This mapping is the contract the badge UI depends on.

4. **Merged-check is local and against the repo's default branch.** "merged" = the branch tip is an ancestor of the default-branch tip. Detect the default branch via `refs/remotes/origin/HEAD`, falling back to a local `main`, then `master`; if none resolves, `merged` is `None` ("couldn't determine" — absent badge). Document the fallback order at the call site.

5. **The git view has its own user-global tracked-repo registry, separate from `workspace.yaml`.** A repo tracked for *visibility* is a superset of directories that *host Switchboard projects* — overloading `workspace.yaml` would blur its meaning. The registry is a new user-global file, **`git-view.yaml`**, resolved through the same `directories`-crate + `SWITCHBOARD_CONFIG_DIR` mechanism as `workspace.yaml` (a sibling file in the same config dir). It holds *data* (the ordered list of tracked repos), structurally a twin of `workspace.yaml` — distinct from `config.yaml`, which holds *preferences* (decision 9). **Auto-sync is one-directional:** adding a working directory to the Switchboard workspace appends its discovered repo root to the git registry; adding a repo directly to the git view does **not** create a Switchboard workspace entry or any project. Removing a repo from the git view touches the registry only — never files, never `workspace.yaml`. **Removing a working *directory* from the workspace does NOT remove its repo from the git view** (the one-directional rule run the other way): the git view is a superset, so a tracked repo persists until the user explicitly removes it *there*. This is intentional, not a leak — workspace churn must not make the git view flicker.

5a. **Repo identity is the canonical main-repo root; everything dedups on it.** A git repository can present as many on-disk folders that are the *same* repo: the main worktree, any subdirectory of it, and every linked worktree (separate folders sharing one git database). Whenever a repo enters the registry — by auto-sync *or* by explicit "Add Repo" — the input path is resolved via git's own discovery to the **canonical main-worktree / common-dir root**, and that root is what's stored and deduped on. Consequences: (a) adding a subdirectory, the main worktree, or any linked worktree of the same repo all collapse to **one** registry entry; (b) two worktrees of one repo render as a *single* repo with two branches (decision 10's model made literal), never two top-level entries. Note the registry's stored root and a Switchboard *working directory* may legitimately differ — a working dir can be a linked-worktree folder while the git-view entry is the repo root; they answer different questions ("where do projects run" vs. "which repo to show"), and project-linking (decision 7) reconnects them at render time. A path not inside any git repo fails discovery — that's the explicit "Add Repo" rejection.

6. **Only the *list of tracked repos* persists; branch/worktree/status data is always read live.** Unlike `workspace.yaml` (which caches project snapshots for offline display), the registry caches no git state — git data is cheap to recompute and dangerous to show stale. An unavailable repo (path missing/unmounted) renders as unavailable with a remove action, mirroring the workspace's unavailable-directory pattern, but carries no cached tree.

7. **Project linking is computed in the backend by *exact* path-matching.** The aggregate command cross-references each worktree's on-disk path against the loaded workspace's directory entries (and their projects) and returns the linkage, so the frontend renders it directly. Path comparison is **exact** (worktree path == project's working-directory path) using the same canonicalization boundary as the rest of `app` (`canonicalize_boundary`) — **not** at-or-under subdirectory matching. A Switchboard project whose working directory is a *subfolder* of a worktree is not linked to it in v1 (recorded as a known limitation); exact-match keeps the linkage unambiguous and is the common case (a project's working dir *is* the worktree).

8. **Slow git work runs off the async runtime's worker via `spawn_blocking`.** `git2` is synchronous and `status`/diff on a large repo can take noticeable time. The aggregate read command is an async Tauri command whose `git2` work is offloaded with `tokio::task::spawn_blocking`; fetch and worktree-add use `tokio::process::Command` directly.

9. **Backend-consumed preferences live in a new user-global `config.yaml`; pure-frontend settings stay in the frontend.** This feature introduces the first *backend-persisted* user preferences (editor command, terminal choice, worktree base path, diff style — used by the backend's open-in-editor / worktree-add commands and read at render time). There is no preferences file today: the only existing setting is **theme**, which lives in frontend `localStorage` (`src/lib/theme.svelte.ts`) and is consumed *only* by the frontend (it toggles a CSS class). The rule going forward: **a setting lives where its consumer is** — backend-consumed or cross-surface settings → the new backend-owned `config.yaml` (the personal-preferences file system-design §3 reserves); pure frontend-render settings (theme) → the frontend store, unmoved. `config.yaml` is built properly now (M3), with the same graceful-degradation load/save plumbing as `workspace.yaml`: a missing/corrupt file degrades to defaults, never an error. Theme is **not** migrated — moving a frontend-only setting behind IPC would add a round-trip for zero benefit; revisit only if theme ever needs to be backend-readable or synced across machines.

10. **Branches are primary; a worktree is an attribute of a checked-out branch; status is two-level.** The user thinks in branches ("is *this branch* done / pushed / dirty?"), and most branches aren't checked out in a worktree at all. So **branch-level status** (sync-vs-remote, behind-base, merged-into-default, dangling) is computed for *every* branch, and **worktree-level status** (uncommitted changes, orphaned/prunable warnings) only for the branch that's checked out somewhere. Both local and remote branches are first-class in the read model, visibly distinguished in the UI. This is a deliberate departure from a worktree-centric model that would leave most branches blank.

10a. **Remote branches carry the two cleanup signals, not the full local set.** A remote-tracking ref (`origin/*`) gets `merged` and `behind_base` — both are pure ancestry walks against the default branch, computable on any commit at the same cost as for a local branch, and they answer the "is this stale `origin/*` branch already merged / far behind main — safe to delete?" cleanup question (design-principle questions 2, 4). It does **not** get `SyncState`, `dirty`, `worktree`, or `dangling`: those are meaningless for a ref with no working tree, no own-upstream, and no deletable upstream. So "first-class" for remote branches means navigable + the two cleanup badges — not the full local field set.

11. **One read-model, many scopes.** M1's read-model is the single source of git state for the whole feature. The big Git view (M3) renders all of it; the project-scoped right-sidebar panel and the "out of date" project indicator (M6) render a *filtered slice* of the same data for the one worktree a project lives in. There is no second, "reverse" data path — only the same `RepoView` viewed at a narrower scope. **Behind-base** is the branch-level signal that answers "main has moved on — my branch is stale": the count of commits the default branch has that this branch doesn't (the mirror of the merged-check's ancestry walk). It is distinct from `SyncState::Behind`, which is behind the branch's *own* upstream (`origin/<this-branch>`); behind-base is behind the *default* branch. Like sync state, it is most accurate after a fetch (the relevant comparison is against `origin/<default>`); locally-only it compares against a possibly-stale local default branch — treat staleness the same way (decision 3).

## M1 — Git read layer (`crates/git/`)

### Goal & Outcome

A pure-Rust crate that, given a path on disk, produces the full read-model the UI needs — with no Tauri dependency and no UI.

Once complete:

- Given any path inside or at a git repo, the crate resolves the **repo root** (main worktree / common dir) and the repo's **default branch**.
- It lists all **local branches** and all **remote branches**, kept distinct, and pairs each local branch with the worktree it's checked out in (if any). **Detached-HEAD** worktrees are grouped separately, labelled by short commit hash.
- For each **local branch** it computes branch-level status: **`SyncState`** vs. upstream (`InSync` / `Ahead(n)` / `Behind(n)` / `Diverged` / `LocalOnly` / `Unknown`), **behind-base** (`Option<u32>` — commits the default branch has that this branch lacks; the "main moved on, you're stale" signal; `None` when the default branch can't be resolved), **merged** vs. the default branch (`Option<bool>` — `None` only when the default branch can't be resolved), and **dangling** (the branch had an upstream that no longer exists — the remote branch was deleted).
- For each **remote branch** it computes **only** `merged` and `behind_base` (the cleanup signals — decision 10a); the local-only fields don't apply to a remote-tracking ref.
- For the branch checked out in a **worktree**, it computes worktree-level status: **dirty** (tracked changes, staged or unstaged) and **untracked** (separately).
- It flags the two **warning states**: orphaned (worktree on disk whose branch was deleted) and prunable (git has a worktree record whose directory is gone).
- It produces, for a single worktree, the list of **changed files** with per-file change kind, and the **unified-diff text** for a given file (consumed by M5).
- A **bare** repo root (no working tree of its own — the `git clone --bare` + worktrees layout) is handled gracefully: its branches and worktrees are listed normally; the bare root simply reports no working-tree status of its own (`is_bare: true`). It is **not** marked unavailable.
- An **unavailable** repo path (missing / unreadable / not a git repo) yields a clearly-marked "unavailable" result rather than an error.

### Implementation Outline

Create the workspace member with `cargo new --lib crates/git` (then wire it into the workspace; add `git2` with `cargo add`, **never** by hand-editing — per AGENTS.md). Name it `switchboard-git` to match the `switchboard-*` convention.

The public surface is a small set of read functions plus the data types they return. The exact field set is the agent's to finalize against `git2`'s API, but the **contract shape** below is load-bearing — the UI and the command layer (M2) are designed around it, so keep these distinctions:

```rust
// Illustrative — the distinctions are the contract (branch-primary; two-level
// status; SyncState as an explicit enum, not ahead/behind Options; local vs.
// remote branches kept distinct; bare-root handling; warning states); exact
// names/fields are the implementer's to settle against git2.
pub struct RepoView {
    pub root: PathBuf,                       // main worktree root (or the common dir for a bare root)
    pub name: String,                        // folder name of root
    pub default_branch: Option<String>,
    pub available: bool,                     // false => path missing/unreadable/not-a-repo; tree empty
    pub is_bare: bool,                       // bare hub repo: no working-tree status of its own
    pub local_branches: Vec<BranchView>,
    pub remote_branches: Vec<RemoteBranchView>,
    pub detached_worktrees: Vec<WorktreeView>,
}
pub struct BranchView {
    pub name: String,
    pub upstream: Option<String>,            // the remote branch it tracks, if any
    pub sync: SyncState,                     // vs. own upstream; replaces ahead/behind Options
    pub behind_base: Option<u32>,            // commits default branch has that this lacks ("main moved on")
    pub merged: Option<bool>,                // None only if default branch unresolved
    pub dangling: bool,                      // had an upstream that no longer exists ("gone")
    pub worktree: Option<WorktreeView>,      // Some => checked out in a linked/main worktree
}
pub enum SyncState {                         // each maps 1:1 to an at-a-glance badge
    InSync,
    Ahead(u32),                              // unpushed commits
    Behind(u32),
    Diverged { ahead: u32, behind: u32 },
    LocalOnly,                               // no upstream configured — "not pushed", a fine state
    Unknown,                                 // genuinely couldn't compute (rare)
}
pub struct RemoteBranchView {
    pub name: String,                        // e.g. "origin/feature-x"
    pub merged: Option<bool>,                // already in default branch? — "stale remote, safe to delete"
    pub behind_base: Option<u32>,            // commits default branch has that this remote ref lacks
    // No SyncState / dirty / worktree / dangling: meaningless for a remote-tracking ref
    // (it IS the remote; it has no working tree, no own-upstream, can't be "gone").
}
pub struct WorktreeView {
    pub path: PathBuf,
    pub dirty: bool,                         // tracked changes, staged OR unstaged
    pub untracked: bool,                     // separate signal; UI may fold into "uncommitted"
    pub warning: Option<WorktreeWarning>,    // Orphaned | Prunable
}
```

Note the status split (decision 10): `SyncState` / `behind_base` / `merged` / `dangling` are **branch-level** (computed for every branch, checked out or not); `dirty` / `untracked` / warnings are **worktree-level** (only present when a branch is checked out). `LocalOnly` vs. `Unknown` is a load-bearing distinction (decision 3): a clean never-pushed branch is `LocalOnly`, not `Unknown`. `behind_base` (vs. the default branch) is distinct from `SyncState::Behind` (vs. the branch's own upstream) — see decision 11.

**Remote branches carry `merged` + `behind_base`, nothing else (decision 10a).** `merged` and `behind_base` are pure ancestry walks against the default branch — they work on *any* commit, including a remote-tracking ref, at the same cost as the local-branch computation. They directly serve the cleanup story for a stale `origin/*` branch ("already merged / fallen far behind main — safe to delete?", design-principle questions 2 and 4). The other branch-level fields are **deliberately absent** for remote refs because they're meaningless there: a remote-tracking ref has no working tree (`dirty`/`worktree`), no own upstream to be ahead/behind of (`SyncState`), and can't have a deleted upstream (`dangling`). So remote rows are navigable *and* carry the two cleanup signals; they are not name-only.

Order of work: (1) repo discovery + default-branch detection (decision 4's fallback order, documented at the site) + bare-root detection; (2) local + remote branch enumeration and worktree pairing; (3) per-local-branch status (`SyncState`, behind-base, merged, dangling), per-remote-branch status (merged, behind-base only — decision 10a), and per-worktree status (dirty, untracked); (4) warning-state detection; (5) changed-files + per-file diff text for M5.

Decisions to honor (from "Architecture decisions"): `SyncState` and `merged` are **local** computations (decisions 3, 4); fetch and worktree-add do **not** live in this crate's M1 surface — fetch is M3, worktree-add is M4, both shelled out from `app`. Keep this crate read-only in M1.

**Read granularity (decision 8, refined for M1Q4).** Expose **two** read functions over the same machinery: a **single-repo read** (`RepoView` for one repo root) and an **aggregate read** (all tracked repos, used by M2's command). The single-repo read powers per-repo refresh and the "fetch this repo, then re-read just it" path so a per-repo refresh never re-walks every other repo; the aggregate powers initial load, global refresh, and the poll. Tree structure and status stay **together** in one `RepoView` for v1 (simple; repos are usually modest) — M3 bounds the poll cadence rather than splitting the read.

Use `thiserror` for a typed error at the crate boundary, per the Rust conventions. Distinguish "this path isn't a git repo" / "path unavailable" (both expected, non-error UI states represented in `RepoView`) from genuine errors (a corrupt repo, an I/O failure mid-read).

### Definition of Done

- **Integration tests** (Cargo `tests/`, the crate's public API) against fixture repos built in a `tempfile` tempdir. Construct, with `git2` or by shelling `git` in the test harness: a clean branch in sync with its upstream; a branch with a worktree that is dirty; one with untracked-only files; one ahead and one behind an upstream; a **diverged** branch (both ahead and behind); a merged branch and an unmerged branch; a **dangling** branch (upstream deleted); a detached-HEAD worktree; an orphaned worktree (delete the branch, keep the dir); a prunable record (remove the dir, keep git's record). Assert the computed `RepoView` for each.
- **`SyncState` cases are distinct assertions:** a branch with **no upstream configured** yields `SyncState::LocalOnly` (**not** `InSync`, not `Unknown`); a branch ahead-and-behind yields `Diverged`; a branch whose default branch can't be resolved yields `merged = None`. These are the "right badge, not a misleading one" contracts the UI depends on.
- **`behind_base` is distinct from `SyncState::Behind`:** a branch behind the **default branch** but in sync with (or ahead of) its **own upstream** reports `behind_base = Some(n)` *and* `SyncState::InSync`/`Ahead` — the two signals are independent. A branch whose default branch can't be resolved yields `behind_base = None`.
- **Local and remote branches are returned distinctly** (a remote branch appears in `remote_branches`, not `local_branches`), and a local branch's `dangling` flag is set when its upstream is gone.
- **Remote branches carry `merged` + `behind_base` (decision 10a):** a remote ref already in the default branch reports `merged = Some(true)`; one behind the default reports `behind_base = Some(n)`. (No `SyncState`/`dirty`/`worktree`/`dangling` on remote rows.)
- **Default-branch detection covers both the clone and the `git init` case (decision 4):** a repo with `origin/HEAD` resolves via it; a `git init` repo with **no `origin/HEAD`** resolves the default via local `main` (the common local case — the fallback is the primary path in practice); a repo where **no default resolves at all** yields `merged`/`behind_base = None`. All three are explicit fixtures.
- A **bare repo root** lists its branches/worktrees and reports `is_bare: true`, **not** `available: false`.
- An **unavailable path** yields `available: false` with an empty tree and no error.
- Changed-files + diff-text functions return correct output for a worktree with staged, unstaged, and untracked changes.
- Both the **single-repo** and **aggregate** read functions exist and return consistent `RepoView`s for the same repo.
- Crate module doc states the **`git2`-reads / shell-out-writes** split (decision 2), the default-branch fallback order (decision 4), and the branch-primary two-level status model (decision 10). No live-network test (fetch is M3).
- **Known limitations recorded** (in the crate doc): submodule status, stash count, and Git-LFS state are intentionally **not** computed; bare roots report no working-tree status of their own.

## M2 — Repo registry, project linking, and Tauri commands

### Goal & Outcome

The backend surface the frontend will call: a persisted tracked-repo list, the auto-sync hook, project linking, and the commands that return the full tree.

Once complete:

- A **user-global tracked-repo registry** (`git-view.yaml`) persists the ordered list of **canonical main-repo roots** (decision 5a) across launches, degrading to empty on a missing/corrupt file (mirroring `Workspace`).
- Adding a working directory to the Switchboard workspace **auto-adds** its repo's canonical root to the registry (one-directional; decisions 5, 5a). Adding a subdirectory or a linked worktree resolves to the **same** root and dedups (decision 5a).
- **Removing a working directory does not remove its repo from the git view** (decision 5) — the repo persists until explicitly removed there.
- Commands exist to **add** (explicit "Add Repo") and **remove** tracked repos; remove touches only the registry.
- **Add Repo accepts any path inside a git repo** (subdirectory or worktree included), resolving to the canonical root; only a path **not inside any git repo** is rejected with a clear inline error (decision 5a).
- A **list-repos aggregate command** returns, for every tracked repo, the `RepoView` from M1 enriched with **which Switchboard projects live in each checked-out branch's worktree path** (decision 7, exact match), with unavailable repos marked. This is the **sole read command** — there is no separate cheap "list repos" command (decision 9-of-the-review: it had no consumer; the aggregate already returns per-repo availability, and Add-Repo returns the resolved root, so a separate list is unmeasured speculative surface — re-add only if the aggregate proves slow in practice).
- All git reads run **off the async worker** (decision 8).

### Implementation Outline

**Registry storage.** Add a registry type in `app` modeled on `workspace.rs` (ordered list, private vec + accessor, `Serialize`/`Deserialize`, graceful-degradation load). Persist it to a new user-global YAML file, **`git-view.yaml`**, resolved by a function parallel to `workspace_config_path()` in `lib.rs` — same `directories`-crate base dir and same `SWITCHBOARD_CONFIG_DIR` debug override, a sibling filename. Add a field to `AppState` for the in-memory registry and (if needed) its path, following the existing `workspace` / `workspace_path` shape; add a `persist_*` helper paralleling `persist_workspace`.

**Lock-order doc — document the real nesting, don't copy the loose framing (review finding #5).** The new mutex is acquired **standalone** by the read commands, but its one mutating acquisition (auto-sync) happens *inside* `init_directory_impl`, which holds `registry_write` for its whole body — so it nests **under `registry_write`**, exactly as `workspace` already does there (`commands.rs` acquires `registry_write` at the top, then `workspace` mid-body). Document its order position accordingly — **after `registry_write`** — with an explicit invariant: *no path may acquire `registry_write` while holding the git-registry mutex* (the inverse order is the deadlock the convention exists to forbid). Do **not** restate the existing doc's "acquired alone / only ever standalone" framing — that wording is already slightly inaccurate for `workspace` (which nests under `registry_write` in `init_directory_impl`); correct it for `workspace` while you're there rather than propagating it to a second lock.

**Root resolution (decision 5a) is the single identity chokepoint.** Add one helper — "given any path, resolve to the canonical main-repo root via git discovery (`git2`), or `None` if not in a git repo" — and route *both* auto-sync and explicit-add through it. This is the dedup key; the registry stores only resolved roots. (Conceptually parallels `canonicalize_boundary` for directories, but resolves repo identity, not directory identity.)

**Auto-sync hook.** In `init_directory_impl`, after the directory is registered in the workspace, run the root-resolution helper on the canonical path and, if it resolves, append the root to the git registry (dedup on the resolved root) and persist. This is the only write to the registry that isn't user-initiated. A non-git directory simply doesn't resolve — skipped, no error.

**Commands** (each a `*_impl` free function + a thin `#[tauri::command]` shim in `lib.rs`, errors stringified at the boundary exactly as the existing commands do):

- **add a tracked repo**: run the root-resolution helper; on `None` reject with a clear typed error ("not a git repository") for the inline-error UX; otherwise dedup on the resolved root and persist. Accepts subdirectories / worktrees (they resolve to the root).
- **remove a tracked repo** (registry only; never touch files or `workspace.yaml`).
- **aggregate repos** (the heavy one): for each tracked repo, compute the M1 `RepoView` and enrich each checked-out branch's worktree with linked projects. **Project linking** (decision 7): cross-reference each worktree path against `AppState.workspace` directory entries (and their cached projects) / loaded projects, matching **exactly** on `canonicalize_boundary` (no subdirectory matching). Return a wire type that nests projects under worktrees. Make this command **async** and wrap the `git2` work in `tokio::task::spawn_blocking` (decision 8). **Provide the single-repo variant too** (per decision 8 / M1's two-read split), powering per-repo refresh without re-walking every repo.

**Wire types.** Add TypeScript discriminated-union / record types in `src/lib/types.ts` mirroring the Rust serde shapes (snake_case keys, the `Option` fields as `T | null`), and `invoke` wrappers in `src/lib/api.ts`, following the existing conventions. No frontend rendering yet (that's M3) — but landing the types + api wrappers here keeps M3 purely presentational.

### Definition of Done

- **Unit tests** on the registry type: add/dedup/remove/order-preservation; missing-file and corrupt-file both load to empty (not error) — the graceful-degradation contract.
- **Integration tests** on the `*_impl` functions (the established `app` test style, targeting free functions):
  - **Root resolution / dedup (decision 5a):** auto-sync appends a repo root when a git directory is added and is a no-op for a non-git directory; adding a **subdirectory** of an already-tracked repo, and adding a **linked worktree** of it, both resolve to the same root and do **not** create a second entry.
  - **Add** rejects a path not inside any git repo with a typed error; accepts a subdirectory/worktree by resolving to the root.
  - **Remove** leaves `workspace.yaml` and the on-disk directory untouched.
  - **One-directional removal (decision 5):** removing a *working directory* (via `remove_directory_impl`) leaves the corresponding repo in `git-view.yaml`.
  - **Linking (decision 7, exact):** the aggregate returns correct project linkage for a worktree path that exactly matches a workspace directory hosting a project, and **empty** linkage for a project whose working dir is a *subfolder* of the worktree (the documented non-link case).
- The aggregate command marks an **unavailable** repo correctly and does not fail the whole call when one repo of several is unavailable (partial success — one bad repo never blanks the view).
- Lock-order doc in `state.rs` updated to place the new mutex **after `registry_write`** (its real auto-sync nesting), with the "no path acquires `registry_write` while holding the git-registry mutex" invariant stated and the stale `workspace` "acquired alone" wording corrected (review finding #5).
- **Known limitations recorded** (in the registry type's doc): registry stores resolved roots only, git data is always live (decision 6); project linking is exact-match — a project in a subfolder of a worktree is not linked (decision 7).

## M3 — Git view: tree, status, refresh, fetch, open-actions

### Goal & Outcome

The read-only view itself — the core value of the feature.

Once complete:

- A **view toggle** in the title bar switches between "Projects" and "Git". The mode is **session-only**: Switchboard always opens to Projects; the Git view is never the persisted landing surface in v1. The Projects experience is unchanged when toggled away.
- The Git view is a **center-pane takeover** (rendered the way Settings is — `AppShell` has no right slot; a top-level view replaces the center pane). The **left Projects sidebar is hidden** while in Git view. The repos→branches **tree is one full-width scrollable area** in the main pane (repos as collapsible top-level nodes), not squeezed into a narrow sidebar — branch rows carry a lot (name, local/remote marker, badges, worktree path, linked projects), and a full-width tree reads better. M5's detail panel will split *within* this pane.
- The tree renders **tracked repos → branches**. Branches checked out in a worktree are emphasized (and show the worktree's path + uncommitted-changes state + **linked Switchboard projects**); branches without a worktree still show their branch-level status. **Local and remote branches are visibly distinguished**, with a toggle for local-only / remote-only / both. A separate toggle hides inactive branches.
- Each branch shows **at-a-glance badges** mapped from the M1 read model (the badge mapping is decision-D2 below) — chosen to answer the four design-principle questions, not to mirror every git detail.
- **Warning states** (orphaned, prunable) render distinctly (not hidden) — but with **no destructive remedy** in v1 (decision: surface, don't fix; record as known limitation).
- **Add Repo** (native folder picker → resolve-to-root → appear, or inline error) and **Remove from view** (context menu, registry-only) work. Unavailable repos render marked, with remove.
- **Refresh** is **staleness-gated on view entry, with no interval polling** (decision-D3 below): entering the view re-reads any repo whose data is stale and kicks a background fetch for any repo whose fetch is stale; manual **global** and **per-repo** refresh/fetch always force. A subtle per-repo "last fetched Xm ago" indicator shows recency; fetch failure degrades silently (sync state falls back), never a toast.
- **Open actions** on a checked-out worktree (open in editor, open in Terminal, reveal in Finder, copy path, copy branch name) and repo (reveal in Finder).

### M3 design decisions (resolved)

- **D1 — Layout: center-pane takeover, full-width tree.** The Git view replaces the center pane (like Settings); the Projects sidebar hides; the tree is a single full-width scroll area, not a master/detail squeeze. M5's detail panel splits within this pane.
- **D2 — Badge → token mapping (calm, one warning tier).** The palette has **no "success/green" role** and the rule forbids introducing a hue without a named token — so the mapping leans on existing tokens and stays quiet:
  - **Clean & in sync** → **no badge** (absence = fine; the calm default).
  - **Uncommitted changes** (dirty/untracked) → **`warning`** (amber).
  - **Behind base / "out of date"** → **`warning`** (amber) — main moved on, actionable.
  - **Orphaned / prunable / dangling** → **`warning`** (amber). **One warning tier** in v1 — no separate "severe" role; revisit only if the single tier proves ambiguous.
  - **Ahead / behind / diverged counts** (vs. own upstream) → **neutral `Badge` chips** with text (e.g. `↑3`, `↓2`) — informational, *not* colored, so counts don't compete with the amber attention states.
  - **Local-only** → **muted text** (`local`) — a fact, not a warning.
  - **Merged** → **muted/neutral label** (`merged`) — a calm "done / safe to delete," not a loud color.
  - **Remote branch rows** (decision 10a) carry only the two cleanup signals: `merged` → the same muted `merged` label; `behind_base` → the same amber `warning`. No sync/dirty/dangling badges on remote rows (they don't carry that data).
  - Net: **no new token** introduced. If the tree later needs a hue with no existing role, add a named role to `app.css` (never a raw Tailwind color).
- **D3 — No polling; staleness-gated refresh on entry; per-repo in-memory timestamps.** There is **no interval poll** — the view is an honest point-in-time snapshot that only changes on entry or manual refresh. On entry: re-read any repo whose **last local read** is older than **~30s** (the local `git2` walk is cheap/offline), and kick a **background fetch** for any repo whose **last fetch** is older than **~1m** (non-blocking — the local read paints immediately; the fetch updates sync/behind-base when it returns). First-ever entry reads/fetches everything. Manual **global** and **per-repo** refresh/fetch always force, ignoring timestamps. Timestamps are **per-repo, in-memory, not persisted** (reset on restart — a fresh process reads fresh); they also drive the "last fetched" label. Thresholds are tunable; revisit during implementation.
- **D3a — Fetch fan-out is bounded and deduped (no unbounded subprocess burst).** Entering the view with N stale repos must **not** spawn N concurrent `git fetch` processes. Add (1) a **per-repo in-flight fetch guard** — a fetch already running for repo X makes a second request (e.g. a manual fetch overlapping the auto-fetch) a no-op that joins the first, never a second subprocess racing on the same `refs/remotes/...`; and (2) a small **global concurrency cap** (semaphore of ~3–4) so a dozen-repo workspace fetches in bounded waves. Reuse the existing in-flight-promise-map shape (`loadStarted` / `hydrationStarted` in `workspace.svelte.ts`) rather than inventing one. This matters because fetch touches credential helpers / SSH agents — an unbounded burst means overlapping auth prompts and agent contention.
- **D3b — Fetch state is modeled `{never | failed | ok(timestamp)}` and the failure is shown (quietly).** Fetch failure stays **no-toast** (calm utility), but the per-repo fetch state distinguishes **never fetched / last fetch failed / fetched at T** — this is correctness, not UX: if the only signal is "last fetched Xm ago" and it can't tell *failed* from *fresh*, a globally-broken fetch environment (e.g. `git` not on the GUI app's PATH — the class of bug fixed in commit `906ed0d`) looks identical to a healthy one and the staleness/sync badges silently lie. Render `failed`/`never` as a **muted "fetch failed" / "not fetched" label in place of the timestamp** — quieter than a toast, shown only when something is actually wrong.
- **D4 — Open-in-editor uses an editor-command preference (blank = OS default).** The worktree "open in editor" row action runs `<editor-command> <worktree-path>` (e.g. `code`, `cursor`, `zed`); a **blank** preference falls back to the OS folder-open (macOS `open <path>`), so it works with zero config. No app-picker UI. (This is opening the *worktree folder* in an external editor — distinct from M5's in-app read-only diff, which opens nothing externally.)
- **D5 — View-mode state lives in the workspace store** (testable), session-only (above).

### Implementation Outline

**Preferences infrastructure (net-new — built here).** This milestone stands up the first backend-persisted preferences (decision 9): a `Preferences` type in `app` and a user-global **`config.yaml`** resolved alongside `git-view.yaml` / `workspace.yaml`, with the same graceful-degradation load/save plumbing (missing/corrupt → defaults, never an error) and a `persist_*` helper. There is **no existing backend prefs pattern to copy** — `theme` lives in frontend `localStorage` and is *not* migrated (decision 9). Add `get_preferences` / `set_preferences` commands (or per-key setters) following the M2 command pattern, plus `api.ts` wrappers and `types.ts` wire types. M3 introduces the **editor-command** + **terminal** keys; M4 adds worktree base path; M5 adds diff style — all into this same file.

**View toggle (D5).** Add a `"projects" | "git"` view-mode to the workspace store (`src/lib/state/workspace.svelte.ts`) — *not* component-local like today's `settingsOpen` — so it's testable. Branch on it in `App.svelte`'s center snippet (the Git view replaces the center pane, hiding the Projects sidebar; D1) and add the toggle control to the title bar. Do not disturb the existing Projects/Settings/Welcome branching — add a sibling, don't refactor the structure. Mode is session-only (not persisted).

**The tree component (D1, D2).** Build it from the existing primitives per `ui-conventions.md` — `StatusDot`/`status-*` and `warning` tokens for status, neutral `Badge` for count chips and incidental labels, `DropdownMenu` for the context menu (never hand-roll a menu), `EmptyState` for the no-repos case. Apply the D2 badge mapping; **every color names a semantic token** — add a named role to `app.css` only if a genuinely new hue is needed (D2 expects none).

**Refresh model (D3) — no polling.** No interval timer. On view entry, run the staleness check over the registry: stale-local repos get a re-read via the **single-repo read** (M1/M2's two-read split, so one repo refreshes without re-walking the others); stale-fetch repos get a background fetch. Global manual refresh re-runs the **aggregate**; per-repo refresh re-runs the single-repo read; both ignore timestamps. Fetch is shelled out from a new `app` command (`tokio::process::Command`), offered globally and per-repo, and **bounded + deduped** (D3a): a per-repo in-flight guard collapses overlapping requests, and a global semaphore (~3–4) caps concurrency so a many-repo workspace fetches in waves. Maintain the per-repo in-memory `last_read` timestamp and the `{never | failed | ok(timestamp)}` fetch state (D3b); failure degrades the sync data silently but the state surfaces as a muted "fetch failed"/"not fetched" label — never an error toast.

**Open actions (D4).** Folder picker reuses the existing dialog plugin (`tauri-plugin-dialog`, already a dependency — see `init_directory` flow). Reveal-in-Finder / open-in-Terminal / open-in-editor and copy-to-clipboard use the Tauri shell/clipboard surface (`tauri-plugin-clipboard-manager` is already a dependency). Open-in-editor runs the `config.yaml` editor command (blank → OS default); open-in-Terminal opens the system terminal at the path. Add the editor-command + terminal preference fields to `SettingsView`.

**Component-level tests are required** (per AGENTS.md's rule for components that wrap `invoke` + reactive state): mock `invoke`, exercise the tree render, the local/remote and inactive-branch toggles, the add/remove/global+per-repo-refresh actions, the badge mapping, and the unavailable-repo and empty states.

### Definition of Done

- View toggle switches cleanly both ways; the Git view takes over the center pane and hides the Projects sidebar; Projects view is unaffected when toggled back. Mode is session-only (a restart opens to Projects).
- Tree renders repos/branches as a full-width tree with correct emphasis (checked-out vs. not), the local/remote distinction + toggle, the inactive-branch hide toggle, **badges per the D2 mapping** (clean → no badge; dirty/behind-base/orphaned/prunable/dangling → `warning`; counts → neutral chips; local-only/merged → muted labels; **remote rows → only `merged`/`behind_base` badges**, decision 10a), and linked-project labels on checked-out worktrees.

- Add Repo: a path inside any git repo resolves to the root and appears; a non-git path shows an inline error; a duplicate (incl. subdir/worktree of a tracked repo) is a no-op. Remove: repo leaves the view; `workspace.yaml` and disk untouched (assert in a backend test if not already covered by M2).
- **No interval polling exists.** On entry, a stale repo (local > ~30s) re-reads and a stale-fetch repo (> ~1m) fetches in the background; a fresh repo is served from the last result; manual global/per-repo refresh always forces. **Per-repo refresh** updates one repo without re-reading the others.
- **Fetch fan-out is bounded (D3a):** entering with N stale repos spawns at most the concurrency cap simultaneously; a manual fetch on an already-fetching repo does **not** double-spawn (joins the in-flight one). Assert both in a state/component test.
- **Fetch failure is visible-but-quiet (D3b):** a failed or never-run fetch shows a muted "fetch failed"/"not fetched" label distinct from a fresh timestamp (not a stale-but-fine timestamp); no toast. A no-remote repo degrades sync/behind-base gracefully.
- **Fetch shell-out has automated coverage:** an `#[ignore]`-gated smoke test fetches a local `file://` remote, exercising the subprocess + ref-update path without network or credentials (git isn't a harness, so AGENTS.md's `live_<harness>_` convention doesn't apply — this is the substitute).

- **Component tests** (mocked `invoke`) cover render, toggles, actions, the badge mapping, and the unavailable/empty states; **reducer/state tests** cover the new view-mode state. **Unit tests** on the `Preferences` type cover the graceful-degradation load (missing/corrupt → defaults). ## M4 — Create-worktree flow (+ optional Switchboard project)

### Goal & Outcome

The one write operation in v1: create a worktree — for a **new** branch or an **existing** one — and optionally spin up a Switchboard project in it in the same action. This is the headline agent workflow: "make a branch + a folder + an agent and turn it loose."

Once complete:

- **Two launch points** (D1): a repo-level **"New worktree"** action opens the dialog in **new-branch mode** (the default); an **existing branch with no worktree** has a per-row **"Create worktree"** action that opens the dialog in **existing-branch mode** pre-targeted at that branch.
- The dialog has a **branch field with two modes** — *new branch* (type a name; runs `git worktree add -b <name> <path>`) or *existing branch* (pick from the repo's branches; runs `git worktree add <path> <branch>`) — and an editable **destination path** pre-filled from the base-path preference.
- On confirm, Switchboard runs the appropriate `git worktree add` (shelled out); on success the new worktree **appears in the tree**.
- On failure (e.g. branch already checked out, `-b` name already exists, destination path occupied), the **git error message is surfaced verbatim** inline — not swallowed (decision 2).
- The dialog includes an **"Also create a Switchboard project in this directory"** checkbox (offered, **checked** by default); when set, after the worktree is created Switchboard runs the existing add-directory + create-project + agent-seeding path against the new worktree directory, then **switches to Projects and activates the new project**.

### M4 design decisions (resolved)

- **D1 — Support both new-branch and existing-branch worktrees; new-branch is the default.** The two are different git invocations (`-b <name>` vs. a plain existing-branch add) and serve different needs — new-branch is the primary "start fresh work for an agent" flow; existing-branch resumes a branch that already exists (last week's branch with no current worktree, a teammate's/agent's pushed branch, parallel review). The backend picks the invocation from the mode; it's a single conditional. Launch points map to modes (above).
- **D2 — Destination default: `<base>/<repo>-<slug>`, editable; base default `~/worktrees/`.** The base-path preference (M3's `config.yaml`) defaults to `~/worktrees/` (keeps worktrees out of the repos folder, predictable), and the dialog pre-fills `<base>/<repo-name>-<branch-slug>`. Slug = branch lowercased with `/` and whitespace → `-` (e.g. `feature/foo bar` → `feature-foo-bar`). The path is fully editable. A path that already exists is **not** auto-deduped — git refuses and its error is surfaced verbatim (D-error).
- **D3 — Partial-success sequencing; never roll back the irreversible step.** Order: (1) backend `create_worktree` (irreversible) — on failure, surface git's stderr verbatim, nothing else ran; (2) if "also create project": the **frontend** `createProjectAndActivate(name, worktreePath)` runs. If step 2 fails *after* step 1 succeeded, **keep the worktree**, **refresh the Git read-model so the new (project-less) worktree is visible**, and show a clear partial-success message — "Worktree created at `<path>`, but the project couldn't be created: `<err>`." (Without the refresh the worktree exists but the user may not see it until a manual refresh — review finding #1 nuance.) Because step 1 is a prior independent committed call, the partial-success contract falls out for free: there is nothing to roll back.
- **D4 — Optional project: name defaults to the branch slug, editable; agents auto-seed via the FRONTEND path (review finding #1).** When the checkbox is set, the project name pre-fills from the branch slug (the worktree *is* that branch's workspace), editable. Project + agent creation reuses the **frontend** `createProjectAndActivate` → `seedAgentsForInstalledHarnesses` path — there is **no backend agent-seeding function** (`create_project_impl` creates a project with zero agents; seeding lives only in `workspace.svelte.ts` and targets the *active* project). So the flow is: backend `create_worktree` returns → frontend `createProjectAndActivate(name, worktreePath)` (which `init_directory`s, `create_project`s, **activates**, then seeds). **Activate-before-seed is a correctness invariant, not just UX** (D5): seeding targets the backend-active project, so the new project must be activated first or agents scatter / fail into `agentCreationFailures`. Checkbox defaults to checked.
- **D5 — Landing after create.** If a project was created → **switch to Projects view and activate it** (the user came here to work in it; drop them in, ready to message the agent), reusing `activateProject`. If no project (checkbox off) → **stay in Git view**; the tree refreshes and the new worktree appears under its branch.

### Implementation Outline

**Backend command.** Add a `create_worktree` `*_impl` + shim. It shells out the correct invocation via `tokio::process::Command` against the repo root (decision 2): **new-branch** → `git worktree add -b <name> <path> [<start-point>]`; **existing-branch** → `git worktree add <path> <branch>`. The command takes the mode + branch name + destination path. On non-zero exit, return a typed error carrying git's **stderr verbatim** so the frontend can show it inline. On success, the next aggregate/single-repo read reflects the new worktree (no need to synthesize it). This is a **write that does not need the `git2` crate** — keep it in `app` as a shelled command, consistent with the fetch command from M3.

**Optional project creation — frontend composition (D3, D4; review finding #1).** This is **frontend orchestration, not a backend single command** — there is no backend path that seeds agents (agent-seeding is `seedAgentsForInstalledHarnesses` in `workspace.svelte.ts`, coupled to the active project; `create_project_impl` makes a project with no agents). The flow: call backend `create_worktree`; on success, call the **existing** `createProjectAndActivate(name, worktreePath)` (it `init_directory`s the new worktree dir — which per M2 auto-sync resolves to the *same* already-tracked repo root, so **no duplicate registry entry**, decision 5a — then `create_project`s, **activates**, and seeds agents). Reuse `createProjectAndActivate` as-is; do not duplicate or hand-roll a parallel backend seeding path (a regression the plan warns against). On a step-2 failure, the prior `create_worktree` success stands: **refresh the Git read-model** so the project-less worktree shows, and surface the partial-success message (D3). The **activate-before-seed ordering inside `createProjectAndActivate` is load-bearing for correctness** (D4/D5) — don't reorder it.

**Worktree base path preference (D2).** Add the "default worktree location" key to the `config.yaml` preferences M3 stands up (not a new mechanism), and a field for it in `SettingsView`. Default `~/worktrees/`. Used with the `<base>/<repo>-<slug>` template to pre-fill the destination.

**Dialog.** Build on the `Dialog` primitive (use `dismissible={false}` while the create is in flight, mirroring the New Project dialog during agent seeding, so the user can't navigate into a half-created state). Branch field with new/existing mode toggle (pre-set by the launch point, D1); destination path editable and pre-filled (D2); the project checkbox + editable project name (D4); a Create button; an inline error region that renders git's verbatim stderr on failure.

### Definition of Done

- **Integration test** on `create_worktree_impl`, both modes: a **new-branch** add creates the branch + worktree on disk that the M1 reader then sees; an **existing-branch** add checks out the existing branch into a new worktree; a failing add (branch already checked out, `-b` name exists, or destination occupied) returns a typed error whose payload contains git's stderr verbatim.
- **Integration test** on the optional-project path: checkbox-on creates a worktree *and* a project bound to it (assert via the existing project listing) with agents seeded; a project-create failure after a successful worktree-add yields a **partial-success** result with the worktree intact (no rollback). Confirm the new worktree's repo does **not** create a duplicate git-registry entry (resolves to the already-tracked root, decision 5a).
- **Component test** (mocked `invoke`): dialog opens in the correct mode per launch point; pre-fills branch + default path (+ project name when checkbox on); surfaces a verbatim git error inline on failure; on success either lands in the activated new project (checkbox on, D5) or refreshes the Git tree to include the new worktree (checkbox off). **Project-create failure after worktree success** (checkbox on): the partial-success message shows *and* the Git read-model is refreshed so the project-less worktree is visible (review finding #1). Agent seeding occurs **after** activation (assert the order, since it's correctness-load-bearing).
- Worktree base-path preference persists to `config.yaml` and pre-fills the dialog.
- Known limitation recorded: no worktree **removal** in v1 — a worktree created here can only be removed outside Switchboard (or via a future destructive-ops milestone).

## M5 — Worktree detail panel + read-only diff

### Goal & Outcome

Inspect a worktree's working changes without leaving Switchboard.

Once complete:

- Clicking a worktree opens a **detail panel** as a **bottom split within the Git view's pane** (D1) — tree on top, panel below — so the diff gets full window width. The **divider is draggable** to resize the split.
- The panel header shows branch name, full clickable path (opens in Finder), and the status badges.
- A **changed-files list** shows each file with its change kind (modified / added / deleted / untracked).
- Clicking a file shows its **diff inline**, syntax-highlighted, **read-only**, in **side-by-side or unified** layout (D4) — a toggle, persisted, defaulting to **side-by-side** (the bottom split makes the panel wide).

### M5 design decisions (resolved)

- **D1 — Bottom split, draggable.** The detail panel splits the Git view's center pane horizontally (tree above, panel below), giving the diff full width (better for long lines and side-by-side). The divider is **draggable** to resize. The split ratio is **session-only** (in-memory, not persisted) — matching how the existing sidebars are session-state, avoiding a low-stakes pref key.
- **D2 — `diff2html` for structure, existing Prism for intra-line highlighting.** `diff2html` (added via `pnpm add`) renders the diff structure and the unified/side-by-side layouts from one unified-diff input. Intra-line **syntax highlighting routes through the existing Prism path** (`markdown.ts`'s `highlightCode` — export/reuse it; `prism-diff` and the language grammars are already imported) rather than diff2html's bundled `highlight.js` — **no second highlighter**, consistent colors. Suppress diff2html's default CSS.
- **D3 — New `diff-added` / `diff-removed` token pair.** The palette has a red-ish role (`status-failed`) but **no green**, and a diff's add/remove line backgrounds are the one place green-for-added is near-universal — a legitimate place to add a named role (the rule: never a hue without a token). Add a matched **`diff-added` / `diff-removed`** pair to `app.css` (each a strong fg + a soft line-background, light + dark mappings, like the other token groups). This is the **only** new token the whole feature introduces. Line backgrounds use this pair; intra-line code tokens use the existing `syntax-*` group (D2).
- **D4 — Unified/side-by-side preference, default side-by-side.** The layout toggle persists in `config.yaml` (decision 9). Default **side-by-side** (the bottom split makes the panel wide enough). On a narrow window side-by-side just gets tight — **degrade gracefully** (columns get cramped; the user can flip to unified), no auto-switch.
- **D5 — Working-tree diff only.** The panel shows uncommitted working-tree changes for the worktree (modified/added/deleted/untracked, staged or unstaged) — *not* commit-to-commit or branch-to-branch comparison. Those (e.g. a PR-style "this branch vs. main" view, a natural pairing with the behind-base signal) are a recorded known limitation / future addition, out of scope for v1.
- **D6 — Diff HTML is DOMPurify-sanitized before render (security; review finding #3).** The diff renders **file contents** — i.e. whatever the user's agents wrote — as HTML, in a desktop webview with privileged APIs. A file containing `<script>` or `<img onerror=…>` must never execute. So all diff HTML (after diff2html + Prism processing) is passed through **DOMPurify** before `{@html}`, mirroring the existing `markdown.ts` contract, with an allowlist that preserves the structural/`diff-*`/`syntax-*` classes the rendering needs. diff2html escapes content by default, but this is the codebase's standing rule for a security-sensitive path — don't rely on a dependency default.

### Implementation Outline

**Backend.** Reuse M1's changed-files + per-file unified-diff-text functions; expose them through `app` commands (changed-files for a worktree path; diff-text for a worktree path + file). Async + `spawn_blocking` for the `git2` work, consistent with M2.

**Panel + split (D1).** Render the detail panel as a draggable bottom split inside the Git view's center pane. Split ratio is session-only component state (not persisted). Header shows branch + clickable full path (reveal in Finder, reusing M3's open action) + status badges.

**Diff rendering (D2, D3, D6).** Feed the backend's unified-diff text to `diff2html` for structure + unified/side-by-side layout; drive intra-line highlighting through the existing Prism `highlightCode`; theme line backgrounds with the new `diff-added`/`diff-removed` tokens and code tokens with `syntax-*`, suppressing diff2html's default palette per `ui-conventions.md`; **sanitize the final HTML with DOMPurify before `{@html}`** (D6). The diff view is **read-only** in v1 — no editing, no staging.

**Spike the diff2html + Prism marriage first (review finding #8).** D2's "diff2html structure + external Prism highlighting, no second highlighter" is the plan's most under-specified technical coupling — diff2html exposes no clean per-token external-highlighter hook, so reconciling its DOM with Prism token spans (post-process its output, or highlight each line through Prism and re-inject) is non-trivial. When M5 starts, **spike this before committing to diff2html**: if it resists, the known fallback is a hand-rolled unified/side-by-side renderer over the backend diff text + Prism (still sanitized per D6) — a deliberate option, not a late surprise that forces accepting a second highlighter.

**Diff-style preference (D4).** Add the unified/side-by-side key to `config.yaml` (M3's preferences infrastructure), default side-by-side, with a toggle in the panel.

**Edge cases (D5).** Binary files: listed with a "binary file" placeholder, no diff body. Very large diffs: render with a sane cap / lazy per-file so a huge generated file doesn't freeze the panel (exact cap is the implementer's call; `log`/note any truncation). Clean worktree: `EmptyState` ("No changes").

### Definition of Done

- **Integration test**: changed-files + diff-text commands return correct output for a worktree with staged, unstaged, and untracked changes (may reuse M1 fixtures through the command layer).
- **Component test** (mocked `invoke`): selecting a file renders its diff; the unified/side-by-side toggle switches layout and the preference round-trips (default side-by-side); an empty (clean) worktree shows an appropriate empty state; a binary file shows the placeholder, not a rendered body.
- The bottom split is **draggable** and resizes the panel; the ratio resets on a fresh session (session-only).
- Diff renders with the new `diff-added`/`diff-removed` line tokens and `syntax-*` intra-line highlighting via the existing Prism path (no second highlighter), correct in light and dark.
- **Diff HTML is sanitized (D6):** a component/unit test feeds diff content containing `<script>` and `<img onerror=...>` and asserts it renders as inert text, no execution (review finding #3).
- Read-only is enforced (no edit affordances).
- Known limitation recorded: diff is working-tree changes only (no commit-to-commit or branch-to-branch comparison in v1).

## M6 — Project-scoped git status (right-sidebar panel + "out of date" indicator)

> **General goals only — details deferred to the M6 design pass.** This milestone surfaces M1's read-model inside the *Projects* experience, scoped to the one worktree the current project lives in. It is **not** a new data path: it renders a filtered slice of the same `RepoView` the Git view already shows (decision 11).

### Goal & Outcome

Bring the git state of the project you're working in into the Projects experience, so the user sees "is my work in progress / pushed / stale vs. main?" without switching to the Git view.

Once complete (directional):

- While a project is active, the **right sidebar** shows a compact git panel for **that project's worktree** — the same badges as the Git view (uncommitted changes, `SyncState`, behind-base, merged, dangling), scoped to one worktree. A subset of the Git view, same data, narrower lens.
- An **"out of date" indicator** — driven by **behind-base** (the default branch has moved on relative to this branch) — is surfaced where the user sees it without opening the Git view. Candidate placements (a dot/badge on the project in the left Projects list; the right-sidebar panel; possibly the title-bar breadcrumb) are **decided at M6 design time**, guided by the clarity principle (prominent enough to act on, not cluttered).
- The indicator's accuracy rides on the existing fetch cadence (decision 11): behind-base is most meaningful after a fetch updates `origin/<default>`; a stale read is treated the same way the Git view treats stale sync state (decision 3), never shown as an error.

### Implementation Outline (sketch — to be detailed at M6)

- Reuse the M1 read layer and M2's project↔worktree linking in reverse: given the active project's directory, resolve its repo + the branch checked out there, and read that one branch's `BranchView`.
- Reuse M3's fetch cadence/commands; no new fetch machinery.
- Build the panel from existing UI primitives and semantic tokens, consistent with M3's badge mapping (reuse the same `SyncState`/behind-base/merged badge components so the two surfaces never drift).
- Component + reducer tests per AGENTS.md (mocked `invoke`), covering the badge mapping and the behind-base indicator's stale/absent states.

### Definition of Done (directional)

- The active project's right sidebar shows its worktree's git status using the **same** badge components as the Git view (no divergent rendering).
- The behind-base "out of date" indicator appears in its chosen placement, reflects backend `behind_base`, and degrades gracefully when behind-base is `None` or the read is stale.
- Known limitation recorded if the chosen v1 placement omits any of the candidate surfaces (e.g. "indicator on the project list only; breadcrumb deferred").
