# Per-agent model & effort selection

**Status:** proposed · **Created:** 2026-05-30 · **Revised:** 2026-06-04 (re-baselined against the session-locator consolidation that landed after the first draft — see the corrected anchors in M1/M3/M5/M6)

Let each agent run on a user-chosen **model** and **reasoning-effort level**. Both are picked when the agent is created, changed later (for harnesses that support them) from the agent's actions menu, shown as the agent's current intent in the sidebar, and recorded **per turn** in the transcript so a mid-conversation switch is visible in history. The two axes have **different per-harness capability sets** (below), so each is independently gated.

## Background — read before implementing

This plan is the product of empirical probes of all four harness CLIs. **Read `docs/harness-behavior.md` §3.3–§3.4 (model selection; reasoning effort) first** — they are the ground truth for the per-harness behavior this plan encodes. The essentials:

**Model** — `--model <alias|id>` (Claude), `-m/--model <id>` (Codex), `-m/--model <id>` (Gemini), and **nothing** for Antigravity (its model is a global, harness-owned config we never touch; its agents run on whatever was last selected inside Antigravity). Model is per-invocation, not configured once — Claude is session-sticky, Gemini reverts to its default without the flag, Codex re-derives per turn — so the rule is **send the flag every turn when set**. Model **values are not enumerable**: no harness has a usable "list models" command (`claude models` only makes the LLM recite a training-frozen table; the others have none), and Codex values are **plan-gated** (only `gpt-5.5` accepted on the probe account; out-of-plan ids 400 from the API, not from us). So model input is **free-text with hardcoded suggestions**, never a validated list.

**Effort / reasoning level** — a *separate* axis with a *different* capability set:
- **Claude** — `--effort <level>` flag. Valid: `low, medium, high, xhigh, max` (empirically enumerated). `max` is session-only and has a historical "flag ignored" bug ([#50099](https://github.com/anthropics/claude-code/issues/50099), v2.1.113) — re-check on current CLI.
- **Codex** — `-c model_reasoning_effort=<level>` config override (there is **no** `-e`/`--effort` flag — a web claim to the contrary was verified false). Valid: `none, minimal, low, medium, high, xhigh`. Recorded **per turn** in `turn_context` right next to `model`.
- **Gemini** — thinking is `settings.json`-only (`thinkingConfig`/`thinkingBudget`/`thinkingLevel`); **no CLI flag exists** (confirmed; open upstream feature requests for one). That's harness-owned config we never touch → **no effort control for Gemini.**
- **Antigravity** — the level is **baked into the model display name** (`"Gemini 3.1 Pro (High)"`, `"Claude Sonnet 4.6 (Thinking)"`); there is no separate effort axis, and we can't set the Antigravity model anyway.

### Capability matrix — what *we* can set per agent

| | Claude | Codex | Gemini | Antigravity |
|---|:---:|:---:|:---:|:---:|
| **Model selection** | ✅ `--model` | ✅ `-m` | ✅ `-m` | ❌ (config-only, off-limits) |
| **Effort selection** | ✅ `--effort` | ✅ `-c model_reasoning_effort=` | ❌ (config-only, off-limits) | ❌ (folded into model name) |

Note the mirror: Gemini gives a model flag but locks thinking behind config; Antigravity locks the model behind config but folds effort into the model name. Both end up with one axis we can't drive — different axis each.

### Two concepts per axis — keep them distinct

This is the central design idea, and it applies to **both** model and effort.

1. **Selected (intent)** — what the *user chose* for the agent. Lives on `AgentRecord.{model,effort}` (`Option<String>` each). Shown in the **sidebar / agent card**. Updates the instant the user changes it — immediate confirmation, no turn required.
2. **Per-turn (history)** — what *actually produced* a given turn, as the harness reported it. Lives on each **`Turn`** and renders in the **transcript footer** next to the timestamp, on **every** turn. Preserves history across a mid-conversation switch.

The two surfaces never compete: the sidebar answers "what will the next turn use," the transcript answers "what did each past turn use." This is why there is no "show both values on the card" reconciliation.

### Decisions locked

1. **Both axes are closed fixed-enum pickers (REVISED 2026-06-05 — was "model = hybrid free-text").** Model is a **curated per-harness picker** that renders as a segmented control for compact lists (≤4 options) and a dropdown for long lists (Gemini); effort is always a **curated per-harness segmented picker** because the labels are short. No free-text. Rationale for the change: the product wants every agent created with a **known, displayed** model (never an opaque "we don't know what it'll run"), and free-text is error-prone. The values aren't API-enumerable, so the lists are **hardcoded from live-verified probes** (`harness-behavior.md` §3.3/§3.4) and patched when models ship/sunset. Claude's model list uses **durable aliases** (`opus`/`sonnet`/`haiku`) — "latest of family", no maintenance — and the M6 footer shows the **resolved** id per turn (intent vs. history). Codex/Gemini use ids (`gpt-5.5`/`gpt-5.4-mini`; `auto` + `gemini-*`). No free-text ⇒ no `Some("")` footgun at the form (the M3 backend normalization stays as defense). Codex `none` is still a *real* effort level distinct from "no override."
2. **Validation: reactive.** We don't validate values server-side beyond the structural capability check (an Antigravity agent can't carry a model; a Gemini/Antigravity agent can't carry an effort). A bad/out-of-plan model surfaces as a normal failed turn with the harness's message. Effort is UI-constrained to its enum, so it has no footgun; for model, an empty/whitespace custom entry must normalize to "unset" at the form boundary, never persist as `Some("")` (which would dispatch `--model ""` and fail every turn).
3. **Always pass an explicit value when *creating* a capable agent (REVISED 2026-06-05 — was "unset = no flag"; attach clause REVISED 2026-06-05 during M5 — see below).** The create form **preselects a default and always submits** a model+effort for Claude/Codex/Gemini, so a freshly-created `AgentRecord.{model,effort}` is `Some` for them and the agent's model is known/displayed from creation. Passing no flag is *not* a nameable default (it's session-sticky / config-resolved / Antigravity-global — see `harness-behavior.md` §3.3), which is exactly why we don't rely on it. The same defaults seed the auto-created agents on a new project. The dispatch rule (M2: "send when `Some`") is unchanged; the UI just always produces `Some` for a *created* capable agent. Preselected create defaults: Claude `opus`/`high`, Codex `gpt-5.5`/`medium`, Gemini `auto`.

   **Attach is the exception — it defaults to "keep current" (`None`), not a preselect.** The original "attach preselects the session's hydrated model" had no implementing mechanism: at form-fill time attach has only a user-typed session UUID, hydration runs *after* attach, and `attach_agent_impl` stores the submitted value verbatim (no pre-attach model peek exists). More importantly, the *goal* of that clause — "attaching doesn't silently switch a running session's model" — is achieved by submitting **no** model (so no `--model` flag is sent), because Claude is **session-sticky** (`harness-behavior.md` §3.3): the resumed session keeps its own model. So attach defaults both pickers to a "keep current" sentinel that submits `None`; the user can still pick a concrete value to switch deliberately. Codex/Gemini aren't sticky, so there is nothing to preserve there anyway — `None` is the honest default uniformly. `None` thus means "no-capability" (Antigravity / Gemini-effort), "pre-feature agent", **or** "attached without an explicit override." A future `peek_session_model` command could *display* the session's real model in the attach picker (informative only — it does not change the protection), tracked as an optional fast-follow, not M5.
4. **Per-turn display, every turn.** The transcript records and displays both model and effort per turn (carry-forward for Antigravity's model; blank only when genuinely unknown). For Codex both come straight from `turn_context`; for Claude the *model* is per-turn from its records, but whether the *effort* it ran at is exposed per-turn is unverified (its `init` carries model, not obviously effort) — if absent, Claude's effort is shown as selected-intent only, with no per-turn footer value (acceptable).
5. **Change from the per-agent actions menu**, taking effect on the **next send** (never mutating an in-flight turn). Each control is absent for harnesses lacking that capability.

### Shared conventions established early (reuse, don't reinvent)

- **Two capability helpers** in `crates/core` (M1): `HarnessKind::supports_model_selection()` (Claude/Codex/Gemini) and `supports_effort_selection()` (Claude/Codex). The frontend mirrors each with one capability map (M5). Every gate — command validation, picker visibility, card action — derives from these, never an ad-hoc `if harness == …`.
- **`model` and `effort` are sibling `Option<String>` fields**, threaded identically through registration, dispatch, commands, and the per-turn `Turn`. They are two explicit fields (not a generic settings map — their value spaces and pickers differ), but they share every pattern: capability-gated, sent every turn when `Some`, displayed per turn.
- **"Send the flag every turn when `Some`, omit when `None`"** (M2) is the dispatch rule for both axes on every model/effort-capable adapter.
- **Closed controls per axis** (M5, REVISED) — model uses an adaptive picker (segmented for compact lists, dropdown for long lists), while effort uses the shared segmented control because values are short categorical levels. Both controls consume the same `{label, value}` option shape and are reused by the M6 change-editors.

## Milestone 1 — Core: `model` + `effort` fields, capability helpers, registration threading

### Goal & Outcome
Give the persisted agent a place to hold both selected values, and establish the two backend facts the rest of the plan gates on.

- An `AgentRecord` carries an optional selected `model` and `effort`; agents created before this change load with neither (backward-compatible).
- The codebase has a single authoritative answer to "does harness X support model selection" and "…effort selection."
- Both values can be set at registration in one step, through every creation path.

### Implementation Outline
- Add `pub model: Option<String>` and `pub effort: Option<String>` to `AgentRecord` (`crates/core/src/agent.rs`). **No `#[serde(default)]` and no custom `deserialize_with`** — a plain `Option<T>` already deserializes a missing field as `None`, which *is* the backward-compat behavior we want; the DoD round-trip test is the proof. **Do _not_ mirror the sibling `session_locator: Option<SessionLocator>`**: it carries a custom `#[serde(deserialize_with = "deserialize_required_locator")]` that deliberately does the *opposite* — it requires the key and fails loud when absent (agent.rs:119-137). Copying that pattern would break the "missing key → `None`" safeguard. (The flat `session_id: Option<Uuid>` field the first draft pointed at as the thing to match no longer exists — the session-locator consolidation replaced it with `session_locator`.) Serialize `null` when `None`.
- Thread both through **all** registration entry points in `crates/core/src/project.rs` — five public plus the private inner; updating only `register_agent` would silently drop the values on every *attach* path: `register_agent` (91), `register_attached_claude_agent` (122), `register_attached_codex_agent` (141), `register_attached_gemini_agent` (164), `register_attached_antigravity_agent` (183), `register_agent_inner_with_id` (205). (The two attach methods lost their `_with_id` suffixes in the same consolidation — only the private inner keeps it.) The Antigravity attach variant always receives `model: None`; Antigravity and Gemini attach always receive `effort: None` (capability invariants — may assert rather than accept arbitrary input).
- Add to `HarnessKind` (`crates/core/src/harness.rs`): `supports_model_selection(self) -> bool` (`Antigravity => false`, else `true`) and `supports_effort_selection(self) -> bool` (`Claude/Codex => true`, `Gemini/Antigravity => false`). Both exhaustive matches (no `_` arm) so a future harness forces deliberate decisions. Put the rationale (per harness-behavior §3.3/§3.4) in doc comments.

### Definition of Done
- Round-trip tests cover `Some`/`None` for both fields **and a JSON object lacking either key deserializes to `None`** (the backward-compat safeguard).
- One attach-with-values test proving both land at registration, not via a follow-up call.
- Unit tests assert both capability helpers per variant.
- `make check` green.

## Milestone 2 — Adapters: pass model + effort on every dispatch

### Goal & Outcome
When an agent has a selected model and/or effort, every turn — first or resumed — uses them; otherwise behavior is unchanged.

- Claude dispatches with `--model` and `--effort` when set; Codex with `-m` and `-c model_reasoning_effort=<level>`; Gemini with `-m` only; Antigravity with neither.
- An agent with nothing selected dispatches exactly as today.

### Implementation Outline
- **Refactor `build_args` for Codex and Antigravity to receive `&AgentRecord`** (Claude and Gemini already do); the `dispatch` methods hold it.
- Apply on **both** the first-turn and resume arg paths (these adapters build args differently per path):
  - **Claude** — `agent.model` → `--model <v>`; `agent.effort` → `--effort <v>`.
  - **Codex** — `agent.model` → `-m <v>`; `agent.effort` → `-c model_reasoning_effort=<v>` (valid on `exec` and `exec resume`).
  - **Gemini** — `agent.model` → `--model <v>`. **Ignore effort** (no flag exists).
  - **Antigravity** — ignore both; one-line comment why (no per-call control — §3.3/§3.4).
- When a field is `None`, append nothing. No value validation here — invalid values are the harness's to reject. Confirm exact flag spelling against each CLI `--help` before wiring.
- **Arg ordering (new since the first draft — the #21 dash-leading-prompt fix that merged in):** Claude and Codex now pass the prompt as a trailing positional after a `--` end-of-options separator (Codex builds a separate `--`-terminated arg list per path — first-turn vs. resume). **Insert the new `--model` / `--effort` / `-m` / `-c model_reasoning_effort=` flags _before_ that `--`** — appending after the prompt makes the flag a positional and breaks dispatch. Claude's `build_args` already carries an in-code warning to this effect (`claude_code/mod.rs:179-180`). Gemini is exempt (it passes the prompt via `--prompt=<v>`, so there's no separator and order is free); Antigravity gets no new flags. Add a `build_args` ordering assertion (flag precedes `--`) alongside the per-field presence tests.

### Definition of Done
- Unit tests on each `build_args`: each field present → its flag+value on first-turn and resume paths; absent → no flag; Gemini never emits an effort flag; Antigravity never emits either.
- **Live tests** (named `live_<harness>_…`): construct the **real adapter**, `dispatch` with an `AgentRecord` carrying a model (and, for Claude/Codex, an effort), driven through our code, **not** a bare CLI probe. **Scope the assertions to what carries a value at this milestone:** model surfaces in the emitted `SessionMeta`, so assert it there; **effort rides no emitted event until the M4 `TurnEnd` carrier exists**, so at M2 assert effort only via a `build_args` unit check + "dispatch completes without error." The end-to-end *emitted* per-turn effort assertion lands in M4. State this M2→M4 dependency so the sequencing is intentional. Tiny prompts per cost discipline.
- **Claude Sonnet reasoning live guard (carried over — depends on `--model` plumbing landing here).** Once Claude dispatches with a selected model, add a `live_claude_…` test that dispatches Sonnet (`--model sonnet`) on a non-trivial, thinking-inducing prompt and asserts a **non-empty** `ContentChunk { kind: Thinking }` arrives. This is the only automated guard that the real Sonnet CLI still returns *un-redacted* reasoning — Claude's redaction is per-model (`harness-behavior.md` §3.2), and until per-agent model selection exists there is no way to pin Sonnet in a live test (the dev default is Opus, which redacts), so today the contract is covered only by a parser unit test plus the mandated per-model re-probe on CLI bumps. If Sonnet silently re-redacts, nothing automated catches it until then. Assert `Thinking` *content presence*, not exact text; tiny prompt per cost discipline. **Close an existing loop:** the merged-in `live_claude_thinking_emits_liveness` test carries a `COVERAGE GAP` comment that names this milestone by path (`live.rs:218-225`, "we can't pin Sonnet because the adapter has no `--model` plumbing yet … until M2 lands"). Once that plumbing exists, either extend that test to pin Sonnet or add the dedicated guard here — and remove the gap comment so the TODO doesn't dangle.
- `make check` green; `make test-live` covers the new live tests.

## Milestone 3 — Backend: create / attach with model + effort, and change them later

### Goal & Outcome
The selected values reach the persisted record, and later changes persist and take effect on the next send.

- Creating or attaching an agent can specify model and/or effort; the record stores them.
- New commands change (or clear) an existing agent's model and effort and re-persist.
- A model can never attach to a harness without model support; an effort can never attach to a harness without effort support (enforced server-side, independent of the UI).

### Implementation Outline
- Extend `create_agent_impl` and `attach_agent_impl` (`crates/app/src/commands.rs`) to accept `model: Option<String>` and `effort: Option<String>`; update the `#[tauri::command]` wrappers.
- **Capability checks run first.** In `attach_agent_impl`, reject `model.is_some() && !supports_model_selection()` *and* `effort.is_some() && !supports_effort_selection()` as the first thing after project/active resolution — **before** the per-harness session lookup and the `register_attached_*` registry append. (Rationale corrected for the current architecture: the session-locator consolidation removed the attach-time **session-link sidecar** — the locator is now written straight onto the `registry.jsonl` record, see `commands.rs:1003` *"written straight onto the registry record — no sidecar"*. So the failure to prevent is committing the **registry record** with an illegal model/effort, not orphaning a sidecar. The only remaining sidecar, `.meta.json`, is written *later* during dispatch via `needs_session_meta`, not at attach.) Same checks on create. **Caveat:** the 7-step doc-comment at `commands.rs:936-941` still narrates the old sidecar-first attach flow — it's stale; don't take it as current.
- Add `set_agent_model_impl` and `set_agent_effort_impl` (+ wrappers), each taking `Option<String>` (so clearing back to the harness default is expressible — a non-optional `String` couldn't say "clear"): look up the agent, set the field, re-persist via the same registry-write path the other mutating commands use (mirror `rename_agent_impl`), return the updated record; reject for an unsupported harness. No effect on any in-flight turn — the new value applies on the next dispatch (M2 reads the field fresh each time).
- **Normalize the empty-string footgun for _both_ axes here, not only in the UI.** `set_agent_model_impl` / `set_agent_effort_impl` and the create/attach `model`/`effort` params trim and map empty/whitespace → `None` before persisting, so no caller (a future IPC path, a workflow step, a test) can bypass the M5 form and persist `Some("")` — which would dispatch `--model ""` (or `-c model_reasoning_effort=`, an empty value) every turn with a non-obvious cause. The IPC command is the real trust boundary; the M5 form normalization stays as the UX layer. This is **footgun-normalization, not value validation** — we still don't judge whether a non-empty value is "valid" (reactive); no allow-list for either. (Corrected from the first draft, which exempted effort as "enum-constrained" — that constraint is UI-only and doesn't bind a direct IPC caller, so effort needs the same boundary normalization as model. Note the `core`-layer gate from M1's `register_agent_inner_with_id` already rejects an effort/model on a harness that can't apply it; this normalization is the *separate* empty-string guard, which `core` deliberately does not do.)

### Definition of Done
- Tests on the free functions: create/attach with each field stores it; `set_agent_model`/`set_agent_effort` update and persist (reload proves durability); **clearing persists `None`**; **a whitespace/empty model _and_ a whitespace/empty effort each persist `None`, never `Some("")`**; setting a model on Antigravity or an effort on Gemini/Antigravity (all paths) returns the capability error **and leaves no persisted record** (no `registry.jsonl` append, no `needs_session_meta` entry); the attach rejection happens before the `register_attached_*` call. (The `core` gate added in M1 also backstops the capability rejection at the persistence boundary — the commands check still runs first for a friendlier error.) (Note: this DoD was reframed from the first draft's "leaves no orphan sidecar" — attach no longer writes a session-link sidecar, so the durable artifact to guard against is the registry record.)
- `make check` green.

## Milestone 4 — Per-turn model + effort on the transcript

### Goal & Outcome
Every turn records the model and effort that produced it (where the harness exposes them), for both live streaming and reopen — so the transcript shows a faithful per-turn history.

- Each completed turn carries the model (all harnesses) and effort (where available) it ran with.
- A mid-conversation switch of either is reflected turn-by-turn.

### Implementation Outline
This replaces any "pick one representative value for the agent" notion — every turn carries its own.

> **Verified against current code + live probes (2026-06-05).** Two things changed since this milestone was drafted; corrections folded into the bullets below.
> 1. **The cost milestone already built the per-turn carrier.** `AdapterEvent::TurnEnd` / `NormalizedEvent::TurnEnd` / `Turn::Agent` / the TS `LoadedTurn` + live-state `Turn` already carry a per-turn `spend` field (stamped by the `turn_end` reducer, passed through `loadedTurnToTurn`). `model`/`effort` mirror `spend` field-for-field at every layer. (The earlier "TurnEnd holds only outcome+usage / no turn-anchored carrier" framing is stale.)
> 2. **model/effort are NOT persisted to a Switchboard sidecar.** The `turnmeta` sidecar exists only for Switchboard-*derived* data absent from harness files (cost/overage). model/effort are harness-*owned* (in the session files) and are reconstructed by the per-harness parsers on reopen — persisting them would duplicate harness data and violate the split-source-of-truth principle. No `turnmeta` entry, no overlay step for model/effort.
> 3. **Probe results (2026-06-05):** Gemini records `model` **per `gemini` record** (flash→pro switch verified reconstructable on reopen); Codex's per-turn effort readback field is **`turn_context.payload.effort`**, *not* `model_reasoning_effort` (renamed since 0.134.0; the input config key is unchanged); Claude exposes **no** per-turn effort anywhere (sidebar-intent only).

- **Add `model: Option<String>` and `effort: Option<String>` to the agent-role turn shape** — the **`Turn::Agent` variant** (`crates/harness/src/transcript.rs`, a `#[serde(tag="role")]` enum), the **agent-role branch** of the hydration `LoadedTurn` (`src/lib/types.ts`), **and the agent-role branch of the live-state `Turn` (`src/lib/state/types.ts`)**. Not the `User` branch (a user turn has neither), and not only the hydration type — the live transcript renders from the live-state `Turn`, which is distinct from `LoadedTurn`. Keep these **separate from** the agent-scoped `SessionMeta` (a standalone, non-turn-anchored event, events.rs:108).
- **Populate per turn, per harness, on both paths** (live parse + session-file hydration):
  - **Claude** — model from each turn's assistant `message.model` (hydration: `claude_code/session_file.rs` already tracks `first_model`; extend to per-turn) and `system/init.model` (live). Effort: **verified absent** (2026-06-05 @ 2.1.163 — no `effort` field in the session file or `init`), so `Turn.effort = None` for Claude on both paths; Claude effort is sidebar-intent only.
  - **Codex** — `turn_context.payload.model` and `turn_context.payload.effort` per turn. **The readback field is `effort`, not `model_reasoning_effort`** (verified @ 0.137.0, 2026-06-05; the *input* config key M2 sends stays `-c model_reasoning_effort=`). Attach per turn in the **hydration builder** (`parse_codex_transcript_content` / `CodexReconstruction`, which tracks no per-turn model/effort today). **Keep the set-once `model_set` gate** in `parse_session_content` — it feeds `Enrichment.model` → `SessionMetaInfo.model` (the sidebar's current-model source until M6), *not* hydration; the earlier "remove the gate" instruction was a conflation (the gate **is** the SessionMeta source — confirmed against the code). For the **live** `TurnEnd` carrier, add *current-turn* (last `turn_context`) model+effort to `Enrichment` (the existing `Enrichment.model` is first-wins, wrong for the current turn).
  - **Gemini** — model is **per-turn on disk**: each `gemini` content record carries its own `model` (verified 2026-06-05 — flash→pro wrote two records with the two models). Hydration (`gemini/session_file.rs`) currently keeps first-model-wins for the agent-scoped `meta.model` (the `self.model.is_none()` guard at ~529); add per-turn stamping of each `Turn::Agent` from its record's `model`. Live: `init.model` per invocation. Effort always `None` (no such concept exposed).
  - **Antigravity** — model via **carry-forward** (below). Effort always `None` (folded into the model name — already visible *in* the model string, so no separate field).
- **Antigravity model carry-forward:** Antigravity writes a `USER_SETTINGS_CHANGE` "from X to Y" sentence only on the turn where the model *changed* (verified: turn 1 `None → X`, and a resume after a settings change wrote `X → Y`). Maintain a running "current model": set it to the announced "to" value on a turn carrying a change, inherit otherwise, and stamp every turn. On **hydrate**, walk turns chronologically from the start; **live**, hold the running value in per-agent state. **Reuse the existing sentence parser, build the per-turn stamp.** `antigravity/mod.rs::extract_model_from_record` already parses the `USER_INPUT`/`USER_SETTINGS_CHANGE` "Model Selection' from X to Y" sentence (trims the tier suffix) — reuse it. But the current `drain_transcript` carry is **first-wins, agent-scoped** (one `SessionMeta.model` via a `model.is_none()` guard), *not* a per-turn running stamp — so the per-turn carry-forward is net-new (model the running value on the existing first-wins logic but stamp every turn). The frontend "empty-keeps-prior" logic (reducers.ts:~450) is likewise agent-scoped `meta.model`, not a per-turn stamp — don't conflate. `None` only before any announcement (e.g. an attached conversation truncated before its first).
- **Live-path carrier — mirror the existing `spend` precedent.** The cost milestone already added a per-turn `spend` field that rides `AdapterEvent::TurnEnd` → `NormalizedEvent::TurnEnd` → the live reducer's `turn_end` case → the live-state `Turn`. `model`/`effort` follow the identical path. (`SessionMeta` stays agent-scoped — and for Codex first-turn-only, emitted *after* `TurnEnd` in enrichment — so it's the wrong carrier for the current turn's value.) So:

## Milestone 5 — Frontend: model + effort pickers in the create-agent form

### Goal & Outcome
A user creating (or attaching) an agent picks its model and effort from **closed, curated, per-harness controls** — each preselected to a sensible default and **always submitted** — so every agent is created with a **known, displayed** model (and effort, where supported). No free-text, no "unset/default" option for capable harnesses.

- The form offers a **model** adaptive picker (Claude/Codex/Gemini) and an **effort** segmented control (Claude/Codex), each a fixed curated list with a preselected default.
- For a harness lacking a capability, the control is replaced by a short **note** (Antigravity: both, with "set the model inside Antigravity"; Gemini: effort only).
- **Create** preselects the harness default (Claude `opus`/`high`, Codex `gpt-5.5`/`medium`, Gemini `auto`) and always submits it. **Attach** defaults both pickers to a "keep current" sentinel that submits **no** model/effort, so attaching never silently switches a running session's model (Claude is session-sticky → no flag means the session keeps its own model); the user can still pick a concrete value to switch deliberately. See decision #3 for why this supersedes the original "attach preselects the hydrated model."
- For **create**, both selections reach the backend as concrete values for capable harnesses; for **attach**, only an explicit user choice does.

### Implementation Outline
- Add two frontend capability maps to `src/lib/harnessDisplay.ts` mirroring `supports_model_selection()` and `supports_effort_selection()` (exhaustive `Record<HarnessKind, …>`).
- Use a reusable adaptive picker for model choices (segmented for ≤4 options, native `Select` for ≥5) and the reusable segmented picker for effort choices. Both take `{label, value}[]`, value `string`. No free-text branch, no empty-string handling needed (the closed lists can't produce `""`); the M3 backend normalization stays as defense-in-depth.
- **Per-harness curated lists (live-verified 2026-06-05 — see `harness-behavior.md` §3.3/§3.4; patch as models ship/sunset):**
  - Model — **Claude** `opus`/`sonnet`/`haiku` (durable aliases; M6 footer shows the resolved id); **Codex** `gpt-5.5`, `gpt-5.4-mini`; **Gemini** `auto`, `gemini-2.5-pro`, `gemini-2.5-flash`, `gemini-2.5-flash-lite`, `gemini-3-pro-preview`, `gemini-3-flash-preview`, `gemini-3.1-pro-preview`, `gemini-3.1-flash-lite-preview`. Display friendly labels, submit the alias/id. **Antigravity** — no list; render the note. Comment: lists are hardcoded because no harness exposes a queryable list and Codex values are plan-gated (a curated entry can still fail per-account → failed turn, by design).
  - Effort — **Claude** `low/medium/high/xhigh/max` (offer all 5; CLI silently downgrades unsupported levels per model — add a one-line caveat "higher levels may run lower on some models", *not* a model-aware matrix); **Codex** `none/minimal/low/medium/high/xhigh` (`none` is a real level). **Gemini/Antigravity** — note, no control.
- Wire both into `CreateAgentForm.svelte` after the harness selector; preselect the default, swap the **harness-switch** to reset to that harness's default; replace unsupported controls with the note. **The mode-switch (create↔attach) also re-seeds the pickers** — create→harness default, attach→a "No override" sentinel (empty value, mapped to `undefined` on submit; labeled "No override" rather than "keep current" because omitting the flag only *preserves* the model on the sticky harness, Claude — Gemini reverts to its default and Codex to its config default, so "no override" is what we actually do for all three).
- **Make attach-id validation harness-aware (folded in during M5 — a pre-existing frontend/backend mismatch the picker work surfaced).** The form UUID-validated every attach id and labeled the field "Existing session UUID", but the backend treats a **Codex** thread-id as an arbitrary string used verbatim (`attach_agent_impl` `parse_uuid`s Claude/Gemini/Antigravity but not Codex; `find_codex_session_file_for_attach` is a filename-suffix match, so a non-matching string simply finds nothing). So validate UUID-shape for Claude/Gemini/Antigravity and non-empty-trimmed for Codex, and label the field "Existing session ID" for Codex. Non-empty is the right floor; the backend's file lookup is the authoritative check. **Mind the indirection — it's a three-layer thread, not just `api.ts`:** `CreateAgentForm.svelte` emits an `AgentFormSubmit` discriminated union (`CreateAgentForm.types.ts`) to `App.svelte`'s `createOrAttachAndRegister` (via `handleCreateFirstAgent`/`handleAddAgent`), which calls `createAgent`/`attachAgent`. Thread `model?: string` / `effort?: string` through **(1)** `AgentFormSubmit`, **(2)** the `App.svelte` handlers, **(3)** the `createAgent`/`attachAgent` signatures in `api.ts`. ## Milestone 6 — Frontend: change model/effort, surface intent vs. history, and document limitations

### Goal & Outcome
For a capable agent the user can change its model/effort after creation; the sidebar shows the selected values (immediate), the transcript shows per-turn values (history), and the user-facing harness limitations are documented.

- The actions menu offers "Change model" (Claude/Codex/Gemini) and "Change effort" (Claude/Codex), each absent where unsupported.
- Changing/clearing calls the backend and the **sidebar reflects the selection immediately** — before any turn runs.
- The **transcript footer shows each turn's model and effort** next to its timestamp.
- The README's user-facing limitations section names the Gemini and Antigravity gaps.

**Sidebar display — intent-first with observed fallback (DECIDED 2026-06-06, was "intent-only / retire SessionMeta.model").** The sidebar shows the user's *selected* model when set, falling back to the harness-*observed* model only when no selection exists — never blank when the harness has reported something:
- `AgentRecord.model` set → show it as the selection (e.g. `model: opus`).
- `AgentRecord.model` is `None` but `runtime.meta.model` exists → show the observed model, distinctly labeled (e.g. `observed: claude-sonnet-4-6`).
- Neither → hide the line.

This **keeps `SessionMeta.model`** (no retirement): it is the observed-model fallback for the two no-explicit-intent cases — Antigravity (which can never carry `AgentRecord.model`) and an attached agent left at M5's "No override" default. The per-turn transcript footer is unchanged and may show a *resolved* id (`claude-opus-4-8`) even when the selection is a durable alias (`opus`); that divergence is **expected, not a warning condition** — no mismatch/alias-comparison warnings in M6 (they'd need harness-specific resolution rules; defer). Update the "until M6" comments around `SessionMeta.model` to describe this continuing purpose rather than implying retirement.

### Implementation Outline
- Add "Change model" and "Change effort" actions for a capable agent, each gated on its capability map (so an Antigravity agent shows neither; a Gemini agent shows only "Change model"). **There is no `AgentActionsMenu.svelte`** — the first draft assumed a component that was never built. Today the per-agent actions (Stop / Resume in terminal / Open session file / Delete) live **inline in `Sidebar.svelte`** as a hover-revealed icon strip, and rename is a double-click inline edit, not a menu item. **Decided:** host the new items in a `DropdownMenu` (built from `ui/DropdownMenu` + `ui/DropdownMenuItem`) added as a "⋯ more" trigger in that strip — a select/menu fits better than another icon (model/effort have no natural glyph), and keeps a growing action set out of the inline bar. ## Out of scope (explicitly)

- *Querying* harness model lists, validating values, or detecting plan-gated Codex models — no harness exposes a list, so the pickers are **hardcoded curated lists** (live-verified, patched as models ship/sunset), and a curated value that's plan-gated for the user's account still fails reactively as a normal failed turn. - Any mechanism to set Antigravity's model or Gemini's effort (both are harness-owned config we never touch; per-`HOME`/config-dir isolation was tested and declined — §3.3). These are read-only/unavailable here and documented as user-facing limitations.
- Changing model/effort mid-turn / interrupting an in-flight turn.
- A global or project-level default for either — selection is strictly per-agent.
- "Show both values on the agent card" reconciliation — obviated by the sidebar-intent / transcript-history split.
