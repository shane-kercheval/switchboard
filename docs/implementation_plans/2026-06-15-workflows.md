# M6 + M7 — Workflow engine, dependency resolver, pause & iteration

**Status:** plan (not started)
**Supersedes outline in:** `docs/implementation_plans/2026-05-12-v1.md` §M6 / §M7 (this is their implementation-grade expansion).
**Authoritative DSL spec:** `docs/workflow-spec.md` — the workflow file format. This plan implements it; where this plan and the spec disagree, the spec wins and this plan is wrong and should be fixed.

## What this delivers

By the end of M7, a user can: from the **manual** compose bar, forward one or more agents' (possibly still-streaming) outputs — individually or by pane — into a message to one or more recipients, with the send held until the sources finish; do the same into the **arguments of a prompt** (Milestone 2.5); author a YAML workflow under `<dir>/.switchboard/workflows/`; invoke it with a per-input form; watch it run in a progress surface; cancel it mid-flight; abandon a run that was interrupted by a crash (resuming/retrying a run is **deferred beyond v1** — see Milestone 5); have a workflow pause for their input mid-run; and run a workflow that iterates over a static list. Two built-in workflows ship (`review-and-aggregate`, `review-analyze-discuss`).

The milestones are split M6 (1–5, plus 2.5 and 4.5: engine + manual forwarding + built-in library, no pause/iteration) and M7 (6–7: pause, iteration, interactive failure-pause), matching the roadmap's `M6 → M7` sequence. Milestone 2.5 (manual forwarding into prompt arguments) was added in the 2026-06-16 functional-alignment review; Milestone 4.5 (built-in content library) was added in the 2026-06-20 planning review — see each milestone for its rationale.

## Architecture decisions (these are the planner's calls — do not re-litigate by reading code; the rationale must survive into crate/module docs)

These were decided in the design discussion that produced this plan. State each one's rationale in the relevant module/crate doc comment so it does not evaporate.

1. **New pure crate `crates/workflow` for the workflow *language*.** It holds the file model, parser, parse-time + invocation-time validation, the MiniJinja template environment + the four helper functions, variable-scope resolution, and the run-status + checkpoint *record types*. It depends only on `switchboard-core` (for `AgentId` etc.), `serde`, `serde_norway`, `minijinja`, `serde_json`, `thiserror` — **no `dispatcher`, `prompts`, `harness`, `app`, or `tauri`.** Rationale: this mirrors the workspace's existing discipline (core/harness/dispatcher are pure, Tauri-free, and heavily unit-tested); the language layer is a large self-contained body of logic with its own fixture-driven test surface, and `crates/app/src/commands.rs` is already ~12k lines. Keeping the language pure is what makes the three worked examples testable as fixtures with no app.

2. **The interpreter/runtime lives in `crates/app`, not in the workflow crate.** The interpreter is a *conductor*: it orchestrates the `Dispatcher`, `PromptService`, transcript loading, checkpoint file IO, and the event emitter — all app-owned. Rationale: it is wiring by nature; forcing it into the pure crate would require inventing an injected-trait zoo (TurnRunner, PromptResolver, OutputReader, CheckpointWriter…) with no payoff, because we can integration-test the interpreter through the **real `Dispatcher` + `MockHarnessAdapter`** exactly as `crates/dispatcher/tests/dispatcher_with_mock.rs` already does. That tests the real concurrency, not a mock of it. Do not build the trait zoo.

3. **Turn-completion observation is added to the dispatcher (M1), not reconstructed from the event stream.** The actor already owns the single authoritative terminal; the completion signal fires from there. Rejected alternative: an app-layer second `EventEmitter` that re-parses serialized `TurnStart`/`TurnEnd` JSON to resolve futures — fragile (string/JSON matching) and duplicates state the actor already holds. See M1 for the contract and the audit obligation (it must fire at *every* terminal-synthesis point). **The completion payload carries the turn's captured text output, not just its outcome** — see decision #7.

4. **One dependency resolver, two surfaces.** The "hold a send → await agent B's referenced turn → resolve B's output text → dispatch to A (or invalidate)" mechanism is built once (M2) and reused by the interpreter (M4). The manual compose bar (M2) is a first-class user of it, per the system-design §7 binding principle — not a workflow-only capability. The single per-surface difference is failure handling: **manual forward** invalidates recoverably (restores the user's text to the composer); **workflow `forward_from`** is a hard step failure. Same resolver, surface-specific terminal handling. This difference is the planner's decision; state it in the resolver's doc.

5. **Sibling-failure policy is phased** (resolved in this discussion; already propagated to `docs/workflow-spec.md` §"Sibling-failure policy" and `system-design.md` §7): M6 ships the **non-destructive floor** (never cancel surviving siblings; step `failed`; outputs retained; retry re-runs the whole step). M7 layers the **interactive failure-pause** once pause machinery exists. M6 must **not** implement the old "SIGTERM the survivors" behavior.

6. **Run checkpoints reuse the journal pattern, not a new store.** `runs/<run-id>.jsonl` under the project dir, written via an injected trait shaped like `ConversationJournal` (app owns the path, core/workflow owns the record format). Add a `RUNS_DIR` path constant alongside the existing `WORKFLOWS_DIR`. **(v1 update — resume deferred):** with resume/retry out of v1 (see "What this delivers" and Milestone 5), the run file no longer needs to carry replay state. It shrinks to lightweight progress/terminal bookkeeping — enough to surface an active or interrupted run and let the user **abandon** it — and notably carries **no persisted agent output text** (see decision #8). Whether a run file is still warranted at all, vs. tracking active runs in memory and writing only a terminal/abandoned marker, is an M5 shape decision; the `CheckpointRecord` types defined in M3 were shaped for replay and will likely be trimmed accordingly. **(Resolved in Milestone 5: keep the on-disk `runs/<run-id>.jsonl` the interpreter writes — it is what survives a crash for interrupted surfacing — *and* add an in-memory active-run registry for live cancel/listing; retention runs in the app task that owns the run, keyed off the returned status. See Milestone 5 "Run lifecycle.")**

7. **Forwarded output is captured from the live stream, never re-joined from disk by id.** A finished turn's completion payload carries the agent's text output, accumulated by the actor as it already drains content events. Rationale (load-bearing — state it in the dispatcher + resolver docs): the dispatcher's `turn_id` is **not joinable** to the harness session file's own turn ids (`crates/app/src/commands.rs` correlates the two by *order* precisely because a direct join is impossible), and the one stable per-turn key that does exist (`hydration_key`) is **absent for Antigravity**. So "hold a `turn_id`, load the transcript, find the matching turn" — the approach an earlier draft of this plan took — cannot work. Capturing text at completion sidesteps identity entirely for the just-awaited turn: no disk read, no key dependency, no race. Disk loading (`load_agent_transcript` + the existing order-correlation) is kept **only** for the "latest completed snapshot when nothing is in flight" case (M2's no-in-flight path). The per-run output-scope map (M4) therefore stores **resolved text**, not a turn-id needing later correlation; `hydration_key` may ride along as diagnostic/reference metadata but is never the load-bearing path.

8. **~~Workflow run checkpoints may persist resolved output text — a scoped exception to the §3 "no agent content" invariant.~~ SUPERSEDED for v1 — the exception is NOT taken.** This decision existed solely to make crash-recovery *resume* work (re-feed an earlier step's output after restart). With resume deferred beyond v1 (see "What this delivers" / Milestone 5), that justification is gone: **v1 persists no agent output text in run files, and the §3 invariant stands unmodified** (do not reword §3). The per-run output-scope map of decision #7 still exists — but **in memory only**, for the duration of a live run, read by the helpers and `forward_from`; it is never written to disk. If a future milestone adds resume, this exception comes back with it (and the §3 rewording would be made then, not now).

# M6 — Workflow engine + manual dependency forwarding + workflow cancel

## Milestone 1 — Per-send turn-completion signal in the dispatcher

*Small, isolated milestone — compressed per the structure guidance. It is the substrate every later await depends on, so it goes first.*

### Goal & Outcome

A caller of `Dispatcher::send_message` can obtain a handle that resolves — when **that specific send's** turn reaches a terminal state — with both the turn's outcome **and its captured text output**.

- A caller (workflow runtime, manual-forward resolver) can `await` a send and learn whether the turn `Completed`, `Failed`, or was `Cancelled`, **and** receive the agent's text output for a completed turn (so forwarding never needs a disk lookup — decision #7).
- A send that never starts a turn (journal write fails, adapter dispatch fails) resolves the handle as a failure, not a hang.
- A turn whose stream ends with **no** terminal and **no** cancel (adapter-contract violation / upstream stream drift) still resolves the handle (as a failure) rather than stranding the awaiter forever.
- Existing callers (the compose-bar `send_message_impl`) are unaffected — the completion handle is opt-in.

### Implementation Outline

Add an **opt-in completion channel** to the dispatcher's send path. The `WorkItem` gains an optional `oneshot::Sender<CompletionResult>` (the receiver is handed back to the caller), where `CompletionResult` is a small struct `{ outcome: TurnOutcome, text: String }` (carrying the captured text-kind output for a completed turn — empty for failed/cancelled). `hydration_key`/`turn_id` may be included as reference fields but are **not** the load-bearing path (decision #7). The contract:

- The signal fires **exactly once**, carrying the same `TurnOutcome` the actor used for the terminal `TurnEnd` it emitted (or synthesized), plus the accumulated text.
- **Text accumulation:** for opt-in awaited sends only, the actor accumulates `ContentChunk` text where `kind == Text` (exclude `Thinking`; exclude tool output) as it already drains those events in `drain_turn`. Non-awaited sends accumulate nothing (no cost on the compose-bar path).
- It must fire at **every** terminal-synthesis point in `run_turn` / `drain_turn`. Audit these against the current code: (a) the journal `record_send` fail-closed path (→ a synthesized `Failed`), (b) the adapter `dispatch()` error path (→ `Failed { AdapterFailure }`), (c) the normal `TurnEnd` from the stream, (d) the synthesized `Cancelled` terminal after a fired cancel token, (e) the force-failed locator-persist path, **(f) the stream-ends-without-terminal-and-without-cancel path** (currently `crates/dispatcher/src/lib.rs:1284` only `warn!`s and advances the backlog — under an awaited handle this strands the caller forever; synthesize a `Failed { AdapterFailure }` terminal here, consistent with (b), and resolve the handle with it). Missing any one turns an awaited send into a hang — this audit is the milestone's main risk.
- A dropped `Sender` (caller stopped awaiting) must not panic the actor — `oneshot` send returning `Err` is ignored, consistent with how the actor already treats dropped reply channels.

Decide the surface shape against the code, but the **contract** is fixed: opt-in (the `Enqueue` compose path passes `None` and is byte-for-byte unchanged in behavior), one-shot, fires-on-every-terminal. Workflow sends will use `OnBusy::FailFast`; a `Busy` outcome means no turn started and therefore no completion handle is meaningful — model this so the caller cannot confuse "Busy, nothing queued" with "queued, awaiting completion" (FailFast already returns `SendOutcome::Busy` synchronously, so the handle is only created once acceptance is certain).

Why a per-send `oneshot` rather than a broadcast/watch keyed by `turn_id`: the caller holds the `MessageId`/handle at send time and there is exactly one awaiter per send; a `oneshot` is the minimal primitive and needs no keying, no cleanup, no missed-event window. Record this rationale in the dispatcher module doc next to the existing "wire contract" paragraph.

### Definition of Done

- Unit/integration tests in `crates/dispatcher/tests/` (extend `dispatcher_with_mock.rs`) covering: completion resolves `Completed` **with the captured text** on a normal turn; thinking-kind chunks are excluded from the captured text; resolves `Failed` when the adapter dispatch errors; resolves `Cancelled` when the turn is cancelled mid-flight; resolves a failure when `record_send` fails (use a journal stub that errors); **a stream that ends with no terminal and no cancel resolves the handle as `Failed` rather than hanging** (case (f) — a mock adapter that drops its stream without a `TurnEnd`); the `Enqueue` path with `None` is unchanged (existing tests still pass) and accumulates no text; dropping the receiver does not disturb the turn or the actor.
- Dispatcher module doc updated with the completion contract + the per-send-`oneshot` rationale + the capture-text-not-disk-join rationale (decision #7).
- **Resolved (shipped):** the awaitable API is **FailFast-only** — `send_message_awaiting_completion` takes no `on_busy` parameter and dispatches `OnBusy::FailFast` internally. This makes the "queued send removed before it starts → handle never fires" path **unreachable by construction** (a FailFast-accepted send is the actor's next action; it never lingers in the backlog), so the handle resolves with a `CompletionResult` for every accepted send and the only `Err` is the caller dropping its own receiver. (Supersedes the earlier "cover the queued-removal case or note it" framing — there is no queued-await path to cover. An awaitable-enqueue variant, if a future milestone needs one, must teach the queue-drop paths to fire the completion first.)

## Milestone 2 — Cross-agent dependency resolver + manual compose-bar forwarding

*This ships standalone user value before any workflow YAML exists, and builds the resolver M4 reuses. It is the milestone the system-design §7 binding principle is about.*

### Goal & Outcome

From the compose bar, a user can compose a message that **forwards one or more agents' outputs** into a send to **one or more recipients** — and if a source is mid-turn, the send is held until that source's turn finishes, then dispatched with the **finished** output. This is the manual surface of the §7 binding principle and builds the resolver M4 reuses. (The forward capability did **not** exist before M6 — it is built fresh here, both the wait *and* the no-wait paths; see the §7 doc correction in the DoD.)

*(The behaviors below were resolved in the 2026-06-16 functional-alignment review and are the binding outcomes for this milestone.)*

- **Sources — fan-in.** The `@`-mention menu (which already lists agents and panes) gains "forward output from {agent | pane}" entries. A user can add **multiple** sources to one send. A **pane** source expands to each member agent's latest completed output (membership captured at send time). Each source's output is composed into its own canonical block.
- **Recipients — fan-out.** Forwarding composes with the compose bar's existing multi-recipient send: the composed message can target multiple agents / a pane at once (one send, grouped by `send_id`, renders once; each recipient gets its own turn with the same body). The two axes are independent — N sources (fan-in) → M recipients (fan-out).
- **Hold + wait.** If any referenced source has an in-flight turn, the send is **held** — a distinct "waiting for {agent}…" state, not "queued — agent busy" — until **all** referenced sources reach terminal, then dispatched. A source with **no** in-flight turn contributes its latest completed output immediately (read from disk; the no-wait case).
- **Busy recipient → queue.** When the held send resolves and a *recipient* is busy, it queues to that recipient (`OnBusy::Enqueue`, the compose-bar default), exactly like any other manual send. `OnBusy::FailFast` is the **workflow-only** contention rule, not the manual one (the per-surface side of decision #4).
- **Empty sources.** Forward only sources that have completed output; **skip** empties. If **every** referenced source is empty, the send fails and the composed text is restored (nothing to forward). An *idle* source already known empty is flagged on its chip up front. When some sources are skipped, the **wire content is unaffected** (only the non-empty canonical blocks — so manual and workflow composition stay identical). The skip is surfaced **UI-only**: a muted caption on the sender's transcript entry (e.g. `↪ forwarded from reviewer-1, reviewer-2 · reviewer-3 had no output`), rendered with `messageMeta` so it stays visible even when the message is collapsed (the held send may resolve after the user has walked away). The caption appears only when ≥1 source was skipped, names the included and skipped agents, and is **never** part of the message the receiving agent reads.
- **Canonical composition.** The user's typed text (if any) appears first, then each source's output in its own **`=== START forwarded from {agent} === / === END ===`** block (per `docs/workflow-spec.md` §`send` `forward_from`; labeled per agent, never per pane), in the order sources were added. This is **wire content** the receiving agent reads — plain text by design, identical between manual and workflow forwarding.
- **Invalidation + cancel.** If a referenced source's turn **fails or is cancelled while the send is held** (the in-flight case), the held send is **invalidated** (never dispatched with empty/stale input) and the composed text + forward chips are **restored** to the composer — an unexpected mid-wait failure hands control back rather than auto-sending. A held send is itself **cancellable** with the same restore; cancelling it does **not** cancel the source agents' turns (they keep running).
- **Source whose latest turn *already* failed (idle case).** If a source is idle but its **most recent turn failed or was cancelled**, the forward carries a **failure note** (e.g. `⚠️ {agent}'s most recent turn failed… Error: …`) as that source's block — *not* a stale older successful turn — so the recipient learns it failed and can react. (The harness session file has no record of a turn that crashed before writing a result, so the failure is read from the journal's outcome marker, the durable record of non-completed terminals.) Distinct from the in-flight case above: here the user deliberately forwarded a known-failed source, so the forward proceeds with the failure rather than invalidating. **Known limitation (accepted):** "most recent turn" is judged from Switchboard's journal; a turn produced *outside* Switchboard (direct CLI / imported history) carries no journal record, so in the rare case where an agent fails in Switchboard and then succeeds outside it, the forward still reports the failure — the app-wide model already treats non-journaled turns as older pre-history, and closing this would be a systemic change, not a forward-resolver patch.
- **Transcript rendering.** A dispatched forwarded message renders **verbatim** — text unchanged, including the `=== … ===` lines, no collapse-within-collapse (messages already collapse whole). The forwarded blocks get **style-only** treatment (accent border / band + per-agent label) so it is obvious what was forwarded and from whom. No text is rewritten or hidden.

### Implementation Outline

Three pieces, in order:

**(a) Two output-text sources, keyed by *who dispatched the turn*.** Decision #7 governs this. The text source depends on whether *this surface* dispatched the forwarded turn — do **not** try to unify them behind a disk-`turn_id` lookup (that join is impossible — see decision #7):

- **A turn this surface dispatched with an M1 handle (the workflow case, and any M4/2.5 send that awaited):** the text comes from the **M1 completion payload** — read `CompletionResult.text` directly. No disk read, no id correlation. This is the load-bearing path for workflows.
- **A turn this surface did *not* dispatch (the manual forward case):** the manual forward references an agent whose turn was started by the user's *earlier* compose send (`completion = None`), so there is **no M1 handle to it**. The manual path reads the latest completed output from disk via `load_agent_transcript` (`AgentRecord → session file → LoadedTranscript`), taking the most-recent completed `Turn::Agent` and concatenating its `TurnItem::Text { kind: Text, .. }` items (exclude `Thinking`; exclude tool output) — positional, not id-keyed. If the referenced turn is still in flight, the manual path first **waits** for it (see (b)), then reads disk.

Give the text-concatenation logic (`Turn::Agent` items → text-only string) a clear shared home so M4's helpers apply the identical filter. The workflow path never touches disk; the manual path always reads disk (after waiting, if needed).

**(b) The dependency resolver (backend, shared) — a thin mechanism; policy is the caller's.** It exposes primitives — **source-wait**, **text resolution** (via (a)), **canonical-block composition**, and **dispatch with a caller-chosen busy policy** — and does *not* own prompt rendering (that is layered by Milestone 2.5). It takes target recipient(s), a composed **text body**, and a set of forward-source **agents**; holds outside any queue; waits for each source to be terminal; resolves each source's text via (a); composes the canonical body; dispatches to each recipient. The four policies that vary by surface are **inputs, not baked in:**

- **Source-wait differs by surface** (the manual case is *not* an M1 handle — see (a)). The interpreter dispatched its workflow sources, so it awaits their M1 handles. The **manual** forward holds no handle to an already-running source turn, so **the dispatcher gains a per-agent "await this agent's current in-flight turn's terminal" capability** — distinct from M1's per-send handle; resolves immediately if the agent is already idle. Both feed the same composition/dispatch tail. *Open implementation question for M2:* after a manual source's terminal, the manual path reads disk — verify the just-completed turn is reliably present before reading (the harness writes its session file independently of the dispatcher terminal; wait it out if needed).
- **Empty-source policy is supplied by the caller**, never resolver-baked: manual plain forward = skip empties, fail only if **all** empty; prompt-argument forward (2.5) = per-argument required/optional; workflow `forward_from` = **fail on any** missing/empty referenced source (per `docs/workflow-spec.md` §`send` — a workflow must surface the clear failure, never silently drop a referenced agent). Baking the manual "skip" rule into the shared resolver would corrupt workflow fan-in, so this stays a parameter.
- **On-busy policy is the caller's:** manual dispatches `OnBusy::Enqueue` (a busy recipient queues, like any manual send); workflow (M4) dispatches `OnBusy::FailFast` (contention = step failure).
- **Invalidation is the caller's:** on a source failing/cancelling it returns without dispatching; manual restores text, workflow fails the step (decision #4).

**Panes are not a backend concept** — the frontend expands a pane source to its member agent IDs (membership captured at send time) before calling the resolver. Expose these primitives so the manual command (this milestone), the prompt-composer command (Milestone 2.5, which layers arg-fill + `PromptService::render` on top), and the interpreter (M4) all reuse them.

**(c) Tauri command + frontend.** A command for "forward from {sources} to {recipients}" that drives the resolver. Frontend: extend the `@`-mention menu (`ComposeBar.svelte`) with "forward output from {agent | pane}" entries that add a forward **source chip** — reuse the existing attachment-chip zone; the textarea stays plain text (inline chips *inside* the textarea are not feasible and not needed, since the composition is always text-then-blocks). Support **multiple** source chips and **multiple recipients** (fan-out via the existing multi-recipient send; a pane source/recipient expands to its members). Render the held send's "waiting for {agent}…" state (new, distinct from "queued — agent busy"); make held sends **cancellable**; on invalidation/cancel, restore the composed text + source chips (reuse the removed-queued-message restore path). Render a dispatched forwarded message **verbatim** with style-only treatment on the forwarded blocks (accent band + per-agent label; the `=== … ===` lines stay visible). The default is **wait-for-finished** when a source is in-flight (system-design §7: the user wants the *finished* response ~99% of the time) — no "snapshot vs wait" prompt.

Sequencing note: (a) and (b) are backend and can land before (c). The per-agent current-turn wait, the held-send state, the invalidation/cancel-restore, and the partial-empty caption are the parts most likely to harbor bugs — they get their own tests per the DoD.

### Definition of Done

- Backend tests (extend the dispatcher mock harness or app integration tests): wait-for-finished forwards a source's completed text; no-in-flight-turn forwards the latest snapshot immediately; a source failing/cancelling invalidates the send (no dispatch); the canonical sentinel composition matches the spec's shape exactly.
- Backend test for the **per-agent current-turn wait**: a manual forward referencing an agent with an in-flight turn (one *this surface did not dispatch*) waits for that turn's terminal, then forwards its disk output; resolves immediately when the agent is already idle.
- Backend tests for the aligned multi-source / fan-out behaviors: **multiple** sources compose in declared order, each in its own per-agent block; a **busy recipient queues** (`OnBusy::Enqueue`) rather than failing; **partial-empty** forwards the non-empty sources, leaves the **wire body unaffected**, and surfaces the UI-only caption; **all-empty** fails and restores; a **pane source** resolves to each member's latest completed output (expansion happens frontend-side, so the backend test passes the expanded agent list). The **empty-source policy is caller-supplied** — assert the manual skip/fail-all-empty policy here; the workflow fail-on-any policy is asserted in M4.
- **Frontend tests are mandatory for this milestone** (the system-design and the M6 outline both call this out explicitly): a component test that mounts the compose bar, mocks `invoke`/`listen`, and exercises "wait for the source(s), then forward" including the dependency-failure path (source fails → composed text + chips restored, send not delivered). - Transcript rendering test: a forwarded message renders **verbatim** (the `=== … ===` lines present, text unchanged) with the forwarded blocks visually marked (style-only, no collapse).
- The output-text reader has unit tests over a fixture transcript (text-only turn, turn with interleaved tool calls, turn with thinking blocks excluded).

## Milestone 2.5 — Manual forwarding into prompt arguments

*Added in the 2026-06-16 functional-alignment review. The plan originally covered only the plain compose-bar forward (M2); manual forwarding into a **prompt's arguments** is a stated primary use case (e.g. a code-review fan-in feeding an aggregation prompt) and the §7 binding principle requires the manual surface to express what a workflow `send` with `template_vars` can. Placed right after M2 because it depends **only** on the M2 resolver + the existing `PromptService` — not on the workflow crate (M3) or interpreter (M4) — and because building it here de-risks M4: the "resolve forwarded text → fill named args → render prompt → dispatch" path is exactly what the workflow `send` step needs, so M4 reuses a proven path instead of inventing one.*

### Goal & Outcome

In the prompt composer (the structured-args form, `PromptComposer.svelte`), each declared argument can be filled by typing **and/or** by forwarding output from one or more agents/panes — the manual analogue of a workflow `send` whose `template_vars` reference `aggregated_responses(...)`.

*(Functional outcomes resolved in the 2026-06-17 alignment review; binding for this milestone. The original "or" was corrected to "and/or" so a single argument behaves exactly like the M2 compose bar — typed text plus forwarded blocks — rather than a typed-XOR-forwarded special case, keeping the two surfaces consistent.)*

- **Per-argument, independent sources.** Each argument gets its own optional "forward from {agent | pane}" affordance — a **↪ control beside the argument's input box** that opens the agent/pane picker; selected sources render as forward source chips **below that input**. Chips appear only when a source is picked (exactly like the compose bar), *not* an always-on toggle list — that all-agents-visible pattern is the recipient "To" field only. Different arguments draw from different sources. The appended-text field is forwardable the same way (it is just another field). (Originally the ↪ sat on the label row; moved beside the input in the 2026-06-18 review so it reads as an action on the field.)
- **A pane is one chip, not expanded.** Picking a pane adds a **single named pane chip** (membership snapshotted at pick time), expanded to its member agent ids only at dispatch — the wire content is always per-agent blocks, so the pane grouping is UI-only. This is deliberately asymmetric with the recipient "To" chips (which expand to per-agent chips): the "To" chips are always visible with easy `⌘+click` pane-switching, while forward chips are an ephemeral point-in-time selection, so a compact single chip is clearer. (Resolved 2026-06-18; `ForwardSource` is an `agent | pane` union shared by both surfaces.)
- **Typed + forwarded per argument.** An argument's resolved value is its typed text (the existing auto-expanding `Textarea`) **first**, then its forwarded blocks — byte-identical to the M2 compose bar's `body + blocks` composition, scoped to one argument (reuses `compose_forwarded_message` verbatim). With no sources, an argument is pure typed text (today's behavior, unchanged).
- **Hold + wait + render + dispatch.** If any referenced source is mid-turn, the prompt send is **held** (M2's mechanism — same "waiting for …" row + cancel) until all referenced sources settle; then each forwarded argument's value is composed, the args map is filled, the prompt is rendered via `PromptService::render`, and the **rendered body** is returned for the frontend to dispatch through the normal send path (the M2 resolve-on-backend / dispatch-on-frontend split, so live grouping/queue/cancel/correlation all work). Fan-out + busy-recipient-queues (`OnBusy::Enqueue`) exactly like M2.
- **Same rules as M2, reused not reinvented.** Empty-source handling, the canonical per-agent block composition, invalidation/cancel-restore, and the after-the-fact partial-empty annotation are identical to M2 and share its resolver + composition function + `ForwardOutcome` + `heldForwards` store. Restore covers the prompt selection + typed args + per-argument source chips (more state than M2's plain forward, same mechanism; the navigate-away restore limitation carries over, documented).
- **Empty handling is per-argument.** Within one argument forwarded from several sources, include the non-empty ones; an *optional* prompt argument may resolve empty and the send proceeds; a **required** argument that resolves fully empty (all its sources empty) is a validation failure → restore (extends the prompt composer's existing required-arg gating).
- **Attachments × forward (resolved 2026-06-18).** **Both** forward paths carry staged attachments — a forward is still a send, so the user's attached files ride it exactly like any message (the forwarded blocks are the message *text*; the attachments are sent alongside as files, orthogonally). Files clear optimistically on submit and restore rebuilds their chips if the forward fails. (This corrects an earlier M2 simplification where the plain message-forward dropped attachments — that was a bug, not a design: a user who attaches files expects them to send regardless of whether content is also forwarded.)
- **A hold is a model-layer wait, not a UI lock.** While a forward (compose-bar or prompt) is held waiting on its sources, the composer stays fully interactive — the user can type and send other messages, which queue normally; restore guards against clobbering anything the user started in the meantime.
- **Forwardable appended text.** The "Appended text" field has its own ↪ + chips, just like an argument. Its forwarded blocks compose into the appended tail (typed lead + blocks); the backend resolves the appended sources in the **same hold** as the argument sources (one invalidation domain) and returns the **already-combined** body (rendered prompt + appended), so the frontend dispatches verbatim. (Added 2026-06-18.)
- **Prompt mode hides the message-level forward affordances.** In prompt mode, forwarding is per-field (per argument or the appended text), so the compose bar's own "Forwarding from" row and ↪ Forward button are hidden — they feed a different (whole-message) forward set the prompt send path doesn't read. The plain-mode forward set is preserved in state and reappears when the prompt is removed. (Bug fixed 2026-06-18.)

#### Also in M2.5 — compose-bar forward affordance + shortcut (additive)

Two small compose-bar additions, made here because the forward-source picker becomes a shared component and the surfaces should stay consistent:

- **↪ "Forward" button next to the "Prompt" button** — a discoverable click entry to the same agent/pane forward-source picker, adding to the compose bar's forward-source set. The `@`-menu "forward from" entries stay as the keyboard path; both add to the same set.
- **`Cmd+Ctrl+1..9` pane-forward shortcut** — adds pane N as a single pane forward chip, mirroring the existing `Cmd+Opt+1..9` ("target pane N") and `Cmd+1..9` ("target agent N"). **Panes only** (a per-agent forward chord would collide with the agent-target chord) and **compose-bar only** (the prompt composer's N per-argument source sets have no unambiguous target for a global digit chord). The chord is surfaced on the pane rows of both the `@`-menu and the ↪ picker (compose-bar instances only). Verify `Cmd+Ctrl+digit` is unbound before wiring.

### Implementation Outline

- Reuse the **M2 resolver** (per-source `wait_for_current_turn` → live-captured text, or idle-disk snapshot) and the **M2 canonical-composition function** for each forwarded argument's value. No parallel mechanism.
- Backend command/path (resolve-only, like M2): take the prompt provider + name, the typed args, the per-argument forward sources, and a `forward_id`; hold for the sources, compose each forwarded argument's value, fill the args map, `PromptService::render(provider, name, args)`, and return `ForwardOutcome::Resolved { body, skipped }` (no dispatch — the frontend dispatches the rendered body through the normal send path). This is the **manual precursor of M4's workflow `send` with `prompt` + `template_vars`** — M4 reuses this resolve → fill-args → render tail. State that linkage in the doc so the two do not drift.
- Frontend: a shared **forward-source picker** component used by both the compose bar (the new ↪ button) and `PromptComposer.svelte` (per-argument ↪). `PromptComposer` gains the per-argument affordance + chips while preserving the existing structured-args + preview lifecycle; a forwarded argument keeps its typed input and shows its chips below, and the **preview** reflects the resolved value, rendering an `«forwarding from {agent}…»` placeholder for sources still in flight (the real render happens at send, after the hold).

### Definition of Done

- Backend tests: resolve sources → fill args → render → return `Resolved`; **mixed** typed + forwarded args on one prompt; held-until-settle; an *optional* arg resolving empty proceeds; a **required** arg whose sources are all empty fails (restore); the forwarded-argument composition is asserted **identical** to M2's shape (shared function).
- Frontend tests (mandatory, per the component-test convention): per-argument forward affordance renders + adds chips on pick; typed-plus-forwarded composes in order; wait-then-render-then-dispatch (resolved → dispatched through the normal send path); failure/cancel restores prompt + args + chips; idle-empty source flagged on its chip; required-all-empty blocks the send. Compose-bar additions: the ↪ button opens the picker and adds a source; `Cmd+Ctrl+N` adds pane N's members as forward sources.
- Doc states the reuse of the M2 resolver + composition + `ForwardOutcome`/`heldForwards`, and the explicit "M4 reuses this resolve → fill-args → render path" linkage.

## Milestone 3 — `crates/workflow`: the workflow language (parser, validation, templating, helpers)

### Goal & Outcome

A pure crate that turns a workflow `.yaml` file into a validated, executable in-memory model, and renders its templates — with no runtime/dispatch concerns.

- A workflow file is parsed into a typed model; malformed files and spec violations are rejected at **parse time** with clear errors (the spec's "Parse-time" validation list).
- **Invocation-time** validation is supported: given parsed inputs + the project's agents + the prompt providers, missing required inputs, non-existent agents, unresolvable prompt ids, empty/duplicate `[agent]` lists, and unresolvable template variables are caught before any dispatch.
- Templates render through the MiniJinja subset the spec defines, with the four helper functions (`responses_from`, `aggregated_responses`, `last_output`, `agent_names`) and the documented scope-precedence order.
- The three worked examples in `docs/workflow-spec.md` parse and validate.
- Run-status (`complete`/`cancelled`/`failed`/`interrupted`) and checkpoint **record types** are defined here (the interpreter in M4 writes them; the format lives with the language).

### Implementation Outline

Create the crate per architecture decision #1 (`cargo new`-equivalent within the workspace; deps via `cargo add`: `serde`, `serde_norway`, `minijinja`, `serde_json`, `thiserror`, `switchboard-core`). The crate is the authoritative implementation of `docs/workflow-spec.md` — read it as the spec, not this section.

- **Model + parser.** Top-level `name`/`description`/`inputs`/`steps`; the input type grammar (`agent`, `[agent]`, `prompt_id`, `text`, `text?`, `[text]`, shorthand + long form); the step enum (`send`, `wait_for`, `wait_for_all`, `pause_for_user`, `for_each`). Use `serde_norway` for the YAML layer. Each step is a mapping with exactly one step-type key (the spec is explicit) — model the "exactly one key" and "unknown step type" rules.
- **Parse-time validation** — implement the spec's full "Parse-time" bullet list, including: reserved top-level/step keys (the forward-compat table) are errors; no nested `for_each`; `item:` not colliding with an input name or `user_input`; hardcoded `[agent]` literals checked for emptiness/duplicates (after hyphen→underscore normalization); all template strings parse as valid templates; `name` slug matches and equals the filename. Use `thiserror` typed errors at the crate boundary (workspace convention).
- **Templating.** Build a MiniJinja `Environment` configured to the spec's **subset**. Two distinct enforcement mechanisms — do not conflate them:
  - *Undefined variables* → render error (strict-undefined). `crates/prompts/local.rs` configures undefined behavior but uses **lenient**-undefined; workflows require **strict** per the spec — set strict.
  - *Unsupported tags* (`{% set %}`, `{% raw %}`, macros, inheritance `{% extends %}`/`{% block %}`, includes, the `do` extension, custom filters) → these are **valid MiniJinja syntax and will NOT self-reject** (strict-undefined does nothing for them). They require an **explicit validation pass** at parse time that rejects the specific tag/feature set. Prefer MiniJinja AST inspection if exposed; otherwise a conservative scan of the parsed template that does not false-positive on the tag text appearing inside string literals or comments. This is a *bounded* check (a fixed list of tags), not a general linter. (Earlier wording implied these "produce render errors" automatically — they do not; this is the correction.)
  - Register the four helper functions. The helpers read from an **injected output-scope map** keyed `agent → resolved completed-turn text` (decision #7 — resolved text, never a turn-id requiring later disk correlation). The crate defines the helper *logic* and the map's shape; the interpreter (M4) populates it. Keep the crate pure: the helper does no IO; it receives already-resolved text. Implement the scope-precedence chain (step-local `template_vars` → iteration var → `user_input` → workflow inputs) and the canonical `aggregated_responses` sentinel shape (identical to M2's forward composition).
- **Run-record + status types.** Define the `runs/<run-id>.jsonl` record enum and the run-status enum here as serde types. With resume/retry **deferred beyond v1** (decision #8 superseded; see "Failure handling" in the spec), these are **progress/terminal bookkeeping, not replay state**: the record carries the run's workflow name + step count (`Started`), a marker per completed step (`StepCompleted { step_index }`), and a terminal marker (`Terminal { status, failed_step?, reason? }`). The terminal's optional `failed_step` / `reason` (the interpreter's operational error message — never agent output) let a failed run surfaced after a restart explain itself. There is **no persisted output-scope map and no `user_input`** — that resolved text is agent content, so the §3 invariant stands unmodified; the per-run output scope (decision #7) lives in memory only, read by the helpers. (Trimmed from M3's original replay-oriented shape during M4 when resume was deferred. M7 can add an iteration-progress marker additively — `RunRecord` is `#[non_exhaustive]` — so there is no schema-migration cost to having omitted it.)

### Definition of Done

- Heavy unit tests (this is the pure, high-value test surface): the three worked examples parse + validate; each parse-time validation rule has a positive and negative case; **each unsupported MiniJinja tag is rejected by the explicit validation pass, and a template with that tag's text inside a string literal/comment is NOT a false positive**; undefined variable is a render error; scope precedence resolves to the innermost binding; `aggregated_responses`/`responses_from` produce the canonical shape and the name→underscore mapping; helper errors when an agent has no completed output.
- Run-record types round-trip through serde — `Started` / `StepCompleted` / `Terminal` (including the optional `failed_step` / `reason` on a failed terminal, omitted on the wire otherwise). These are progress/terminal bookkeeping only: no output-scope map, no `user_input`, no iteration/replay state is persisted (decision #8 superseded — the in-memory output scope never reaches disk; §3 stands). A later variant (e.g. an iteration-progress marker) is an additive, non-breaking change (`#[non_exhaustive]`).
- Crate-level doc states decision #1's rationale (why pure, why separate from the interpreter) and points at `docs/workflow-spec.md` as the spec.
- Known limitation recorded: the crate validates template *parsing* at parse time but variable *resolution* only at invocation/render time (matches the spec) — state this so M4 knows render errors surface at step execution.

## Milestone 4 — Workflow interpreter / runtime

### Entry conditions carried from the M3 review

Two items surfaced reviewing the M3 crate that M4 must respect (neither was M3 work):

1. **`responses_from` cannot feed a prompt argument; `aggregated_responses` can.** `PromptService::render` takes `BTreeMap<String, String>` (string args only), so the *structured map* `responses_from` returns can't be a `template_vars`→prompt-arg value — it would stringify uselessly. The string helper `aggregated_responses` is the prompt-arg path; `responses_from` is for inline workflow `text:` rendering (where M4 renders the whole template, `.items()` and all). The built-in `review-and-aggregate` already uses `aggregated_responses`, so it's fine. Wire `template_vars`→prompt args accordingly.

2. **Resume / retry is deferred beyond v1 (RESOLVED — engineer's call, 2026-06-18).** A run executes start→finish; a `failed` / `interrupted` / `cancelled` run is terminal and is **abandoned**, never resumed (the snapshot-persistence question that motivated this is therefore moot). Consequences M4 should bank on: (a) **no checkpoint replay** — the interpreter does not re-feed earlier-step outputs after a restart, so the per-run output-scope map is **in-memory only** for the live run (decision #7 stands; decision #8 / the §3 agent-content exception is **not taken** — no agent text on disk); (b) the workflow snapshot is held in memory for the live run (so a mid-run file edit doesn't change the running program) and is **not persisted**; (c) run-file content shrinks to progress/terminal bookkeeping for surfacing + abandon (see decision #6), and the M3 `CheckpointRecord` types — shaped for replay — will likely be trimmed once M5 settles the minimal run-state shape. Rationale: a crash leaves dead agent turns + a half-run transcript, so correct resume needs runtime replay *and* a non-trivial UI; crashes are rare and the cost spans interpreter + progress UI, so v1 surfaces-and-abandons instead.

### Goal & Outcome

The app can execute a parsed workflow against a project's agents: dispatch steps, synchronize, resolve forwarded/aggregated output, record run progress, handle failures non-destructively, and cancel. (Resume/retry is deferred — see "Entry conditions" above; the run record is progress/terminal bookkeeping, not replay state.)

- `send` (single + parallel list), `wait_for`, `wait_for_all` execute with correct parallelism (list dispatches issued in order, agents run concurrently) and synchronization (barrier waits on all).
- `forward_from` and the helper functions resolve against the **per-run output-scope map** (only turns this run dispatched and observed reaching terminal — the spec's "Output scope" rule), reusing the M2 resolver and the M2 output-text reader.
- A workflow run terminates in exactly one of `complete`/`cancelled`/`failed`, per the spec's status table; trailing in-flight turns hold the run open until they settle.
- **Sibling-failure non-destructive floor** (decision #5): a participating agent failing marks the step `failed` but never cancels surviving siblings; their output is retained.
- Run progress (current step / status) is recorded so M5 can surface an active run and, if a process dies mid-run, an `interrupted` run the user can **abandon**. No replay state is written (no resume in v1).
- Workflow-level cancel stops orchestration and cancels in-flight turns via `CancelSource::Workflow`; cancelling a participating agent's turn marks the whole run `cancelled`.
- Runs are **project-scoped and concurrent across projects** (separate runtime instance per run; no cross-project coordination).

### Implementation Outline

The interpreter lives in `crates/app` (decision #2). It consumes the M3 model, dispatches each step via the **M1 completion signal** (`send_message_awaiting_completion`), renders prompts with `PromptService::render`, and writes run-progress records.

**Reuse is the standalone primitives, not a shared "tail" (clarified during the M4 review — the plan's earlier "reuse the 2.5 tail / M2 resolver" wording overstated it):**
- The genuinely shared atoms are already standalone and M4 calls them **directly**: `PromptService::render` (prompts), `compose_forwarded_message` in `crates/harness/src/forward.rs` (the canonical block shape), and the M3 `render` + helpers. There is **no reusable fill-args→render→dispatch "tail"** — the M2.5 logic is inline in `forward_prompt_resolve`, hardcoded to the manual `ForwardArg`/`ForwardOutcome` shapes; do **not** extract or contort it. The two surfaces' front halves differ (2.5 resolves from manual source chips via `wait_for_current_turn`/disk; the workflow fills `template_vars` by rendering M3 templates whose helpers read the **in-memory per-run output-scope map**), and their dispatch differs (next item).
- **Workflows dispatch backend-side.** The manual surface's resolve-on-backend / dispatch-on-frontend split exists for live-UI `send_id` correlation with a user in the loop; a workflow is background orchestration with no per-step frontend round-trip, so the interpreter dispatches each step itself via `send_message_awaiting_completion` (minting one `send_id` per `send` step, shared across a fan-out's recipients) and awaits the M1 completion to drive the next step. `forward_from`/helpers therefore resolve from the per-run output-scope map (populated from M1 completions as `wait_for`/`wait_for_all` settle), **not** from `wait_for_current_turn`/disk.

- **Execution model.** A step-based interpreter (the spec and system-design §4 are explicit that a general DAG scheduler is *not* required for v1 — do not build one). Walk the snapshotted steps; for a list `send`, issue all dispatches (FailFast) in declared order and collect their completion handles; `wait_for`/`wait_for_all` await the relevant handles. On a handle resolving `Completed`, store its `CompletionResult.text` (decision #7) into the per-run output-scope map keyed by agent — `agent → resolved text this run`, **not** a turn-id needing disk correlation. This map is what the M3 helpers read through. It is per-run and sees only this run's turns (the spec's out-of-band-invisibility rule); implement it so a manual send or another run targeting the same agent is not observed. The map is **in-memory only** — decision #8 (persisting it) is superseded; no agent text is written to disk in v1.
- **Workflow file snapshot at invocation** (spec §"Workflow file snapshot"): the run executes against an immutable copy of the parsed workflow + bound inputs captured at invoke time. Edits to the file mid-run do not affect the run or its retries. Prompt resolution still happens at each step's dispatch (so prompt edits take effect next invocation). Capture the snapshot; do not re-read the file per step.
- **Contention = step failure** (system-design §7): workflow sends use `OnBusy::FailFast`; a `Busy` is a step failure with the spec's error ("agent X is busy…"), not a queue. This is the deliberate difference from the manual compose bar's queue.
- **Completion fires at terminal, not when the agent is re-dispatchable — fixed dispatcher-side (resolved in the M4 review).** The M1 signal resolves the instant the terminal is observed, but the actor then drains post-terminal adapter events before it parks idle; a FailFast re-send to the **same** agent during that window currently returns `Busy` (dispatcher doc + the FailFast arm in `drain_turn`), so a back-to-back same-agent `send`→`wait_for`→`send` would spuriously fail. The rejected interpreter-side fixes both have flaws: a `wait_for_idle`/`AgentIdle` wait **masks genuine contention** (it would wait out an unrelated queued manual send, violating the workflow FailFast rule), and a bounded `Busy` retry can't distinguish "my own drain window" from real contention (the drain can be a session-file read, not microseconds). The correct fix is in the dispatcher's FailFast arm: accept the send when it is purely the agent's **own post-terminal drain** (`terminal_seen && backlog empty`), and return `Busy` only when a turn is genuinely mid-flight or work is already queued. This preserves "workflow contention = step failure" while bridging the self-drain window. (The two DoD worked examples don't hit back-to-back same-agent, but the fix lands so the shape isn't a latent footgun.)
- **Sibling-failure floor.** When a parallel/fan-in participant fails, mark the step `failed`, do **not** cancel siblings, let them settle, retain their output. M6 stops here — no pause (that is M7). Implement so the M7 interactive pause can wrap this without re-architecting: the floor's "collect all sibling outcomes, then fail" shape is what the pause will hook.
- **Status state machine + trailing settle.** The run does not go `complete` until every turn it dispatched is terminal, including turns still in flight after the last step (spec §"Failure handling"). A trailing failure → `failed`. Implement the trailing-settle hold.
- **Run progress recording (no replay).** Record run progress (current step / status) so M5 can surface active and interrupted runs and offer **abandon**. App owns the path (`<dir>/.switchboard/projects/<project-id>/runs/<run-id>.jsonl`); add the `RUNS_DIR` path constant in `crates/core/src/paths.rs`. The record carries **no agent output text** (decision #8 superseded) and no replay state (no resume in v1). The exact minimal shape — a lightweight `runs/<run-id>.jsonl` vs. in-memory active-run tracking with only a terminal/abandoned marker on disk — is settled with M5's surfacing UI (decision #6); trim the M3 `CheckpointRecord` types to match. If a file is written, reuse the journal's fsync/durability posture (`crates/core/src/io.rs`).
- **Cancel.** Workflow-level cancel fires the **same** per-turn `CancellationToken` with `source = Workflow` (decision: reuse M4.2, not a new path) on whichever agents have in-flight turns for the run, and stops the interpreter. A user cancelling a participating agent's turn (observed via the completion signal resolving `Cancelled`) marks the run `cancelled` uniformly (spec §"Failure handling").

### Definition of Done

- Integration tests through the **real `Dispatcher` + `MockHarnessAdapter`** (model: `dispatcher_with_mock.rs`): sequential handoff (worked example 1) runs end-to-end; fan-in review (example 2) dispatches reviewers in parallel, waits for all, aggregates, dispatches to primary; `forward_from` composes the canonical body; helper output-scope correctly ignores an out-of-band manual send to a participating agent; contention on a `send` step fails the step; **sibling failure marks the step failed without cancelling the surviving sibling** (assert the survivor's turn still completes and its output is retained); workflow-level cancel marks the run `cancelled` and fires `CancelSource::Workflow`; cancelling one participant's turn marks the whole run `cancelled`; the trailing-settle hold keeps a run open until a fire-and-forget final turn completes, and a trailing failure marks it `failed`.
- Run progress is recorded and round-trips; a simulated mid-run drop leaves a record whose last entry identifies the step reached, which M5 surfaces as `interrupted` (for abandon — no resume). Assert the record carries **no agent output text** (decision #8 superseded; §3 invariant intact).
- `RUNS_DIR` constant added; run path matches the spec.
- Interpreter module doc states: step-based-not-DAG rationale, the output-scope-map rule, the snapshot-at-invocation rule, and the reuse of M1/M2 + `CancelSource::Workflow`.
- **Capability gating (not parse rejection).** M3 parses the **full** v1 DSL — `pause_for_user` and `for_each` are syntactically valid and the spec's worked example 3 (which uses both) validates. They are simply **not executable until M7**. So the gate is a *runtime* capability check, not a parse error: M5's invocation-validation rejects a workflow containing either step type with a clear "step type not supported in this version" message (list it as syntactically valid; block invoke), and the M4 interpreter errors clearly as defense-in-depth if ever handed one. (This corrects an earlier internal contradiction in this plan that said these were "rejected at parse time" — they are not; that would break worked example 3.)

## Milestone 4.5 — Built-in content library (read-only, in-app, toggleable)

*Added in the 2026-06-20 planning review. The plan originally shipped built-in prompts/workflows by **seeding copies into the user's folders** on first run (the existing `prompts_setup` pattern). Review surfaced three problems with that model: it can't deliver an improved or newly-**required** built-in to an install that has already seeded (the single marker suppresses all further seeding); a renamed prompt argument meeting a stale seeded copy fails the run **mid-execution** (the renderer rejects the unknown arg after the form has already accepted it); and it needs marker/version bookkeeping to paper over both. This milestone replaces seeding with an **app-owned, read-only built-in library** surfaced alongside the user's own content — the conventional "built-in vs. user" split (editor built-in snippets/themes are the mental model). It lands before Milestone 5 because the flagship built-in workflow depends on built-in prompts, and because it establishes the built-in-content pattern Milestone 5 reuses for workflows.*

### Goal & Outcome

Built-in prompts ship **inside the app** as a read-only library — always current with the installed version, surfaced alongside the user's own prompts. The app writes **nothing** to the user's folders unless the user explicitly copies a built-in to customize it.

- Built-in prompts appear in the prompt picker tagged **built-in / read-only**, alongside the user's local and MCP prompts. They are **not files in any folder the user browses** — they live in the app, so there is nothing to accidentally edit-in-place and nothing for an update to clobber.
- A user-global **"Show built-in prompts & workflows"** setting (default **on**) hides the built-ins from the pickers when off, so a user who doesn't want examples sees only their own content. (The same setting governs built-in workflows in Milestone 5.)
- A **"Copy to my prompts"** action duplicates a built-in into the user's prompts directory as a normal, owned file; thereafter the app never touches that copy and built-in updates don't affect it. This is the **only** path that writes a built-in into the user's folder, and it is explicit.
- A built-in's name **never collides** with a user's same-named prompt — built-ins carry their own provider identity, so a user can author their own `code-review` and both coexist (theirs is editable and theirs; the built-in is read-only).
- Bundled examples are **current with the app**: improving a built-in or adding a new one reaches every install on update — no seeding step, no marker, no manual distribution.
- The strengthened `code-review` and the new `ai-review-feedback` ship as built-in prompts (Milestone 5's flagship workflow consumes both).

### Implementation Outline

- **Built-in prompt source (prompts crate).** Add a third read-only `PromptProvider` impl that serves the bundled prompts from binary-baked content (`include_str!`), under a **reserved provider identity** — add a `BUILTIN_PROVIDER` constant (e.g. `"builtin"`) and **reject it in `is_valid_provider_name`** (which today reserves only `local`), so a user-configured MCP provider can't claim the name and break the no-collision guarantee. Wire it into `PromptService`'s **existing local-then-MCP dispatch** in `sync`/`render` — there is *no* `Vec<dyn PromptProvider>` registration list today, so this edits the bespoke wiring (and the `render` provider-name match), not a plug-in seam. `list` includes the built-ins; `render` resolves them by id. **`is_builtin` is derived from `provider == BUILTIN_PROVIDER`** (document that as the wire contract) — no new field on `Prompt`. The pure crate owns the baked content + provider; the app wires it in.
- **Remove the seed-on-first-run machinery.** Delete `prompts_setup`'s seed-into-the-user-folder logic and its marker — built-ins now come from the library, not a copy. (Existing users keep whatever was previously seeded into their folder; those are ordinary user files the app leaves alone — see Migration.)
- **Visibility toggle.** Add a **new** `show_builtins` field (default true) to `Preferences`/`config.yaml` (which today carries only `editor_command`/`terminal_app`/`diff_style`). **Filtering lives in the app command layer** (`list_prompts_impl` and the workflow list command), **not** in `PromptService` — the pure prompts crate must stay ignorant of app-owned preferences. So `PromptService` always lists and renders every provider, and the app command drops built-in rows when the toggle is off. Consequence (the chosen rule, state it in the command doc): the toggle controls **visibility in the pickers/list only**, *not* resolution-by-id — `render("builtin", …)` still succeeds when the toggle is off, so a workflow already wired to a built-in never breaks.
- **"Copy to my prompts" command.** Reads a built-in's baked content and writes it into the user's default prompts directory as `<name>.md`, refusing to clobber an existing file (surface a clear "already exists", never overwrite). Follows the `*_impl` + thin shim convention.
- **Built-in content.** Author the strengthened `code-review` (realistic architectural-review guidance; keep a `{% if context %}…{% else %}…{% endif %}` conditional — local prompts render through the same MiniJinja engine; argument `context` — the clean name is now safe, because the built-in workflow and built-in prompt are co-versioned in the app and there is no stale seeded copy in the built-in resolution path) and the new `ai-review-feedback` (decision-ready review-analysis; argument `review`). Both live under the app's bundled-resources dir and compile into the built-in provider.
- **Frontend.** The prompt picker tags built-ins read-only and offers "Copy to my prompts"; Settings gains the "Show built-in prompts & workflows" toggle. A built-in opened for viewing shows read-only with the copy affordance as the path to editing — there is no in-place edit because there is no user-folder file to edit.
- **Pattern for Milestone 5.** This establishes the built-in-content mechanism (read-only library + toggle + copy-to-mine + own-identity-no-collision). Milestone 5 **reuses** it for built-in **workflows** (a built-in workflow source merged into the workflow list, governed by the same toggle, with "Copy to my workflows"). Build the prompt side so the workflow side is a direct application, not a re-implementation.

### Definition of Done

- Built-in prompts list alongside local/MCP, tagged via `provider == BUILTIN_PROVIDER`; resolve by id through `PromptService::render`; a user's same-named local prompt and the built-in coexist without collision (test both list and resolve).
- The built-in provider name is **reserved**: `is_valid_provider_name(BUILTIN_PROVIDER)` is false, so an MCP provider configured with that name is rejected (test it) — the no-collision guarantee holds by construction, not convention.
- `show_builtins` hides built-ins from `list` when off and restores them when on; per the chosen rule, an explicit resolve-by-id of a built-in still succeeds when off. Round-trips through `Preferences`/`config.yaml`.
- "Copy to my prompts" writes the built-in into the user's prompts dir as an owned file, refuses to overwrite an existing file, and the copy is thereafter independent (a later built-in change doesn't touch it).
- The seed-on-first-run machinery is removed; a fresh install writes **no** prompt files into the user's folder (assert nothing is written), yet `code-review` + `ai-review-feedback` are available as built-ins.
- Frontend tests (mock invoke/listen): built-ins render with the read-only tag + "Copy to my prompts"; the toggle hides/shows them; copy invokes the command.
- Crate/command docs state the built-in-library rationale (read-only-in-app over seed-into-folder: auto-delivery, never clobbers user files, no marker/version bookkeeping, no stale-copy arg mismatch) and the visibility-vs-resolution rule.

**Known limitations (recorded):**
- Migration: a user's previously-seeded `code-review.md` remains in their folder as their own file (harmless; coexists with the built-in under its separate identity); the app neither deletes nor updates it. Such a user keeps their old copy until they remove it themselves.
- Per-item dismiss of individual built-ins is deferred — v1 has a single global toggle (sufficient while the catalog is small; per-item visibility becomes worthwhile as it grows).

## Milestone 5 — Workflow Tauri commands, invocation + progress UI, built-in workflows

### Goal & Outcome

A user can discover, invoke, watch, and cancel workflows from the desktop UI, abandon a run interrupted by a crash, and two built-in workflows ship.

**Resume/retry is deferred beyond v1 (2026-06-18 — see decision #8 and "Failure handling" in the spec).** This milestone therefore does **abandon-only** recovery: it surfaces an interrupted/failed run and lets the user clear it, but does not resume or retry one. Bullets below reflect that; the deferred resume design lives in the spec's "Retry from inside a `for_each`" note.

**A workflow is invoked like a prompt, and parameterizes its recipients rather than naming them.** The discussion that produced this milestone settled that the invocation surface mirrors the prompt composer: a `+ Workflow` entry beside `+ Prompt`, a workflow menu mirroring the prompt menu, and a structured-args form built like `PromptComposer`. The load-bearing distinction from a prompt (state it in the form's doc): a prompt composes one message to whoever the compose bar's **To** field targets, whereas a workflow's recipients are its **declared `agent`/`[agent]` inputs** — named slots (`reviewers`, `analyst`) bound to real agents/panes at invocation, so the same workflow file runs against any project. In workflow mode the compose bar's own **To** field and message-level forward affordances are therefore hidden (the workflow owns routing), exactly as prompt mode already hides the forward row.

- A `+ Workflow` entry in the compose bar opens a menu listing the **built-in workflows** (from the Milestone 4.5 library, read-only) and the **directory's own workflows** (parsed from `<dir>/.switchboard/workflows/`), files that failed to parse shown with their error. Built-ins are always present, so the entry is always useful; the Milestone 4.5 **"Show built-in prompts & workflows"** toggle hides them for a user who only wants their own. A Settings **"Open workflows folder"** action (mirroring the prompts-folder action) lets a user add their own files. *(This supersedes the earlier "hide the entry when the folder is empty" idea — with an always-present built-in library there is no empty state to key off, and the global toggle is the off-switch.)*
- Invoking presents a form with one field per declared input — agent/pane pickers for `agent`/`[agent]`, the prompt menu for `prompt_id`, text fields for `text`/`text?`/`[text]` — validates inputs (invocation-time rules from M3), and **launches a background run, returning immediately** (the interpreter dispatches every step backend-side; there is no per-step frontend round-trip).
- A workflow whose file contains a `pause_for_user` or `for_each` step still lists as syntactically valid, but the form surfaces "step type not supported in this version" and disables invoke (the capability gate — these parse and invocation-validate, but do not run until M7).
- A **glanceable progress indicator** (chosen shape — see Implementation Outline) shows, whenever ≥1 run is active, interrupted, or retained-failed, each run's name, current step / total, and status, with **Cancel** per active run; multiple concurrent runs (within and across projects) are listed; it disappears when none remain.
- On project open, a run left unfinished by a crash is surfaced there as "interrupted at step N" with an **Abandon** action (clears the run record). No resume/retry in v1.
- An OS notification fires on run **completion** and **failure** (the two real terminals), suppressed when the app window is focused; not on user-initiated cancel, and **not** on interruption (a crashed run has no live process to notify — it is surfaced in the indicator on restart instead). Notification permission is requested at first invoke.
- Two built-in workflows ship via the Milestone 4.5 library (read-only, surfaced alongside the user's own, "Copy to my workflows" to customize): **`review-and-aggregate`** (generic fan-in: review in parallel → aggregate → primary) and **`review-analyze-discuss`** (the flagship: fan-in review → analyze → round-trip discussion → analyst). The built-in **prompts** they consume (`code-review`, `ai-review-feedback`) come from the Milestone 4.5 library; M5 does not seed or strengthen prompts itself.
- `docs/agent-instructions/workflows.md` exists so a user can point an agent at it to author a workflow.

### Implementation Outline

**Commands** (`crates/app`, `*_impl` free function + thin `#[tauri::command]` shim, tagged-error pattern at the IPC boundary):

- **list** — returns the **built-in workflows** (from the Milestone 4.5 library, tagged read-only) merged with the project directory's own workflows (scan `<dir>/.switchboard/workflows/*.{yaml,yml}`, `parse_workflow(stem, content)` each). Per entry: parsed metadata (name, description, declared inputs with type/optional/description, in declaration order) **or** the parse error, an `is_builtin` flag, and an **`invocable`** flag that is false when the workflow contains a `pause_for_user`/`for_each` step (so the UI can disable invoke up front). The directory scan is **directory-scoped** (workflows live under the working directory; prompts are user-global); built-ins are governed by the same Milestone 4.5 `show_builtins` toggle and, being baked-in, never produce a parse error.
- **validate-invocation** — re-parse the named file; run M3's `validate_invocation` against the project's agent roster (names) and a **`PromptService`-backed prompt-resolution predicate**; AND apply the capability gate (reject a workflow containing `pause_for_user`/`for_each` with the fixed message "step type not supported in this version"). Inputs cross the boundary in the M3 `InputValue` shape (scalar text vs. list).
- **invoke** — validate (as above), `bind_invocation` → the bound input scope, build the agents map (project roster `name → AgentId`), mint `run_id = Uuid::now_v7()`, build `run_path` via the existing `Project::run_path(run_id)`, construct a **new production `DispatchFactoryProvider`** — this does not exist yet; only the trait does. It must resolve **each** agent at `factory_for` call time via `lookup_agent` + `adapter_for` and build a per-agent `ProjectDispatchContextFactory`, so it owns the AppState-derived clones (`agents_by_id`, `emitter`, `needs_session_meta`, `registry_write`, and all four harness adapters) — it is *not* a copy of `send_message_impl`'s single-agent construction. Then construct the `WorkflowRun`, register a run handle in the active-run registry, **spawn `execute()` on a background task**, and return `run_id`. The workflow is **snapshotted at invoke** (spec §"Workflow file snapshot"): a directory workflow is re-read once here (never per step); a built-in's snapshot is its library content (no file on disk).
- **cancel** — fire the registered run's `CancellationToken`; the interpreter observes it and finishes `cancelled` (firing `CancelSource::Workflow` on its in-flight agents, already implemented in M4).
- **list-runs** (`list_workflow_runs`) — return, for the project, **three classes** in one query: the live registry's runs; **retained failed** runs (run files whose last record is `Terminal { status: Failed }`); and **interrupted** runs (run files whose last record is *not* `Terminal` and which are not in the live registry — the process died mid-run). This is the fix for the earlier "active/interrupted" shape, which silently excluded retained failed runs (a failed run *is* a terminal record, so a "no-terminal" rule skipped it) and left them unreachable/unabandonable after restart.
- **abandon** — delete a `failed` **or** `interrupted` run file. (Complete/cancelled files are already pruned on terminal — see Run lifecycle.)

**Run lifecycle, active-run registry, and retention (resolves decision #6's open run-state shape).** Keep the on-disk `runs/<run-id>.jsonl` the M4 interpreter already writes — it is what survives a crash for interrupted surfacing — **and** add an **in-memory active-run registry** to `AppState`: `run_id → { cancel token, project_id, workflow name, run_path, latest progress snapshot }`. The snapshot (updated by the progress sink) lets `list_workflow_runs` report a live run's current step without re-reading disk; `run_path` lets cancel/abandon act without re-deriving it. **Lock order:** the registry is a new tail-leaf mutex (peer of `forwards`); document it in the `AppState` lock-order contract and never hold it across an await. **Teardown (ordering is load-bearing):** `remove_directory` / `delete_project` must, for the affected project(s), **cancel the workflow runs *before* the existing agent-drain** — collect the run ids under the write lock, fire their cancel tokens, release the lock, and await the runs reaching terminal **first**, *then* proceed to drain agents / delete the directory. If agents drain first, a run observes an out-of-band terminal and resolves `failed`/ambiguous instead of `cancelled` (and may write to a directory being deleted); cancel-runs-first is what makes the run resolve `cancelled` deterministically and stops it dispatching against unloaded state or emitting stale `workflow:<project-id>` events. The background task that owns `execute()`: on terminal, removes the registry entry; applies retention keyed off the returned `RunStatus` (the D6 policy) — **prune** the run file on `Complete`/`Cancelled`, **retain** it on `Failed`; an `Interrupted` file is never produced here (it exists only because the process died) and is pruned only by **Abandon**, never auto-pruned before it is surfaced on restart; then emits the terminal progress event and fires the **completion / failure** notification (not cancel, not interruption). Retention lives in the app task, **not** the interpreter — the interpreter stays a pure writer (it already only `record()`s and returns a status). Run records carry **no agent output text** (decision #8 superseded; §3 stands), so they are ordinary bookkeeping. (Forecloses a "workflow run history" view — out of v1 scope; the unified transcript already shows the turns.)

**Progress emission seam — the single reach-back into the M4 interpreter.** The interpreter today writes run records to disk but emits nothing live, so the progress indicator has no event source. Add a thin **progress sink** injected into `WorkflowRun` (an `EventEmitter`-shaped handle the app supplies; tests supply a recorder). The interpreter emits a progress event (a) at run start (workflow name, total steps), (b) as it **begins** each top-level step (current step index — so the indicator advances when a step starts, not only when the prior one completes), and (c) at terminal (status, plus `failed_step`/`reason` on failure). The app maps these onto a **project-scoped channel `workflow:<project-id>`** with a minimal payload (run id, name, step index/total, run status, optional failure reason). Rationale (state in the interpreter + emitter docs): a project-scoped channel means the frontend subscribes **once per open project** rather than chasing per-run channels for runs that come and go; the indicator needs **step** granularity, and the agents' turns themselves keep streaming on the existing `agent:<id>` channel and render in the unified transcript — **do not duplicate them here**. Emission points coincide with the existing record/step boundaries (plus one step-start), so live events and disk records cannot drift. The payload carries **no agent output text** (§3).

**Capability gate (runtime check, not parse rejection).** `pause_for_user`/`for_each` parse and invocation-validate as syntactically valid — worked example 3 must still validate — but are **not runnable** until M7. Enforce in two layers: validate-invocation and invoke refuse them with the fixed message; list flags them non-`invocable` so the UI disables invoke. The M4 interpreter already errors defense-in-depth if ever handed one.

**Notifications.** Add `tauri-plugin-notification` (via `cargo add` + the JS plugin per the plugin docs; add the capability entry if the JS API is used). Fire an OS-native notification on run **completion** and **failure** only, **suppressed when the app window is focused** (so it fires when the user has walked away, per system-design §7, not when they're watching). **Not** on user-initiated cancel; **not** on interruption (a crashed run has no live process to fire from, and notifying at restart-scan would re-spam stale runs on every open — interrupted runs are surfaced in the indicator instead). Request notification **permission at first invoke** (contextual), not at cold startup. (Pause notifications come in M7.)

**Built-in workflows via the Milestone 4.5 library (not folder-seeding).** The two built-ins are delivered exactly like built-in prompts: baked into the app, surfaced read-only in the workflow list, governed by the `show_builtins` toggle, with **"Copy to my workflows"** writing a YAML copy into `<dir>/.switchboard/workflows/` for a user who wants to customize one. M5 does **not** seed files into the directory and adds **no** seed marker — that approach is replaced wholesale by Milestone 4.5. The built-in **prompts** the workflows consume (`code-review`, `ai-review-feedback`) are owned by Milestone 4.5; M5 only references them by id (e.g. the built-in `code-review`'s argument is `context`, defined in 4.5).

Built-in shapes (both use only M5-runnable steps — no `pause_for_user`/`for_each`; recipients are parameterized inputs):

- **`review-and-aggregate`** (the generic fan-in) — inputs `primary_agent: agent`, `reviewer_agents: [agent]`, `review_prompt: prompt_id`, `user_context: text?`. **No `aggregation_prompt` input** — the aggregation wrapper is **inline `text`**, so the built-in is self-contained and needs no second prompt. `send review_prompt → reviewer_agents` (parallel) → `wait_for_all` → `send` inline `text` to `primary_agent` wrapping the combined reviews, e.g.:
  ```
  Here's feedback from several reviewers:
  {{ aggregated_responses(reviewer_agents) }}
  Let me know what your recommendations are.
  ```
  This keeps worked-example-2's *behavior* (fan-in → aggregate → primary) while dropping its saved-prompt indirection — so the workflow runs out of the box with only `code-review` shipped (the M6 acceptance test needs no extra prompt). The "fill a saved prompt's argument with forwarded output" path is still demonstrated by the flagship (its `ai-review-feedback` `review` arg). Reconcile spec worked-example-2 with a note that the shipped built-in uses the inline form (the spec example remains a valid illustration of the saved-prompt path).
- **`review-analyze-discuss`** (flagship) — inputs `reviewers: [agent]`, `analyst: agent`, `review_prompt: prompt_id` (intended: `code-review`), `analysis_prompt: prompt_id` (intended: `ai-review-feedback`), `context: text?`. Steps: `send review_prompt(context) → reviewers` → `wait_for_all` → `send analysis_prompt → analyst` with `template_vars: { review: "{{ aggregated_responses(reviewers) }}" }` → `wait_for analyst` → `send` inline `text` wrapping `{{ last_output(analyst) }}` → `reviewers` → `wait_for_all` → `send` inline `text` wrapping `{{ aggregated_responses(reviewers) }}` → `analyst` → `wait_for analyst`. The reviewers re-engage as the discussion panel (no separate panel input). The "appended text" of the manual flow is modeled as the `context` input passed to the prompt — workflows have no separate appended-text field; an inline-prompt step references prior output via the helpers (`last_output`/`aggregated_responses`) inside `text`, not a `{{ response }}` variable.

Because `prompt_id` inputs have no DSL default, the invocation form **pre-selects** a recommended built-in prompt for each, read from an **app-side `recommended_prompts` map keyed by built-in-workflow id** (built-ins are app-owned, so this needs no DSL change; a user-authored `prompt_id` input gets no pre-selection). The map: `review-analyze-discuss` → `code-review` (its `review_prompt`) + `ai-review-feedback` (its `analysis_prompt`); `review-and-aggregate` → `code-review` (its `review_prompt`) — its aggregation step is inline, so it has no second prompt input to pre-select. Pre-selecting the co-versioned built-ins keeps the default path render-consistent — the only way to hit an argument mismatch (e.g. a prompt that doesn't declare `context`) is to deliberately swap in an incompatible prompt, which surfaces as a failed run (handled by the failed-run surfacing above, not a hang).

**Frontend.**

- **Entry** — a `+ Workflow` button beside `+ Prompt` opening a workflow menu modeled on `PromptMenu` (lists built-in + directory workflows, built-ins tagged read-only, per-file parse errors shown). Always available (built-ins are always present unless the user toggles them off). Built-in rows offer **"Copy to my workflows"**; a Settings "Open workflows folder" action mirrors the prompts-folder action for adding your own.
- **Invocation form** — model on `PromptComposer.svelte` (structured args + preview lifecycle). One field per declared input; reuse the agent recipient-picker for `agent`/`[agent]` (a **pane** picked for an `[agent]` input expands to its member agent names at submit — panes are not a backend concept, same as M2) and the prompt menu for `prompt_id`. Validate before invoke; a capability-gated workflow shows the "not supported" message and disables invoke. Hide the compose bar's **To** + message-level forward affordances in workflow mode.
- **Progress indicator (chosen shape: a glanceable run indicator in the top bar that expands to a run list).** Among the system-design §7 "shape TBD" options (header indicator / side panel / modal) this milestone commits to the **top-bar indicator**: it stays out of the way, disappears when idle, and prominently surfaces interrupted runs on open — and the run's *substance* is already in the transcript, so this surface is a lightweight orchestration tracker, not a content view. It shows the active + interrupted + retained-failed run count and expands to a panel of run rows (name, step N/total, status, **Cancel**; interrupted/failed → **Abandon**). Driven by `listen` on `workflow:<project-id>` + a `list_workflow_runs` query on project open for crash-surfacing. **The `workflow:<project-id>` subscription is established at project-*load* (alongside `registerAgent` in `ensureProjectLoaded`), not gated to the active project, and torn down with the project** — otherwise switching away from a project freezes its running runs' updates, defeating the "watch concurrent runs across projects" point. Progress payloads carry `run_id` (the channel covers all of a project's runs).

**Doc.** `docs/agent-instructions/workflows.md` — the tutorial-style authoring doc (written *for an AI coding agent to consume* per system-design §2) **already exists**; M5 *updates* it, not drafts it from scratch. Bring it current: its partial-dispatch wording was reconciled to the non-destructive floor during the M5 planning pass, and M5 adds the two shipped built-ins as worked examples and verifies every claim still matches the M4/M5 behavior.

**Known limitations (recorded, accepted):**

- "Interrupted" is inferred as "a run file without a `Terminal` record that isn't in the live registry." # M7 — Pause for user input + iteration + interactive failure-pause

## Milestone 6 — `pause_for_user` (Modes 1 & 2) + interactive sibling-failure pause

### Goal & Outcome

A workflow can suspend mid-run for the user, and the M6 sibling-failure floor gains the interactive human-in-the-loop pause from the design discussion.

- **Mode 1 (no `recipient`):** the run suspends, an OS notification fires, the user submits (or skips), `user_input` is bound, the next step runs — no dispatch.
- **Mode 2 (with `recipient`):** the run suspends, the compose bar is pre-targeted at the recipient, the user's response is dispatched to it, and the step **implicitly waits** for that turn to terminate before continuing. A Mode-2 dispatch failure marks the run `failed` and retry re-enters the pause pre-filled with the prior `user_input`, requiring explicit re-submit.
- Both modes honor `required: true` (skip → `cancelled`) and `required: false` (skip → empty `user_input`, proceed).
- **Interactive failure-pause** (decision #5, M7 half): when a parallel/fan-in participant fails **and ≥1 sibling is still alive or has produced output**, the run enters a pause presenting the failed agent + error and each sibling's status/output, offering: retry the failed agent and continue / continue with survivors' output only / cancel the workflow. All-agents-failed (no survivor) falls back to ordinary step `failed`.

### Implementation Outline

- **Pause machinery.** Build the suspend/resume primitive: the runtime parks at a `pause_for_user` step, emits a "paused, awaiting input" progress state + OS notification (reuse the M5 notification plugin), and resumes when a new command delivers the user's input (or a skip). `user_input` binds into the scope per the spec (and is per-iteration-scoped inside `for_each`, which lands next milestone — build the scoping so M7's iteration reuses it). Mode 2 reuses the **M2 resolver / M4 dispatch path** for the dispatch + implicit wait — it is "capture, then a normal dependency-aware send to one agent." Implement the Mode-2 dispatch-failure → `failed` + retry-prefill rule (spec §`pause_for_user`).
- **Compose-bar pre-targeting.** The pause surfaces the compose bar pre-targeted at the configured recipient (Mode 2) or as a plain capture (Mode 1). Reuse existing compose-bar targeting; the new piece is the runtime binding the submitted text back to the suspended step.
- **Interactive failure-pause.** Wrap the M4 non-destructive floor: instead of "collect sibling outcomes → fail the step," when ≥1 sibling survives, route into the pause surface with the three options. "Continue with survivors" feeds the fan-in helpers only the succeeded agents (the per-run output-scope map already keys by agent — omit the failed one). "Retry failed and continue" re-dispatches just the failed agent and resumes. Reuse the pause machinery built above (same suspend/resume + notification); this is the payoff of deferring it from M6. The boundary cases (no survivor → ordinary `failed`; single-recipient send → ordinary `failed`; user-cancel during the step → `cancelled`) are in the spec's §"Sibling-failure policy" — implement exactly those.
- **Progress interaction (no crash-resume).** A pause is a progress boundary the surface reflects ("paused, awaiting input"). A run interrupted *while paused* is surfaced for **abandon** like any interrupted run — it is not resumed (resume deferred, decision #8). The live suspend/resume within a running session (parking at the pause, continuing on submit) is unaffected; only crash-recovery of a paused run is out of scope.

### Definition of Done

- Integration tests (mock harness): Mode 1 binds `user_input` and runs the next step with no dispatch; Mode 2 dispatches to the recipient and blocks until that turn terminates; Mode 2 dispatch failure → `failed` and retry re-enters the pause pre-filled; `required: true` skip → `cancelled`; `required: false` skip → empty `user_input` + proceed.
- Interactive failure-pause tests: sibling failure with a live survivor enters the pause (not bare step-failure); "continue with survivors" aggregates only succeeded agents; "retry failed" re-dispatches only the failed agent and resumes; "cancel" → `cancelled`; all-failed → ordinary `failed`; single-recipient failure → ordinary `failed` (no pause).
- Frontend tests for the pause surface (Mode 1 capture, Mode 2 pre-targeted dispatch, the failure-pause option buttons) with mocked `invoke`/`listen`.
- Notification fires on pause entry.
- `docs/workflow-spec.md` §"Sibling-failure policy" M7 half is now implemented — verify the doc matches the build; no doc edit expected unless a detail shifted.

## Milestone 7 — `for_each` iteration

### Goal & Outcome

A workflow can iterate a sub-sequence of steps over a static, invocation-time list, with correct scoping. (Resume/crash-recovery of an interrupted iteration is **deferred beyond v1** with all resume — see decision #8; an interrupted iterating run is abandoned like any other.)

- `for_each` runs its body once per item in a `[text]`/`[agent]` list, binding the iteration variable for that body only; iterations are sequential.
- The iteration variable is available in templates; `user_input` is scoped per-iteration (iteration N+1 does not see iteration N's input); cross-iteration **agent output** remains visible (spec §"Cross-iteration visibility").
- The progress surface shows the iteration dimension ("iteration 2 of 3 (milestone = 'X'), step 3 of 8").

### Implementation Outline

- **Iteration execution.** Bind the `item` variable per iteration (scope layer already modeled in M3); run the body steps (which reuse all existing primitives). Empty list → zero iterations (no-op, not an error). A failure inside iteration N halts the whole run (no per-iteration error handling in v1). Nested `for_each` is already a parse-time error (M3).
- **Scoping.** Per-iteration `user_input` reset; cross-iteration agent output preserved in the output-scope map (the map is per-run, not per-iteration — so later iterations see earlier ones' turns, which is the documented behavior; the author uses an explicit fresh `wait_for` to avoid stale reads).
- **Iteration progress (no resume).** The progress record tracks the current iteration index + value so the surface can show it and a crash surfaces an interrupted iterating run for abandon. Resume-oriented iteration checkpointing — restoring the output-scope map and `user_input` to re-enter iteration K at step N — is **deferred with all resume** (decision #8); do not implement replay here. The deferred design is preserved in the spec's "Retry from inside a `for_each`" note for the future milestone that adds resume.
- **Progress.** Extend the M5 progress payload with the iteration dimension (variable name, value, index/total) and the M6/M7 surface to render it.

### Definition of Done

- This milestone + Milestone 6 together are the **M7 acceptance test** (v1 plan): the milestone-iteration worked example (spec example 3) runs over a 3-item list and the progress surface shows the iteration index. (No resume assertion — resume is deferred; an interrupted iterating run is abandoned.)
- Tests: iteration over a 3-item list runs the body 3×; empty list is a no-op; per-iteration `user_input` isolation; cross-iteration agent-output visibility; a failure inside an iteration halts the run.
- Progress surface renders the iteration dimension (frontend test).
