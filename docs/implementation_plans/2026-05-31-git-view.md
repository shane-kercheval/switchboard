# Git view — implementation plan

A parallel-track feature: a top-level **Worktrees view** that sits alongside the existing Projects view, giving the user a single-glance tree of their git repos → branches → worktrees, with local status, the ability to create worktrees (optionally spinning up a Switchboard project in the new worktree), and a read-only diff inspector.

## How to read this doc

This is a standalone implementation plan for a feature that is **not** part of the v1 milestone roadmap (`2026-05-12-v1.md`). It does not block, and is not blocked by, any v1 milestone. It is sequenced as its own arc of five milestones, each independently demoable and reviewable.

The plan is written for an implementing agent that will **read the referenced code before acting but was not part of the design conversation.** Everything in the conversation that can't be recovered from the codebase is captured here; everything recoverable from the code (naming, local structure, which helper to reuse) is intentionally left to the agent's judgment against the actual code.

## Scope

**In scope (v1 of this feature):**

- A top-level **view toggle** between "Projects" (existing) and "Worktrees" (new).
- A read-only **tree**: tracked repos → branches → worktrees, with local status badges (dirty, untracked, ahead N, behind N, merged / not merged, synced) and the two warning states (orphaned worktree, prunable record).
- **Project linking**: each worktree row shows which Switchboard projects (if any) live in that directory.
- **Repo registry**: a user-global tracked-repo list with an explicit "Add Repo" action, auto-population when a Switchboard working directory that lives in a git repo is added, and a "Remove from view" action that touches only the list.
- **Non-destructive actions**: create worktree (the one write operation), open in terminal / editor / Finder, copy path/branch name, reveal repo in Finder, manual refresh, background fetch for ahead/behind.
- **Create-worktree flow** with an optional "also create a Switchboard project here" checkbox wired to the existing project-creation path.
- **Read-only detail panel**: changed-files list + inline syntax-highlighted diff.

**Explicitly out of scope (deferred):**

- **All destructive git operations**: `git worktree remove`, `git worktree prune`, branch deletion. The tree *surfaces* the prunable/orphaned warning states (so the user isn't confused), but offers no one-click destructive remedy in v1. Recorded as a known limitation per milestone.
- Commit / push / pull / stash / merge / rebase — anything that mutates history or the index.
- GitHub/GitLab integration (PRs, issues, CI status), branch-to-branch comparison, clone/init, remote-branch browsing.
- Filesystem watching for instant refresh (poll + manual refresh is the v1 commitment; see M3).
- Agent-tagging on worktrees (the standalone plan's future item — Switchboard's project-linking supersedes the need in v1).

### Relationship to system-design non-goals

`system-design.md` §2 lists "**Managing git, CI, or PR workflows** — Out of scope." This feature is deliberately positioned **adjacent to, not in violation of, that non-goal.** The boundary: Switchboard does not run `git commit`, open PRs, or integrate with CI — that remains out of scope and this feature adds none of it. What it adds is **workspace navigation** — seeing the worktree layout that multi-agent development naturally produces (one branch/worktree per agent) and creating a worktree + project in one action. "Git client" stays out; "see and set up the directories your agents run in" is in.

**Decision for the implementing agent:** when this feature's first milestone lands, add a one-line clarifying note to `system-design.md` §2 distinguishing the git-client non-goal from this navigation aid. Do **not** silently expand the §2 non-goal's meaning; surface the note in the PR for the maintainer to confirm the framing.

## Architecture decisions (made in the design conversation)

These are load-bearing and chosen over alternatives; they must survive into the code (as the new crate's module docs and as comments at the decision sites), not just live here.

1. **New pure-Rust crate `crates/git/` (`switchboard-git`), no Tauri dependency.** Mirrors `core`/`harness`/`dispatcher`. All git logic is unit/integration-testable without Tauri. The `app` crate depends on it and exposes it via `#[tauri::command]` shims, exactly as it does for the other crates.

2. **`git2` for local reads; shell out to the `git` CLI for fetch and worktree creation.** Reads (branches, worktrees, status, ahead/behind, merged-check, diffs) are pure-local and fast through `git2` — no credentials, no network. **Fetch** and **`git worktree add`** are shelled out via `tokio::process::Command` because (a) fetch needs the user's configured credential helpers / SSH agent, which `git2`'s credential callbacks reproduce poorly and fragilely, and (b) `git`'s own error messages for a failed worktree add are better than library-level errors and must be surfaced verbatim to the user. State this split in the crate's module doc; it's the single most important design choice here.

3. **Ahead/behind is computed locally; fetch only refreshes the inputs.** Ahead/behind against the remote-tracking branch is a local `git2` computation (cheap, offline) — it's simply *stale* until a fetch updates `refs/remotes/...`. So: the status read never blocks on the network; a separate, slower background fetch (shelled out, graceful-degradation on failure) keeps the remote-tracking refs current so the next local read is fresh. If there is no upstream / fetch has never run, ahead/behind is reported as "couldn't determine" (absent badge), distinct from "synced."

4. **Merged-check is local and against the repo's default branch.** "merged" = the branch tip is an ancestor of the default-branch tip. Detect the default branch via `refs/remotes/origin/HEAD`, falling back to a local `main`, then `master`; if none resolves, merged is "couldn't determine" (absent badge). Document the fallback order at the call site.

5. **The git view has its own user-global tracked-repo registry, separate from `workspace.yaml`.** A repo tracked for *visibility* is a superset of directories that *host Switchboard projects* — overloading `workspace.yaml` would blur its meaning. The registry is a new user-global file resolved through the same `directories`-crate + `SWITCHBOARD_CONFIG_DIR` mechanism as `workspace.yaml`. **Auto-sync is one-directional:** adding a working directory to the Switchboard workspace appends its discovered repo root to the git registry; adding a repo directly to the git view does **not** create a Switchboard workspace entry or any project. Removing a repo from the git view touches the registry only — never files, never `workspace.yaml`.

6. **Only the *list of tracked repos* persists; branch/worktree/status data is always read live.** Unlike `workspace.yaml` (which caches project snapshots for offline display), the registry caches no git state — git data is cheap to recompute and dangerous to show stale. An unavailable repo (path missing/unmounted) renders as unavailable with a remove action, mirroring the workspace's unavailable-directory pattern, but carries no cached tree.

7. **Project linking is computed in the backend by path-matching.** The aggregate command cross-references each worktree's on-disk path against the loaded workspace's directory entries (and their projects) and returns the linkage, so the frontend renders it directly. Path comparison uses the same canonicalization boundary as the rest of `app` (`canonicalize_boundary`).

8. **Slow git work runs off the async runtime's worker via `spawn_blocking`.** `git2` is synchronous and `status`/diff on a large repo can take noticeable time. The aggregate read command is an async Tauri command whose `git2` work is offloaded with `tokio::task::spawn_blocking`; fetch and worktree-add use `tokio::process::Command` directly.

## M1 — Git read layer (`crates/git/`)

### Goal & Outcome

A pure-Rust crate that, given a path on disk, produces the full read-model the UI needs — with no Tauri dependency and no UI.

Once complete:

- Given any path inside or at a git repo, the crate resolves the **repo root** (main worktree / common dir) and the repo's **default branch**.
- It lists all **branches** and all **worktrees**, pairing each worktree 1:1 with its branch (and grouping **detached-HEAD** worktrees separately, labelled by short commit hash).
- For each worktree it computes local **status**: dirty (tracked changes), untracked (separately), ahead N / behind N vs. upstream (or "couldn't determine"), merged / not-merged vs. the default branch (or "couldn't determine"), and the derived "synced."
- It flags the two **warning states**: orphaned (worktree on disk whose branch was deleted) and prunable (git has a worktree record whose directory is gone).
- It produces, for a single worktree, the list of **changed files** with per-file change kind, and the **unified-diff text** for a given file (consumed by M5).
- An **unavailable** repo path yields a clearly-marked "unavailable" result rather than an error.

### Implementation Outline

Create the workspace member with `cargo new --lib crates/git` (then wire it into the workspace; add `git2` with `cargo add`, **never** by hand-editing — per AGENTS.md). Name it `switchboard-git` to match the `switchboard-*` convention.

The public surface is a small set of read functions plus the data types they return. The exact field set is the agent's to finalize against `git2`'s API, but the **contract shape** below is load-bearing — the UI and the command layer (M2) are designed around it, so keep these distinctions:

```rust
// Illustrative — the distinctions (1:1 branch↔worktree, detached grouping,
// Option = "couldn't determine" vs. a definite value, warning states) are the
// contract; exact names/fields are the implementer's to settle against git2.
pub struct RepoView {
    pub root: PathBuf,                 // main worktree root
    pub name: String,                  // folder name of root
    pub default_branch: Option<String>,
    pub branches: Vec<BranchView>,
    pub detached_worktrees: Vec<WorktreeView>,
    pub available: bool,               // false => path missing/unreadable; tree empty
}
pub struct BranchView {
    pub name: String,
    pub worktree: Option<WorktreeView>, // Some => checked out in a linked/main worktree
}
pub struct WorktreeView {
    pub path: PathBuf,
    pub status: WorktreeStatus,
    pub warning: Option<WorktreeWarning>, // Orphaned | Prunable
}
pub struct WorktreeStatus {
    pub dirty: bool,
    pub untracked: bool,
    pub ahead: Option<u32>,   // None = no upstream / never fetched
    pub behind: Option<u32>,
    pub merged: Option<bool>, // None = default branch unresolved
    pub synced: bool,         // clean && untracked==false && ahead==Some(0) && behind==Some(0)
}
```

Order of work: (1) repo discovery + default-branch detection (decision 4's fallback order, documented at the site); (2) branch + worktree enumeration and pairing; (3) per-worktree status; (4) warning-state detection; (5) changed-files + per-file diff text for M5.

Decisions to honor (from "Architecture decisions"): ahead/behind and merged are **local** computations returning `Option` for the "couldn't determine" case (decisions 3, 4); fetch and worktree-add do **not** live in this crate's M1 surface — fetch is M3, worktree-add is M4, both shelled out from `app`. Keep this crate read-only in M1.

Use `thiserror` for a typed error at the crate boundary, per the Rust conventions. Distinguish "this path isn't a git repo" / "path unavailable" (both expected, non-error UI states represented in `RepoView`) from genuine errors (a corrupt repo, an I/O failure mid-read).

### Definition of Done

- **Integration tests** (Cargo `tests/`, the crate's public API) against fixture repos built in a `tempfile` tempdir. Construct, with `git2` or by shelling `git` in the test harness: a repo with a clean synced branch; a branch with a worktree that is dirty; one with untracked-only files; one ahead and one behind an upstream; a merged branch and an unmerged branch; a detached-HEAD worktree; an orphaned worktree (delete the branch, keep the dir); a prunable record (remove the dir, keep git's record). Assert the computed `RepoView` for each.
- A repo with **no upstream configured** yields `ahead/behind = None` (not `Some(0)`); a repo where the default branch can't be resolved yields `merged = None`. These are distinct assertions — they're the "absent badge, not zero" contract the UI depends on.
- An **unavailable path** yields `available: false` with an empty tree and no error.
- Changed-files + diff-text functions return correct output for a worktree with staged, unstaged, and untracked changes.
- Crate module doc states the **`git2`-reads / shell-out-writes** split (decision 2) and the default-branch fallback order (decision 4). No live-network test (fetch is M3).
- Known limitation recorded if any status (e.g. submodule state) is intentionally not computed.

## M2 — Repo registry, project linking, and Tauri commands

### Goal & Outcome

The backend surface the frontend will call: a persisted tracked-repo list, the auto-sync hook, project linking, and the commands that return the full tree.

Once complete:

- A **user-global tracked-repo registry** persists the ordered list of repo roots across launches, degrading to empty on a missing/corrupt file (mirroring `Workspace`).
- Adding a working directory to the Switchboard workspace **auto-adds** its discovered repo root to the registry (one-directional; decision 5).
- Commands exist to **list**, **add** (explicit "Add Repo"), and **remove** tracked repos; remove touches only the registry.
- A **list-worktrees aggregate command** returns, for every tracked repo, the `RepoView` from M1 enriched with **which Switchboard projects live in each worktree path** (decision 7), with unavailable repos marked.
- All git reads run **off the async worker** (decision 8).

### Implementation Outline

**Registry storage.** Add a registry type in `app` modeled on `workspace.rs` (ordered list, private vec + accessor, `Serialize`/`Deserialize`, graceful-degradation load). Persist it to a new user-global YAML file resolved by a function parallel to `workspace_config_path()` in `lib.rs` — same `directories`-crate base dir and same `SWITCHBOARD_CONFIG_DIR` debug override, a sibling filename. Add a field to `AppState` for the in-memory registry and (if needed) its path, following the existing `workspace` / `workspace_path` shape; add a `persist_*` helper paralleling `persist_workspace`. **Update the documented lock-order convention in `state.rs`** to place the new mutex — it is read mostly standalone by git commands; position it adjacent to `workspace` and document the choice.

**Auto-sync hook.** In `init_directory_impl`, after the directory is registered in the workspace, discover the repo root for the canonical path (via the M1 crate) and, if found, append it to the git registry (dedup on canonical path) and persist. This is the only write to the registry that isn't user-initiated. Guard it so a non-git directory is simply skipped — no error.

**Commands** (each a `*_impl` free function + a thin `#[tauri::command]` shim in `lib.rs`, errors stringified at the boundary exactly as the existing commands do):

- list tracked repos (returns each repo's path, name, availability — *not* a full tree; cheap).
- add a tracked repo (validate it's a git repo via M1; reject with a clear error if not — mirror the inline-error UX the standalone plan describes; dedup; persist).
- remove a tracked repo (registry only; never touch files or `workspace.yaml`).
- **aggregate worktrees** (the heavy one): for each tracked repo, compute the M1 `RepoView` and enrich each worktree with linked projects. **Project linking** (decision 7): cross-reference each worktree path against `AppState.workspace` directory entries (and their cached projects) / loaded projects, matching on `canonicalize_boundary`. Return a wire type that nests projects under worktrees. Make this command **async** and wrap the `git2` work in `tokio::task::spawn_blocking` (decision 8).

**Wire types.** Add TypeScript discriminated-union / record types in `src/lib/types.ts` mirroring the Rust serde shapes (snake_case keys, the `Option` fields as `T | null`), and `invoke` wrappers in `src/lib/api.ts`, following the existing conventions. No frontend rendering yet (that's M3) — but landing the types + api wrappers here keeps M3 purely presentational.

### Definition of Done

- **Unit tests** on the registry type: add/dedup/remove/order-preservation; missing-file and corrupt-file both load to empty (not error) — the graceful-degradation contract.
- **Integration tests** on the `*_impl` functions (the established `app` test style, targeting free functions): auto-sync appends a repo root when a git directory is added and is a no-op for a non-git directory; add rejects a non-git path with a typed error; remove leaves `workspace.yaml` and the on-disk directory untouched; the aggregate command returns correct project linkage for a worktree path that matches a workspace directory hosting a project, and empty linkage for one that doesn't.
- The aggregate command marks an **unavailable** repo correctly and does not fail the whole call when one repo of several is unavailable (partial success — one bad repo never blanks the view).
- Lock-order doc in `state.rs` updated; the rationale for the new mutex's placement is in the comment.
- Known limitation recorded: registry stores paths only; git data is always live (decision 6) — state it in the registry type's doc.

## M3 — Worktrees view: tree, status, refresh, fetch, open-actions

### Goal & Outcome

The read-only view itself — the core value of the feature.

Once complete:

- A **view toggle** in the title bar switches between "Projects" and "Worktrees"; the choice is part of the app's view state and the Projects experience is unchanged when toggled away.
- The Worktrees view renders the **tree**: tracked repos → branches → worktrees. Branches with worktrees are emphasized; branches without are dimmed and sortable below; a toggle hides inactive branches. Worktrees show abbreviated path (`~`), status badges, and **linked Switchboard projects**.
- **Warning states** (orphaned, prunable) render distinctly (not hidden) — but with **no destructive remedy** in v1 (decision: surface, don't fix; record as known limitation).
- **Add Repo** (native folder picker → validate → appear, or inline error) and **Remove from view** (right-click / menu, registry-only) work. Unavailable repos render marked, with remove.
- **Refresh**: a manual refresh button always available; a **poll** of local status on an interval while the view is visible; a **background fetch** (shelled out, graceful degradation) on a slower cadence and on demand, surfacing a subtle "last fetched" indicator and degrading silently (absent ahead/behind badges) when a remote is missing or fetch fails.
- **Open actions** on a worktree (terminal, editor, Finder/Explorer, copy path) and repo (reveal in Finder); the editor command and terminal are read from preferences in the existing Settings surface.

### Implementation Outline

**View toggle.** Extend the top-level view state in `src/lib/state/workspace.svelte.ts` (today: `selection` + a `settingsOpen` boolean) with a `"projects" | "worktrees"` mode, and branch on it in `App.svelte`'s center/left snippets. Add the toggle control to the existing title bar. Do not disturb the existing Projects/Settings/Welcome branching — add a sibling, don't refactor the structure.

**The tree component.** Build it from the existing primitives per `ui-conventions.md` — `SidebarPanel`/`SidebarSection` for structure if the tree lives in the left column, `StatusDot`/`status-*` tokens for run-like status, `Badge` for incidental labels, `DropdownMenu` for the right-click/context menu (never hand-roll a menu), `EmptyState` for the no-repos-tracked case. **Every color names a semantic token** — if the status badges need a hue not in the token set, add a named role to `app.css`, don't reach for a raw Tailwind color. Status mapping (dirty / untracked / ahead / behind / merged / synced / warning) should reuse `status-*` where the meaning matches and get new named roles only where it genuinely doesn't.

**Refresh model** (decision: poll + manual is the v1 commitment; fs-watch is deferred). Poll the aggregate command on an interval **only while the Worktrees view is visible** — don't poll in the background from the Projects view. Manual refresh re-invokes immediately. Keep local-status polling and fetch on **separate cadences**: status poll is frequent and offline (decision 3); fetch is slower, shelled out from a new `app` command (`tokio::process::Command`), and its failure is swallowed into "ahead/behind simply absent" plus a stale "last fetched" timestamp — never an error toast. Add the fetch command following the M2 command pattern.

**Open actions.** Folder picker reuses the existing dialog plugin (`tauri-plugin-dialog`, already a dependency — see `init_directory` flow). Opening in Finder/terminal/editor and copy-to-clipboard use the Tauri shell/clipboard surface; read the editor command and terminal preference from the existing Settings (`SettingsView`) — add the two preference fields there following its current persistence. Keep these preferences minimal (editor command string, terminal choice); the standalone plan's other prefs land in M4/M5 where they're used (worktree base path → M4, diff style → M5).

**Component-level tests are required** (per AGENTS.md's rule for components that wrap `invoke` + reactive state): mock `invoke`, exercise the tree render, the inactive-branch toggle, the add/remove/refresh actions, and the unavailable-repo and empty states.

### Definition of Done

- View toggle switches cleanly both ways; Projects view is byte-for-byte unaffected when toggled away and back.
- Tree renders repos/branches/worktrees with correct emphasis (active vs. dimmed inactive), correct sort, the inactive-branch hide toggle, status badges matching backend data, and linked-project labels.
- Orphaned and prunable worktrees render with a distinct warning indicator and **no** destructive action; the absence of a remedy is recorded as a known limitation in the plan and a code comment at the render site.
- Add Repo: valid git folder appears; non-git folder shows an inline error; duplicate is a no-op. Remove: repo leaves the view; `workspace.yaml` and disk untouched (assert in a backend test if not already covered by M2).
- Background-fetch failure / no-remote degrades to absent ahead/behind + stale "last fetched", never an error surface.
- Open-in-terminal/editor/Finder and copy actions work on macOS; editor/terminal preferences persist and round-trip.
- **Component tests** (mocked `invoke`) cover render, toggles, actions, and the unavailable/empty states; **reducer/state tests** cover the new view-mode state. Theming: the view renders correctly in light and dark (token-driven).

## M4 — Create-worktree flow (+ optional Switchboard project)

### Goal & Outcome

The one write operation in v1: create a worktree from a branch, and optionally spin up a Switchboard project in it in the same action.

Once complete:

- A branch's context menu offers **Create Worktree**, opening a dialog pre-filled with the branch name (read-only) and an editable destination path (sensible default from a configurable base).
- On confirm, Switchboard runs `git worktree add` (shelled out), and the new worktree **appears in the tree** on success.
- On failure (e.g. branch already checked out elsewhere), the **git error message is surfaced verbatim** inline — not swallowed (decision 2).
- The dialog includes an **"Also create a Switchboard project in this directory"** checkbox (default per the conversation: offered, checked); when set, after the worktree is created Switchboard runs the existing add-directory + create-project path against the new worktree directory.

### Implementation Outline

**Backend command.** Add a `create_worktree` `*_impl` + shim. It shells out `git worktree add <path> <branch>` via `tokio::process::Command` against the repo root (decision 2). On non-zero exit, return a typed error carrying git's **stderr verbatim** so the frontend can show it inline. On success, the next aggregate read reflects the new worktree (no need to synthesize it — refresh re-reads). This is a **write that does not need the `git2` crate** — keep it in `app` as a shelled command, consistent with the fetch command from M3.

**Optional project creation.** When the checkbox is set, after a successful worktree add, invoke the **existing** flow: `init_directory_impl` (registers the new worktree dir as a working directory — which, per M2's auto-sync, also keeps the git registry consistent) followed by `create_project_impl`. Reuse these as-is; do not duplicate their logic. Decide sequencing in the command layer so a worktree-add success followed by a project-create failure surfaces a clear partial-success message (the worktree exists; the project didn't get created) rather than a silent or misleading state — the worktree-add is the irreversible step and must not be rolled back to satisfy a later project-create failure.

**Worktree base path preference.** Add the "default worktree location" preference (used to pre-fill the destination) to the existing Settings surface, following M3's preference pattern. Default to a sensible base (the conversation referenced `~/worktrees/<repo>-<branch-slug>`-style); the exact slugging is the agent's to choose against the codebase.

**Dialog.** Build on the `Dialog` primitive. Branch name read-only; path editable and pre-filled; the checkbox; a Create button; an inline error region that renders git's verbatim stderr on failure.

### Definition of Done

- **Integration test** on `create_worktree_impl`: a successful add against a fixture repo produces a worktree on disk that the M1 reader then sees; a failing add (e.g. branch already checked out) returns a typed error whose payload contains git's stderr.
- **Integration test** on the optional-project path: checkbox-on creates a worktree *and* a project bound to it (assert via the existing project listing); a project-create failure after a successful worktree-add yields a partial-success result with the worktree intact (no rollback of the worktree).
- **Component test** (mocked `invoke`): dialog pre-fills branch + default path, surfaces a verbatim git error inline on failure, and on success the tree refreshes to include the new worktree.
- Worktree base-path preference persists and pre-fills the dialog.
- Known limitation recorded: no worktree **removal** in v1 — a worktree created here can only be removed outside Switchboard (or via a future destructive-ops milestone).

## M5 — Worktree detail panel + read-only diff

### Goal & Outcome

Inspect a worktree's working changes without leaving Switchboard.

Once complete:

- Clicking a worktree opens a **detail panel** (placement — right split or bottom split — is the agent's call against the layout; the conversation left it TBD).
- The panel header shows branch name, full clickable path (opens in Finder), and the status badges.
- A **changed-files list** shows each file with its change kind (modified / added / deleted / untracked).
- Clicking a file shows its **diff inline**, syntax-highlighted, **read-only**, with a unified/split toggle persisted as a preference.

### Implementation Outline

**Backend.** Reuse M1's changed-files + per-file unified-diff-text functions; expose them through `app` commands (changed-files for a worktree path; diff-text for a worktree path + file). Async + `spawn_blocking` for the `git2` work, consistent with M2.

**Diff rendering.** Use **`diff2html`** (add via `pnpm add`): feed it the unified-diff text from the backend; it emits line-by-line or side-by-side HTML, satisfying the unified/split preference. The repo already ships `prismjs` for syntax highlighting and a `markdown.ts` rendering path — prefer integrating `diff2html`'s output with the existing highlighting approach over introducing a second highlighter, and **theme the diff with semantic tokens** (the `syntax-*` group exists for code; reuse it rather than `diff2html`'s default palette) per `ui-conventions.md`. The diff view is **read-only** in v1 — no editing, no staging.

**Diff-style preference.** Add the unified/split toggle to Settings (or a panel-local control persisted the same way), following M3/M4's preference pattern.

### Definition of Done
